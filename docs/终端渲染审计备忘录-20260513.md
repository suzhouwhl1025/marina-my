## 1. PTY → xterm 字节路径

### OSC-1 [H] · node-pty `onData` 已经按 UTF-8 解码，二次编码不可逆
**位置**：`src/main/session-manager.ts:746-748`
**现象**：在中文 Windows 上 `type` 一个 GBK 文件、`cat` 二进制文件、ANSI art 字符，xterm 显示成 U+FFFD 或乱码。
**根因**：node-pty 内部把 ConPTY 字节流按 UTF-8 解码成 JS 字符串，无效字节被替换成 U+FFFD（不可逆）。我们 `Buffer.from(data, 'utf8')` 再编一遍依然是 utf8，**字节信息已丢**。
**修复方向**：长期方案是让 node-pty 暴露 raw bytes 模式（社区有 fork），短期承认限制，README 写明"非 UTF-8 输出可能乱码"。

### OSC-2 [H] · scrollback ring buffer 在字节中间裁切，导致接管时首屏乱码
**位置**：`src/main/session-manager.ts:796-807`
**现象**：长跑 session 切窗口接管时，首屏出现：颜色异常、大段不可见（被吞）、个别字符变 `?` / 替换字符。
**根因**：`scrollback.subarray(length - 2MB)` 是裸字节裁切，落点可能在：
- 多字节 UTF-8 序列中间 → 首字符替换字符；
- **CSI/OSC/DCS 转义序列中间** → xterm 进入 OSC parse 状态，吞掉数十-数百字节直到下一个 BEL/ST；
- SGR 颜色序列中间 → 颜色错位直到下一个完整 SGR。
**修复方向**：写一个 `findSafeTruncationBoundary(buf, fromOffset)` helper，从目标偏移向前找最近的非 ANSI/UTF-8 边界（ASCII printable 或 `\n` 之后）作为实际裁切点。

### OSC-3 [M] · OSC1337 stash 满 16KB 时整段前缀作为普通字节透传
**位置**：`src/main/osc1337-parser.ts:126-130`
**现象**：偶发地在终端里看到字面量 `\x1b]1337;CurrentDir=...` 渲染出来。
**根因**：永远收不到 BEL/ST 的破损 OSC 序列被 STASH_LIMIT 截掉后 push 到 passthrough。
**修复方向**：满 stash 时应当 drop（不进 passthrough）+ logger.warn 记录，便于排查触发场景。

### OSC-4 [M] · OSC 0x9D（C1 单字节 OSC 指示符）未识别
**位置**：`src/main/osc1337-parser.ts:97`
**现象**：使用 C1 字符的 shell（罕见 CJK 终端、某些跨平台二进制）发的标题 / cwd 不被解析，session.displayName / currentCwd 不会按 OSC 更新。
**根因**：parser 只认 `ESC ]` (0x1B 0x5D)，不认 0x9D。
**修复方向**：在 `indexOf(0x1b)` 同步找 0x9D，复用同样的 terminator 查找。优先级低，因为 modern shell 几乎都用 ESC ]。

### OSC-5 [L] · OSC 8 超链接整段透传但无点击 handler
**位置**：`src/main/osc1337-parser.ts:150-152`
**现象**：用户在终端里看到带下划线的 URL（OSC 8 包装），但点击无反应。
**根因**：xterm 5.x 支持 OSC 8 渲染但需要在 main 装 `webContents.setWindowOpenHandler` 路由到 `shell.openExternal`；当前未装。同时 WebLinksAddon 的正则识别 URL 也走同样的 window.open 路径，CSP 拦掉。
**修复方向**：在 `window-manager.ts` 的 createWindow 里 `win.webContents.setWindowOpenHandler(details => { if (/^https?:|^mailto:/.test(details.url)) { shell.openExternal(details.url); } return { action: 'deny' }; })`。

### OSC-6 [L] · `sanitizeTitle` 防控制字符，但不防 RTL/双向重写攻击
**位置**：`src/main/session-manager.ts:1078-1089`
**现象**：恶意进程在 PTY 内发 `OSC 0;<‮>evil.exe<‬> safe.txt` → tab 标题视觉上变成 `safe.txt exe.live`（顺序反转）。
**根因**：sanitizeTitle 只替控制字符 (<0x20 / 0x7F)，没过滤 Unicode 双向重写字符 (U+202A-202E, U+2066-2069, U+200E/200F)。
**修复方向**：扩展 sanitize 列表。

### OSC-7 [M] · 字节流分片 base64 在 renderer 端 JS 循环解码
**位置**：`src/renderer/components/TerminalView.tsx:953-966`
**现象**：高速输出（`find /` 等）时 renderer CPU 单核 30-50%，xterm 渲染 lag。
**根因**：`decodeBase64ToBytes` 用 `for` 循环 + `String.charCodeAt`，每个 PTY chunk 走一次。
**修复方向**：用 `Uint8Array.from(atob(b64), c => c.charCodeAt(0))` 让 V8 优化；终极方案是 Electron IPC 传 ArrayBuffer，main 端不 base64 直接 Buffer.buffer transfer。

### OSC-8 [M] · main 端 `appendScrollback` 每次 PTY chunk 都 `Buffer.concat([scrollback, bytes])`
**位置**：`src/main/session-manager.ts:796-807`
**现象**：scrollback 接近 2MB + PTY 持续输出时，main 单核 CPU 100%，IPC 反压，sendInput 排队。
**根因**：每次 concat 是 O(scrollback.length) 内存拷贝。每秒数百次 × 2MB = GB/s 内存拷贝。
**修复方向**：预分配 4MB Buffer + 双指针 ring buffer。或限制 scrollback 为更小（512KB）权衡。

## 2. xterm 实例与渲染器

### XTM-1 [H] · 未装 WebGL/Canvas 渲染器，默认 DOM renderer 是最慢的
**位置**：`src/renderer/components/TerminalView.tsx:414-432`
**现象**：长瀑布输出（`npm install`、`find /`、Claude Code 流式 token）时 renderer 卡顿、CPU 100%、frame drops 可见。
**根因**：xterm 默认 DOM renderer 为每个 cell 创建 DOM 节点，渲染开销大；WebGL/Canvas 渲染器 10-50× 更快。
**修复方向**：`npm i @xterm/addon-webgl`，mount 时 `term.loadAddon(new WebglAddon())`；WebGL context loss 时 fallback Canvas。需走 AGENTS.md 1.2 边界 2 询问加包。

### XTM-2 [M] · `allowProposedApi: true` 长期开
**位置**：`TerminalView.tsx:425`
**现象**：未来 xterm 升级时 SearchAddon 的 `registerDecoration` 等 proposed API 改签名 → 命中高亮静默失效。
**根因**：xterm 在 proposed API 上不做语义版本承诺。
**修复方向**：把使用的 proposed API 调用包成 try/catch，调用前 feature-detect。或锁 xterm minor 版本不自动升。

### XTM-3 [M] · `windowsMode: true` 全局开
**位置**：`TerminalView.tsx:431`
**现象**：git-bash 下 vim 行尾偶尔多/少一个空格；某些 POSIX TUI（在 WSL 内）也有微小渲染偏差。
**根因**：windowsMode 让 xterm 用 Windows 风格处理 LF / CR / 行尾，对 ConPTY 友好但对 POSIX 路径过度处理。
**修复方向**：按 session 的 shell.id 动态 set：pwsh/cmd 用 true，git-bash 用 false。

### XTM-4 [H] · `XTERM_THEMES` 与 `global.css` 是两套真理
**位置**：`TerminalView.tsx:84-246` vs `src/renderer/styles/global.css` 的 `[data-theme="..."]` 块
**现象**：改一处忘改另一处，xterm 颜色与 UI 颜色割裂（已经多次踩过）。
**根因**：手工同步无编译期保障。
**修复方向**：提一个 `src/shared/themes.ts` 单源，主题对象同时导出 ITheme 与 CSS 变量集；CSS 变量在 ConnectedShell 用 `document.documentElement.style.setProperty` 注入（已经有 UI 字体的先例）。

### XTM-5 [M] · `attachCustomKeyEventHandler` 闭包抓住挂载时的 callback 引用
**位置**：`TerminalView.tsx:457-505`，与 `useCallback` 依赖共同决定
**现象**：当前无 bug（handleCopy/handlePaste 的 deps 都是 [session.id] 或空），但**未来加 settings 相关分支**时极易踩到"键盘拦截行为停留在旧值"。
**根因**：mount effect 的 deps 是 `[session.id, myWindowId]`，handler 闭包内调用的 callback 是 effect 首次跑时的版本。
**修复方向**：handler 内通过 ref 读最新 callback，类似 `searchTextRef` 的模式；或干脆把 handler 内联在 useEffect 内，所有依赖显式列。

### XTM-6 [M] · `selectOnCopy` 每次 selectionChange 都写剪贴板
**位置**：`TerminalView.tsx:716-726`
**现象**：拖选过程中输入法 / 剪贴板管理器 / 截图工具频繁触发；选完后剪贴板被密集写。
**根因**：xterm `onSelectionChange` 在每个 mousemove 新选中 cell 时 fire，handler 直接 IPC。
**修复方向**：trailing debounce 100ms；或改成 mouseup 时一次性写。

### XTM-7 [M] · ResizeObserver 150ms debounce 期间打字以旧 cols/rows 处理
**位置**：`TerminalView.tsx:644-651`
**现象**：拖窗 + 同时编辑长命令时，PowerShell readline 折行错位、Backspace 删错位置。
**根因**：debounce 把 resize 推迟，但 dataHandler 立即工作；PTY 拿到 keystroke 时按旧 cols 计算 prompt 行宽。
**修复方向**：`dataHandler` 内检查 `resizeTimer !== null` → 立刻 `clearTimeout` + 同步 `performResize()` 再 sendInput。

### XTM-8 [L] · `fit()` 在 0×0 容器下返回 cols/rows = 0 但代码不防御
**位置**：`TerminalView.tsx:527-531, 619-625`
**现象**：mount 瞬间容器还没 layout 时 fit 算出 0；后续 ResizeObserver 修正前 PTY 用 cols=0 工作 → xterm InputHandler 状态错乱。
**根因**：try/catch 只捕获异常，不验证返回值；`validateDimensions` 在 main 端可能也兜不住 0（待确认）。
**修复方向**：fit 后断言 `cols >= 20 && rows >= 5`，否则跳过 IPC 等下一轮 RO；fit 用 `requestAnimationFrame` 延后到 layout 完成。

### XTM-9 [L] · webfont 首次加载时 measure 不准
**位置**：`TerminalView.tsx:678-692`（字号/字体 effect）
**现象**：用户切自定义 terminalFontFamily 后，第一屏边缘空 1-2 列。
**根因**：xterm 在 webfont 加载完成前 measure 用 fallback metrics，加载完成不主动 re-fit。
**修复方向**：`document.fonts.ready.then(() => fitAddon.fit())` 兜底；监听 `document.fonts.onloadingdone`。

### XTM-10 [L] · `term.write` 是异步批量但跨进程 IPC 难保同帧到达
**位置**：`TerminalView.tsx:572`（output 监听器内）
**现象**：连续敲键时回显感觉"延迟一下、滑一下"，光标移动不流畅。
**根因**：每个 PTY echo 字节是单独 SESSION_OUTPUT IPC，跨 main/renderer 边界，xterm 内部的 RAF 合并机制看不到它们是连续的。
**修复方向**：main 端 `handlePtyData` 做 8ms 合并窗口，把窗口内多个 chunk concat 后一次性 emit。

## 3. 焦点管理（用户主诉的核心区域）

### FOC-1 [H] · TerminalView mount 后从不调用 `term.focus()`
**位置**：`TerminalView.tsx:409-668`（整个 mount effect）
**现象**：切 tab / 创建 session / 接管 orphan / 退设置 / 窗口恢复后必须再点一下终端区才能打字。
**根因**：grep 全工程 `term.focus()` 只出现三处：`handleCloseSearch`、`handleTerminalDrop`、`handleOpenSearch`（聚焦 search input 而非 term）。mount effect 内**完全没有** focus 动作。
**修复方向**：mount effect 末尾（`fitAddon.fit()` 之后或 scrollback replay then 之后）`term.focus()`；同时给 TerminalView 加一个 `useEffect` 监听 `displayable.id` 变化时 refocus。

### FOC-2 [H] · Tab `<button>` click 后焦点留在按钮上
**位置**：`src/renderer/components/MainPane.tsx:566-595, 481-535`
**现象**：用户点 tab 切 session 后焦点在 button 上，敲键被 button 默认行为吞，Enter 重新触发 click。
**根因**：原生 `<button>` 的浏览器默认 :focus 行为；handleClick 没归还焦点。
**修复方向**：handleClick 末尾 `requestAnimationFrame(() => { (document.querySelector('.xterm-helper-textarea') as HTMLElement | null)?.focus(); })`；或暴露一个 ref 路径让 store 通知 TerminalView 主动 focus。

### FOC-3 [H] · WindowChrome 三按钮、模板按钮、新建占位 tab、设置入口全是 `<button>`，点完都留焦
**位置**：`WindowChrome.tsx`、`MainPane.tsx:TemplateLaunchButton`、整个 SettingsView
**现象**：用户切窗口（点最大化）、新建 session（点模板）、退设置后，焦点都不会自动给 xterm。
**根因**：同 FOC-2，是系统性问题不是单点。
**修复方向**：在 ConnectedShell 顶层注册一个 `focus-terminal` global helper，所有需要"焦点应该在终端"的回调结尾调它一次；或给 TerminalView 加一个"接管焦点信号"机制（store 的 lastFocusBus）。

### FOC-4 [M] · 搜索面板内 ↑/↓/Aa/× 按钮交互后焦点留 button
**位置**：`TerminalView.tsx:908-947`
**现象**：在搜索过程中点击导航按钮后，再键入想跟下一个匹配的 ↑/↓ Enter 但 Enter 被 button 重新触发其 click。
**根因**：search input 失焦后没有归还。
**修复方向**：每个 button 的 onClick 末尾 `searchInputRef.current?.focus()`。

### FOC-5 [M] · ContextMenu / Toast 关闭不实现 focus return
**位置**：`src/renderer/components/ContextMenu.tsx`、`Toast.tsx`
**现象**：右键弹菜单后无论选/不选关闭，焦点漂到 body，xterm 不能继续输入。
**根因**：未实现标准 ARIA 模式 focus return。
**修复方向**：菜单打开时记录 `document.activeElement`，关闭时 `prev.focus()` 兜底（如果是按钮则再走 FOC-3 的路径转给 xterm）。

### FOC-6 [M] · `evt:window:focus-requested` 不让 xterm 拿焦点
**位置**：`src/renderer/store.tsx:291-309`
**现象**：托盘"运行中的会话"子菜单点击 → main 推 focus-requested → store dispatch select-session → MainPane 重渲染 → TerminalView 仍不聚焦。
**根因**：reducer 只改 selectedSessionId，不触发副作用。
**修复方向**：reducer 不动，在 ConnectedShell 加一个 `useEffect` 监听 `selectedSessionId` 变化后 focus terminal。

### FOC-7 [M] · IME composition 期间 React re-render 可能偷焦点
**位置**：跨 `store.tsx`（事件 dispatch 风暴）+ `TerminalView.tsx`（attachCustomKeyEventHandler 不防 composition）
**现象**：中文输入到一半，按下空格/Enter 没反应；切换英文也切不动；只能点别处再点回来。
**根因**：xterm `.xterm-helper-textarea` 上的 composition 状态机被某个不期而至的 React commit 中断，IME 卡在"开始组合但收不到 keyup"。同时 attachCustomKeyEventHandler 也没短路 `ev.isComposing`。
**修复方向**：handler 头部加 `if (ev.isComposing || ev.keyCode === 229) return true;`；store 改用细粒度 selector 减少全树 re-commit（长期）。

## 4. 闪烁与视觉抖动

### FLK-1 [H] · 接管时 scrollback 同步 write 2MB
**位置**：`TerminalView.tsx:583-585`
**现象**：切到一个长跑 session 的瞬间，屏幕黑一下（100-300ms）后突然出现全屏内容；期间敲的键被 OS event queue 暂存，replay 完成后一次性涌入 PTY。
**根因**：`term.write(2MB)` 内部同步走 InputHandler 解析，主线程被占。
**修复方向**：分片 write，每片 16-32KB，间隔 `requestIdleCallback(() => term.write(chunk))` 或 `Promise.resolve().then(...)`；分片之间让出主线程。

### FLK-2 [H] · MainPane 与 TerminalView 双 ResizeObserver 同观察重叠区
**位置**：`MainPane.tsx:113-133` + `TerminalView.tsx:644-651`
**现象**：拖窗时 chrome（statusbar/tabbar）每帧 layout 更新但 xterm 网格滞后 150ms，体感"周边在跳但终端慢半拍"。
**根因**：MainPane 的 RO 同步 dispatch dims，TerminalView 的 RO trailing debounce。两套节奏不同步。
**修复方向**：去掉 MainPane 的 RO 全局 dispatch（只在第一次 mount 算一次兜底值）；让 TerminalView 单独负责 dims 权威。

### FLK-3 [H] · `fit()` 在容器尚未布局好时执行，初始 cols 极小
**位置**：`TerminalView.tsx:527-531`
**现象**：第一帧出现"窄窄一条字" → 紧接着撑开到正常宽度。
**根因**：mount 时容器 layout 未收敛，第一次 fit 算出 cols=10 之类，xterm 用此尺寸渲染。
**修复方向**：fit 用 `requestAnimationFrame` 延后；fit 后断言 cols/rows 合理值，否则跳过 IPC。

### FLK-4 [H] · Ctrl+滚轮调字号每个 tick 立即广播 SETTINGS_UPDATE
**位置**：`TerminalView.tsx:817-829`
**现象**：滚两下字号阶梯跳跃 + 所有窗口的所有 xterm 同时 re-fit + 抖动。
**根因**：每个 wheel event 直接 IPC，无节流。settings 广播 → 所有窗口跑 `[fontFamily, fontSize, lineHeight]` effect → 全部 re-fit。
**修复方向**：本地 `term.options.fontSize = next` 立即视觉反馈；100ms trailing 才发 SETTINGS_UPDATE。

### FLK-5 [M] · `fontFamily / fontSize / lineHeight` 任一变化都 re-fit
**位置**：`TerminalView.tsx:678-692`
**现象**：跨窗口设置同步时，未被操作的窗口也一起 re-fit + 重绘。
**根因**：settings 广播到所有窗口，所有 TerminalView effect 都跑。
**修复方向**：effect 内对比 `term.options.fontSize === fontSize` 等再决定是否跑 fit；或加 effect 防抖。

### FLK-6 [M] · `data-theme` / `data-window-style` 切换不与 xterm.options.theme 同帧
**位置**：`App.tsx:171-173`、`TerminalView.tsx:671-675`
**现象**：切主题瞬间能看到 1-2 帧"UI 已切但终端区还旧"的半切状态；最大化/还原时 app-root 圆角变化让边缘有反 alias 闪烁。
**根因**：CSS 变量切换、`term.options.theme` 设置、最大化布局重算不在同一 commit。
**修复方向**：把 `term.options.theme` 改用 ref，主题切换时配合 `requestAnimationFrame` 同帧 set；最大化圆角用 transform 而非 border-radius。

### FLK-7 [M] · 多个 evt:* IPC 串行到达，每条独立 reducer commit
**位置**：`src/renderer/store.tsx`（整个 reducer + IPC sync hook）
**现象**：密集事件流下 status bar 上 cwd / pid / status-dot 肉眼可见闪一下。
**根因**：每个 IPC 消息独立 dispatch，React 不跨事件批处理。
**修复方向**：renderer 加一个简单 batcher，连续到达的 evt 在 setTimeout(0) 内合并；或拆 context 让每个 evt 只影响订阅它的子树。

### FLK-8 [M] · ConPTY resize 后整屏重发，xterm 字节级重画可见
**位置**：本质是 KI-001，副作用扩散到视觉
**现象**：拖完窗口的瞬间屏幕"自己重画了一遍"。
**根因**：见 `docs/known-issues.md:KI-001`。
**修复方向**：上游问题，应用层不可彻底修。

### FLK-9 [M] · cwd 漂移图标基于字符串等价，PowerShell PSDrive 前缀触发误闪
**位置**：`TerminalView.tsx:831-834`、`session-manager.ts:1094-1107`
**现象**：每个 prompt 输出后 statusbar 的 ⚠️ 图标偶发闪一下。
**根因**：PowerShell 偶尔报告 `Microsoft.PowerShell.Core\FileSystem::C:\foo`，normalize 不一致导致 currentCwd 字符串比较失败。
**修复方向**：`normalizeCwd` 增加 PSDrive 前缀剥离；`cwdDrifted` 用 path normalize 后比对。

### FLK-10 [L] · `cursorBlink: true` 在 exited 时仍闪
**位置**：`TerminalView.tsx:418`
**现象**：会话已死但光标在闪，用户以为还能输入。
**根因**：xterm 不知道我们的状态机。
**修复方向**：监听 `session.state` 变化，exited 时 `term.options.cursorBlink = false`，可选 `cursorStyle = 'underline'`。

### FLK-11 [L] · `selectOnCopy` 拖选每格写剪贴板（见 XTM-6 的视觉副作用）
**现象**：拖选过程中输入法候选框闪烁。

### FLK-12 [L] · WindowChrome `max-state-changed` 触发 setState → 整 chrome re-render
**位置**：`WindowChrome.tsx:43-62`
**现象**：最大化/还原时图标按钮重绘可见。
**根因**：useState + IPC 监听同步。
**修复方向**：当前可接受。

### RSZ-7 [L] · 第一个 session 创建用 store 默认 dims 120×30 而非首次 fit 真实值

## 9. IPC / Scrollback Replay 协议

### IPC-1 [M] · `claimOwner` 与 owner-changed 事件 ordering 中字节可能短暂迷路
**位置**：`session-manager.ts:577-619`、`ipc.ts:938-945`
**现象**：跨窗口接管的瞬间，新 owner 的 TerminalView 还没 mount 就被发送字节，无人接听。
**根因**：main 在 owner 切换时立即把后续 sessionOutput 推给新 owner，但新 owner renderer mount + 监听注册需要时间。scrollback 兜住，但延迟可见。
**修复方向**：当前可接受；scrollback replay 已涵盖。

### IPC-2 [M] · scrollback 用 base64 string 传 2MB → 2.67MB IPC payload
**位置**：`session-manager.ts:733-740`、`TerminalView.tsx:577-600`
**现象**：接管时 IPC 慢。
**根因**：base64 编码 + V8/IPC 双重序列化。
**修复方向**：Electron `webContents.send` 支持 ArrayBuffer，可直接 transfer。

### IPC-3 [M] · `sendInput` 不校验 ownership
**位置**：`ipc.ts:344-349`
**现象**：renderer 状态落后于 main 时，对一个已不归本窗口的 session 敲键，main 接收并写入 PTY，但输出不会推回本窗口（owner 校验在 output 那一侧）。用户视觉上"打进去了但没回显"。
**根因**：IPC handler 透传，无 ownership 校验。
**修复方向**：sendInput handler 内检查 `session.ownerWindowId === envelope.windowId`，否则返回错误码。

### IPC-4 [M] · sendInput / resize 在 sessionId 不存在时静默
**位置**：`session-manager.ts:665-671, 680-705`
**现象**：renderer 永远不知道自己的请求被丢了。
**根因**：设计选择"竞态时静默"。
**修复方向**：返回 `{ accepted: false, reason: 'session-not-found' | 'pty-exited' }`，renderer 据此 toast / refresh state。

### IPC-5 [M] · `appendScrollback` 在 emit 前的不变量没有单测断言
**位置**：`session-manager.ts:761-772`
**现象**：埋雷：未来改动可能颠倒顺序导致接管时 race 丢 1 行。
**根因**：测试覆盖不到这条偏架构层的不变量。
**修复方向**：加单测：在 emit 触发时检查 `managed.scrollback` 已包含本次字节。

### IPC-6 [M] · evt:session:state-changed 在 active/idle 转移时 broadcast 全 SessionInfo
**位置**：`session-manager.ts:1000-1010`
**现象**：每次 active↔idle 切换 reducer 创建新 sessions Map → 全树 re-commit。
**根因**：单 context 巨型 state；payload 复制整个 SessionInfo（而非 partial diff）。
**修复方向**：partial diff 已经在 changes 字段里，但 reducer 仍生成新 Map。可拆 sessions 到独立 context；或用 useSyncExternalStore 让消费者选择性订阅。

## 10. 输入丢失 / 打不进去（系统性根因汇总）

### TYP-1 [H] · exited session 上 sendInput 静默丢，无视觉反馈
**位置**：`session-manager.ts:665-671`
**现象**：PTY 死后用户继续敲键，xterm 光标还在闪，但什么都没发生。重开窗口 / 关 tab 才"修好"。
**根因**：PTY 退出后 `managed.pty = null`，sendInput 直接 return；renderer 不知道；状态栏的"已退出"小字用户看不到。
**修复方向**：sendInput 返回 `{ accepted: false, reason: 'exited' }` → renderer toast "会话已退出，请关闭 tab 或新建终端"。

### TYP-2 [H] · `pty.write()` 抛错时静默
**位置**：`session-manager.ts:670`
**现象**：Windows ConPTY pipe half-closed 等场景下，每次敲键控制台一行红字，用户无感。
**根因**：pty.write 无 try/catch，异常变成 IPC reject → renderer `.catch(console.error)` 吞。
**修复方向**：包 try/catch；首次失败时把 session 标记为 unhealthy 并 toast。

### TYP-3 [H] · IME composition 期间没有任何防御
**位置**：`TerminalView.tsx:457-505`
**现象**：中文输入半途卡死，候选字浮在屏上按空格无反应。
**根因**：attachCustomKeyEventHandler 没短路 `ev.isComposing`；React re-render 风暴可能中断 composition。
**修复方向**：handler 头加 `if (ev.isComposing || ev.keyCode === 229) return true;`；长期减少 re-render 范围。

### TYP-4 [M] · 主线程被 PTY 高速 chunk 卡住时 sendInput 排队 1-3 秒
**位置**：`session-manager.ts:746-794`（handlePtyData 同步处理）
**现象**：`find /` 期间敲键，几秒后字符突然涌入。
**根因**：handlePtyData 同步跑 concat + parse + base64 + emit，IPC handler 在队列里等。
**修复方向**：handlePtyData 内大 chunk 拆分 + setImmediate 让出；或 sendInput 走更高优先级队列（Electron 不直接支持，需 PTY 写队列改造）。

### TYP-5 [M] · renderer 状态短暂落后 main，对 exited / not-owned session 敲键
**位置**：跨 `session-manager.ts` + `store.tsx`
**现象**：evt:session:exited 在 IPC pipe 排队的几十毫秒内，用户继续敲；keystrokes 全丢。
**根因**：IPC 异步特性。
**修复方向**：见 IPC-4 (sendInput 返回 accepted=false)。

### TYP-6 [M] · 焦点漂走（FOC-1 ~ FOC-6 全部归为此类）
**现象**：用户以为还在 xterm 里敲，实际焦点在 button / body / 隐藏的 input。
**根因**：见 FOC-* 系列。

### TYP-7 [M] · 多行 paste confirm 期间敲键被 dismiss / 误发
**位置**：`TerminalView.tsx:336-348`
**现象**：confirm 弹出期间用户烦躁按 Enter 几次：第一个 dismiss confirm，后续 Enter 直接送 PTY 把刚粘的命令立即提交。
**根因**：`window.confirm` 是阻塞模态，期间键事件被 OS 排队，dismiss 后顺序 replay。
**修复方向**：用自绘 Modal + 焦点 trap 替代 `window.confirm`。

### TYP-8 [M] · 拖文件 drag-over 期间焦点漂移
**位置**：`TerminalView.tsx:778-814`
**现象**：拖完文件回头敲键偶发无反应。
**根因**：dragover/drop 期间焦点状态不稳定。
**修复方向**：handleTerminalDrop 已 `termRef.focus()`；可在 dragenter 时 preview 一下也 focus。

### TYP-9 [L] · 多 TerminalView 实例 race（乐观接管回滚）
**位置**：`MainPane.tsx:496-509`
**现象**：claim 失败回滚瞬间 displayable session 跳变 → TerminalView key 变 → 旧 dispose + 新 mount → 期间敲键丢。
**根因**：乐观更新的常见副作用。
**修复方向**：claim 失败时 toast 提示"会话已被其他窗口占用"，不立即回滚 view，等用户主动操作。

### TYP-10 [L] · 双击 EmptyPathState 模板按钮可能创建两个 session
**位置**：`MainPane.tsx:200-229`
**现象**：连点两次新建按钮，IPC race 让 setCreating 兜不住。
**根因**：setState 异步。
**修复方向**：button disabled 改用 ref 而非 state，或加 1 秒级 cooldown。

### TYP-11 [L] · 搜索栏 Esc 关闭后第二个 Esc 仍被吞（ref 异步更新窗口）
**位置**：`TerminalView.tsx:500-503`
**现象**：连按两个 Esc，第二个本应送 PTY（vim 等），但被 searchVisibleRef 旧值吃。
**根因**：setState → ref 更新跨帧。
**修复方向**：handleCloseSearch 内同步 `searchVisibleRef.current = false`。

| 拖文件 drop | handleTerminalDrop（非典型，但路径相似） | `TerminalView.tsx:789-814` | 否 |

### CPB-P1 [H] · 用户主诉根因 · handlePaste 完成后**永不调用 `term.focus()`**

**位置**：`TerminalView.tsx:327-357`
**现象**：粘贴后开始打字，键盘事件没有目的地，xterm 完全无反应。必须关窗口重开（或点终端区一次）才恢复。
**根因**：
- handlePaste 末尾只有 `await window.api.invoke(SESSION_SEND_INPUT, ...)`，没有 focus 归还。
- 同文件 `handleCloseSearch:385` 和 `handleTerminalDrop:811` 都做了 `termRef.current?.focus()`，handlePaste 漏了。
- 单行粘贴且 focus 还在 xterm 的场景下表现不出问题；只要焦点曾经被偷走（confirm / 右键菜单 / 任何同步模态），就再也回不来。
**修复方向**：handlePaste 末尾（finally 块内）`termRef.current?.focus()`。1 行解决 80% 用户主诉。同样在 handleCopy 末尾也加一行（防右键菜单复制后丢焦点）。

### CPB-P2 [H] · `window.confirm` 是 Electron 同步原生 dialog，焦点归还不可靠

**位置**：`TerminalView.tsx:341-346`
**现象**：多行粘贴每次必弹 confirm；确认后即便 handlePaste 加了 focus 归还也未必生效，因为 Chromium 在 confirm dismiss 后**到底把焦点还到哪个 element 是未定义行为**，Electron 对此无官方保证。
**根因**：
- `window.{alert,confirm,prompt}` 在 Electron renderer 调起 Chromium 同步模态。
- modal 期间 JS 执行被 block，包括 React 重渲染。
- dismiss 后 Chromium "通常"把焦点还给文档 body，**而不是 modal 弹出前的 activeElement**。
- 业界共识：Electron 应用避免使用 `window.confirm`，改用：
  - main 端 `dialog.showMessageBox` + IPC（异步，焦点行为由 main 控制）
  - 或自绘 Modal + React 焦点 trap
**修复方向**：把这个 confirm 改成自绘 Modal（项目里 ContextMenuProvider / ToastProvider 已经是自绘风格，复用其样式系统），关闭时主动 `termRef.focus()`。约 50 行新组件 + 改 handlePaste。

### CPB-P3 [M] · 单行带尾换行的剪贴板 (`ls\n`) 会触发 confirm 误判

**位置**：`TerminalView.tsx:334-336`
**现象**：从浏览器/IDE 复制一行命令（很多编辑器复制时自带行尾 \n），粘贴时仍弹 confirm。用户被打扰，体感"为什么单行也要确认"。
**根因**：`normalized.split('\n').length` 对 `"ls\n"` 返回 `['ls', '']` → length=2 → 走 confirm 分支。
**修复方向**：line 计算改为 `normalized.replace(/\n$/, '').split('\n').length`；或 `lineCount > 1 && normalized.replace(/\n$/, '').includes('\n')`。

### CPB-P4 [M] · 大粘贴（>100KB）触发 renderer 短暂冻结

**位置**：`TerminalView.tsx:961-966`（encodeStringToBase64） + `TerminalView.tsx:349-353`
**现象**：粘贴 5MB+ 内容时窗口短暂无响应（500ms-2s），期间无法 cancel。
**根因**：
- `encodeStringToBase64` 是 JS for 循环 + 字符串 concat：`for (const byte of utf8) binary += String.fromCharCode(byte)`。1MB 文本约 100ms 主线程。
- `await window.api.invoke(SESSION_SEND_INPUT, { data })` 传 base64 字符串过 IPC bridge，也是 sync serialize。
**修复方向**：
- 用 `btoa(String.fromCharCode.apply(null, utf8))` 短文本快；长文本切片 base64 后拼接。
- 长期方案是 main 端 IPC 接受 ArrayBuffer（transferable），避开 base64 全开销。
- 加 size 阈值检测：>1MB 时再弹 confirm "粘贴 5.2MB 内容到终端可能让 shell 卡住，继续?"

### CPB-P5 [M] · 粘贴过程 async，与并发打字 race 产生乱序

**位置**：`TerminalView.tsx:327-357` 的多个 await
**现象**：用户在 `await readClipboardText()` 期间（10-30ms IPC 往返）继续敲键，键盘字符通过 dataHandler 直接 sendInput 到 PTY；之后粘贴内容也到 PTY，**字节流交错**。
**根因**：handlePaste 是 async 但 dataHandler 不被它阻塞；两条 sendInput 路径并行。
**修复方向**：进入 handlePaste 后设 `pasting=true` ref，dataHandler 检测到 pasting 时把字符 buffer 起来等粘贴完成再 flush；或干脆 paste 期间禁用键盘输入（视觉上灰一下）。

### CPB-P6 [M] · 右键菜单"粘贴"选项：ContextMenu close 不归还焦点

**位置**：`MainPane.tsx ContextMenuProvider`（与 FOC-5 同源） + `TerminalView.tsx:753-755`
**现象**：右键 → 选"粘贴" → 菜单关闭 → 焦点到 body → 即便单行粘贴成功内容到 PTY，**用户也无法立即打字**。
**根因**：ContextMenuProvider 未实现 focus return（ARIA modal 标准模式）。
**修复方向**：见 FOC-5；如果先修 CPB-P1（handlePaste 末尾 focus），间接修这一处。

### CPB-P7 [M] · 粘贴内容不做任何 sanitize，可被注入 ANSI / 控制字符

**位置**：`TerminalView.tsx:349`（直接 encode 原始 text）
**现象**：从恶意网页复制看似无害的 "Hello World"，实际剪贴板里是 `Hello\x1b[2J\x1b[1;1H\x1b]0;owned\x07World`：
- `\x1b[2J\x1b[1;1H` 清屏并重置光标 → 视觉欺骗
- `\x1b]0;owned\x07` 改 session title → 我们的 OSC parser 会更新 displayName（如果 manuallyRenamed=false）
- 更恶意：嵌入 `\x1b[200~` 但不闭合 `\x1b[201~` 可能让某些 shell 进入 bracketed paste 模式锁死
**根因**：粘贴内容 1:1 透传给 PTY，没有任何过滤。
**修复方向**：
- 至少剥离 `\x1b`、`\x07` 这类 C0 控制字符（保留 `\t` `\n` `\r`）。
- 多行 confirm 的 preview 也应做同样 sanitize 以免预览里也被欺骗。
- 长期方案：识别剪贴板有 ANSI 转义时单独弹强警告。

### CPB-P8 [M] · 缺少 bracketed paste 协议支持

**位置**：N/A（当前未实现）
**现象**：粘贴多行到 bash / zsh / fish / PowerShell ReadLine，shell 立即执行每行（不等用户编辑）。这就是为什么我们需要 confirm。Bracketed paste 协议（`\x1b[200~`...`\x1b[201~`）能让支持的 shell 把粘贴内容当 literal 字符而非命令，**用户能编辑后再 Enter**。
**根因**：没有发起 / 协商 bracketed paste 模式。xterm-256color terminfo 默认支持，但需要 shell 端启用且发送 wrapper。
**修复方向**：
- xterm.js 5.x 支持 `term.write('\x1b[?2004h')` 启用 bracketed paste 模式（通常 shell 收到后自动响应）。
- handlePaste 内自动包裹：`'\x1b[200~' + text + '\x1b[201~'`。
- 用户感知：多行粘贴**不再被立即执行**，可在 prompt 内编辑后回车。
- 这同时让 CPB-P3（单行带 \n 误判）的影响降到无（即便误判，bracketed paste 也防止意外执行）。

### CPB-P9 [L] · Shift+Insert 路径里 attachCustomKeyEventHandler 没短路 IME 组合状态

**位置**：`TerminalView.tsx:494-497`
**现象**：中文输入法 composing 状态下按 Shift+Insert，handler 仍然触发 handlePaste；可能与 IME 状态冲突。
**根因**：handler 头部没有 `if (ev.isComposing) return true`（也是 TYP-3 / FOC-7 同一根问题的延展）。
**修复方向**：见 TYP-3。

### CPB-P10 [L] · 多模态时序 race：Ctrl+Shift+V 按下后 modal 弹出前的修饰键 keyup

**位置**：`TerminalView.tsx:457-505`
**现象**：用户按 Ctrl+Shift+V 后立刻松开 Ctrl/Shift，keyup 事件到达期间 confirm 还没弹出（handlePaste 异步），keyup 被 attachCustomKeyEventHandler 透传（因为 `ev.type !== 'keydown'`），无副作用。**但**：如果用户在 confirm 显示期间按 Enter，第一个 Enter dismiss confirm，**后续打字键** 在 modal 关闭后立刻 fire → 见 CPB-P5。
**根因**：modal 期间 OS event queue 排队，dismiss 后 replay。
**修复方向**：见 CPB-P2 (干掉 window.confirm)。

### CPB-C1 [H] · handleCopy 也不归还焦点

**位置**：`TerminalView.tsx:318-325`
**现象**：右键菜单选"复制"后焦点丢失（菜单关闭→body），打字无反应。比粘贴的场景频率低（用户复制后多半是切到别处粘），但闭环上同病。
**根因**：handleCopy 末尾没有 `termRef.current?.focus()`。
**修复方向**：handleCopy 末尾加 `termRef.current?.focus()`。1 行。

### CPB-C2 [M] · selectOnCopy 拖选时**每移动一个 cell 就 IPC 写一次剪贴板**

**位置**：`TerminalView.tsx:716-726`
**现象**：
- 拖选 50 字符 → 触发 50 次 `writeClipboardText` IPC。
- Windows 剪贴板 OLE 锁短暂占用，输入法候选框、Quicker / Ditto / ClipboardFusion 等剪贴板管理器对每次写都弹通知 → 用户屏幕闪烁。
- 50 次 IPC 在 main 端 50 次 `clipboard.writeText`，每次都是 OS 调用。CPU 浪费。
**根因**：`onSelectionChange` 在 mousemove 选中新 cell 时持续 fire，handler 直接 IPC。
**修复方向**：trailing debounce 100ms；或改用 `mouseup` 时一次性写。**这条已经在原备忘录 XTM-6 里**，此处重申。

### CPB-C3 [M] · Ctrl+C 在"误选状态"下不发 SIGINT

**位置**：`TerminalView.tsx:483-486`
**现象**：用户运行死循环 → 想 Ctrl+C 终止 → 但终端区因为前一次拖选还有残留 selection → handler 走"有选区 → 复制"分支，不发 ^C → **程序继续跑**。
**根因**：判断 `hasSel = !!termRef.current?.getSelection()` 不区分"用户有意选区"和"残留未清"。
**修复方向**：
- Ctrl+C 在 hasSel 时复制后立即 `term.clearSelection()`，下一次 Ctrl+C 就发 ^C。
- 或：Ctrl+C 永远发 ^C（用户用 Ctrl+Shift+C / Ctrl+Insert 复制）。这符合 Linux 终端习惯，但与 Windows 用户预期不同，需评估。

### CPB-C4 [L] · 复制使用 LF 不带 CR，粘到 Windows 程序丢换行

**位置**：xterm 内部 `getSelection` 行为
**现象**：从 Marina 复制多行命令粘到 Notepad 显示为单行（Notepad 严格要求 CRLF）。
**根因**：xterm 默认换行符是 LF。
**修复方向**：handleCopy 内 `text.replace(/\n/g, '\r\n')` 后再 writeClipboardText（Windows 平台）。

### CPB-C5 [L] · writeClipboardText 异步 fire-and-forget，连续写时序未保证

**位置**：`src/renderer/clipboard.ts:32-46`
**现象**：selectOnCopy 短时间连续写 N 次，OS clipboard 最终值是哪一次的不保证（虽然实际 IPC 串行排队，但极快连写下 main 端 clipboard.writeText 也可能被 OS 节流到最后一次）。罕见但行为不可预测。
**根因**：fire-and-forget + 无序号同步。
**修复方向**：见 CPB-C2 节流根治。

### CPB-DROP-1 [L] · 拖文件 drop 后调 `termRef.focus()`，对比 handlePaste 是"开发者偶尔记得"的不一致

**位置**：`TerminalView.tsx:811`
**现象**：drag-drop 后 focus 自动归还，但 paste 不归还。代码风格不一致，将来维护时容易再次漏掉。
**根因**：缺乏统一约定。
**修复方向**：抽 `function ensureTerminalFocus(): void { termRef.current?.focus(); }`，所有"会动 focus 的副作用 callback"末尾统一调用。

### 复合症状链路（解释用户主诉）

用户报告"粘贴后打不进字，必须关窗口"的精确触发：

```
[Step 1] 用户在 xterm focus 状态按 Ctrl+Shift+V，剪贴板内容 > 1 行
[Step 2] attachCustomKeyEventHandler 拦截，handlePaste() 异步启动
[Step 3] readClipboardText() IPC ~20ms，焦点仍在 xterm helper-textarea
[Step 4] window.confirm 弹出 (CPB-P2)：
         - 焦点被 Chromium native dialog 接管
         - JS 主线程 block 直到 dismiss
[Step 5] 用户点击"确定"：
         - confirm 关闭
         - Chromium 把焦点还到 document.body（不是 xterm.helper-textarea）
[Step 6] handlePaste 继续：
         - encodeStringToBase64(text) → CPU 占用
         - sendInput IPC → 粘贴内容成功到 PTY
[Step 7] handlePaste 结束，**没有 termRef.focus() 兜底** (CPB-P1)
[Step 8] 用户开始打字：
         - 键盘事件目标 = document.body
         - xterm.helper-textarea 不接收 → onData 不 fire
         - PTY 收不到任何字节
         - 体感"完全打不进"
[Step 9] 用户关闭窗口重开：
         - 新 BrowserWindow，焦点初始空
         - 用户更倾向于点击终端区 → 焦点回 xterm
         - 恢复
```

**最小修复**：handlePaste 末尾加 `termRef.current?.focus()` 兜底 [CPB-P1]。
**根治**：把 window.confirm 换成自绘 Modal [CPB-P2]，并启用 bracketed paste 协议消除多行 confirm 的必要性 [CPB-P8]。

### 推荐修复顺序

1. CPB-P1 + CPB-C1（各 1 行）→ paste/copy 兜底 focus。**这一项立刻消除用户主诉**。
2. CPB-P3（2 行）→ 单行带 \n 不再误触发 confirm。
3. CPB-C3 → Ctrl+C 残留选区不发 SIGINT 的死循环陷阱。
4. CPB-P4 + CPB-C2 → 大粘贴 / 选中即复制的性能修复。
5. CPB-P2 → 自绘 Modal 替换 window.confirm（架构改动）。
6. CPB-P8 → bracketed paste 协议（架构改动，但用户体验飞跃）。
7. CPB-P7 → ANSI 控制字符 sanitize（安全增强）。

**根因**：node-pty disposable dispose 与 onData 不同步。
**修复方向**：handlePtyData 头部加 `if (!this.sessions.has(managed.info.id) || !managed.pty) return;`，显式 guard 兜底。

### CPT-4 [L] · `useConpty: true` 强制 → Win10 < 1809 spawn 失败但错误码不引导用户升级
**位置**：`session-manager.ts:429-443`
**现象**：旧系统用户看到 "PtySpawnFailed"，不知道是 OS 版本问题。
**根因**：错误消息没区分原因。
**修复方向**：catch 后根据 err.message 或 process.getSystemVersion 给具体提示。
