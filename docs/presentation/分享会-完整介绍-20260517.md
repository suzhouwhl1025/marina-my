## 1 · 真实痛点:AI agent 时代,旧终端模型崩了(2:00–6:00)

### 1.1 一个典型的崩盘场景(讲故事,30 秒)

> 你在 Project A 里 `npm run dev`,B 里让 Claude Code 改 bug,C 里跑 pytest。三件事三个窗口。
> 老板叫你切显示器,你**关错一个窗口**——Claude 半小时的上下文没了。
> 你想用 tmux 防一手,但你不熟 tmux,你同事也不熟 tmux。

### 1.2 这不是"工具做得不好",这是**设计前提**的问题

主流终端(WT / Tabby / Hyper / iTerm2)都继承 **1970 年代 VT100 物理终端的隐喻**:

> Session = 一个 shell 进程 + 一块屏幕缓冲区 = 一台"虚拟终端机"

**这个定义带来三条副作用:**

- 关窗 → 进程死(窗口持有 PTY)
- 组织维度是 profile / tab,**不是项目**
- 你打开 Windows Terminal 看见 "PowerShell"、"PowerShell"、"Bash"——**你根本不记得哪个 tab 是哪个项目的**

### 1.3 AI agent 把这个痛**放大了一个数量级**

不再是几秒钟能重启的 dev server,而是:

| 跑的是什么 | 关错窗口的代价 |
|---|---|
| Claude Code / Codex / Aider 长对话 | 整段上下文丢失 |
| 半小时的回归 / 集成测试 | 全部重来 |
| docker compose 起的一堆服务 | 所有客户端重连 |

**关键句:**
> AI agent 的工作方式天然是"一个项目、一条长任务、跑几十分钟",而传统终端的设计前提"shell 跟着窗口生"——**跟这套工作方式是冲突的。**

## 2 · Marina 是什么:三件事(6:00–9:00)

> 一句话:**一个以"路径"为中心、对 AI agent 友好的桌面终端管理器,会话独立于窗口存活。**

具体做了三件事:

### 2.1 关窗不杀任务

- 任何窗口都是"任务的查看器",任务跑在 Electron 主进程(守护)里
- 关掉所有窗口,应用沉到托盘,任务继续跑
- 重开窗口,所有 session 还在,scrollback 完整

**演示路径(如果现场放图/录屏):**
开三个 session → 关整个窗口 → 任务管理器看 `Marina.exe` 还在 → 点托盘 → 窗口回来,session 全活着。
**那一刻,观众的表情就是这个产品的卖点。**

### 2.2 按"项目路径"组织,不按"shell 进程"组织

- 侧栏分三栏:**收藏(用户管)/ 临时(自动)/ 最近(自动 LRU)**
- session 挂在路径下,UI 上你看到的是 `~/projects/auth`,不是 `PowerShell #3`
- 路径靠**点击**,不靠 `cd` 打字 —— `cd` 是 1971 年的设计

### 2.3 多窗口共享会话池

- 没有"主窗口",所有窗口平等
- 任一 session 同一时刻最多 1 个 Owner Window
- 在 Window 2 看到 Window 1 的 session 是灰色的,点一下系统会聚焦到 Window 1

### 2.4 对位坐标(放对比表)

| | WT | Tabby | Wave | Warp | **Marina** |
|---|:-:|:-:|:-:|:-:|:-:|
| 关窗不杀会话 | ❌ | ❌ | ✅ | ❌ | ✅ |
| 路径为中心组织 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 自动 cwd 跟踪 + 漂移告警 | ❌ | ❌ | ✅ | ✅ | ✅ |
| 多窗口共享会话池 | ❌ | ❌ | ❌ | ❌ | ✅ |
| Win 原生为先 | ✅ | ✅ | ❌ | ❌ | ✅ |
| 鼠标优先,无强制快捷键 | ❌ | ❌ | ❌ | ❌ | ✅ |

**关键句(用来收这一段):**
> Wave 证明了"session 不死"是可能的;tmux 证明了"session 该独立于 UI";iTerm2 给了我们 OSC 1337。
> **Marina 把这三件事第一次同时塞进一个原生 Windows GUI 里,而且对 AI agent 工作流是一等公民。**

## 3 · 设计哲学:两条底层判断 + 四条工程原则(9:00–14:00)

> 这一段是为什么我说 Marina 不是"又一个 Warp"。

### 3.1 底层判断一:Session 不是"虚拟终端机",是"一件事"

**旧定义(VT100):** session 是容器,装着 cwd/env/history/进程。容器丢了等于灾难,所以 tmux/screen 一整套工具都在**护住 session**。

**Marina 的定义:** Session 是 **path 下的一个任务实例**。

- 它的身份不是"第 3 个 shell",是"my-blog 下的 dev server"
- 它的归属是 path,不是窗口
- 它是**廉价的** —— 用完关掉,要重跑就开新的

**跟 tmux 的关键差别(一定要讲):**
> tmux 让 session 不死,**Marina 让 session 不重要**。
> 真正重要的是 path —— path 才是用户会命名、收藏、记忆的东西。

### 3.2 底层判断二:CLI 是 UI 协议,不是命令接口

打开任何现代开发者的终端,真正"敲命令"的时间占比都在快速下降。更多的时间是:

- 看 `npm run dev` 的输出
- 跟 Claude Code 对话
- 看 `docker compose` 状态行
- 回应 `apt` / `rm -i` 的 y/n

**这些不是命令输入,是"使用文本 UI 应用"。**

**这条洞见对 Marina 是反向的** —— 它告诉我们**不该做什么**:

- ❌ 不自己实现 prompt(像 Warp 那样)
- ❌ 不内嵌 AI 补全
- ❌ 不拦截命令做 magic
- ❌ 不重做 shell

> Marina 的位置是**所有 CLI UI 应用之下的容器编排器**,不是其中任意一个。
> xterm.js 已经把"渲染 CLI 这种 UI 协议"做到工业级了,我没有必要也没有能力重做。

### 3.3 推论:四条工程原则

| 原则 | 一句话 |
|---|---|
| Path 稳定 / Session 廉价 / UI 临时 | 收藏 path,别珍惜 session,别怕关窗口 |
| 不让用户输入路径,只让用户点路径 | `cd` 是易错动作,文件夹选择器 + 拖拽是答案 |
| 用户决策最少化 | 状态机自动跑分类、自动跟 cwd、自动 resize |
| 窗口与应用解耦 | 应用 ≠ 窗口,关窗免费 |

**第 4 条 Linux 上有例外**(GNOME 没 system tray,Linux 版回到"普通桌面 app",关最后一个窗口 = 退出,带二次确认 modal) —— 这是 ADR-013,顺手提一句"跨平台不是 ifdef 复刻"。

### 3.4 一句话归纳(留观众脑子里的最重要一句)

> **Path 是项目。Session 是项目里的一件事。Window 是看这件事的临时镜头。**
> **CLI 工具自己负责呈现,Marina 负责让它们好好活着、好好被找到。**

### 4.3 为什么能这么快(说服力的关键)

**不是"AI 抡大锤瞎写",而是:**

1. **下笔前先写软件定义书** —— 一份从产品定位、设计哲学、状态机到 IPC 协议全覆盖的 spec,**每个有争议的决策都落 ADR**(已有 ADR-008 ~ 014),不靠"我觉得"。
2. **每个 checkpoint 闭环为"实现 → 自测 → 用户测试单 → 勘误 pass"**,feat:fix 比 ≈ 1 : 0.52,反映明显的"实现 → 勘误"节奏。
3. **错误分类法**:渲染审计备忘录里 12 个错误类前缀(CPB-P/C、FOC、TYP、IPC、CRA、OSC、PER、SEC、FLK、RSZ、XTM、STM、CUR)全部分类编号,每个 fix 引这个编号。
4. **AI 辅助 ≠ AI 代写**:架构、抽象边界、ADR 全是人脑判断,AI 是肌肉,不是大脑。

### 4.4 跨平台不是 ifdef

`src/main/platform/{windows,linux,macos}.ts` 适配层从 CP-1 就留好了 —— Linux 不是"复刻 Windows 行为",而是**承认 GNOME 没有 system tray** 这个事实,引入 `lifecycleModel` 抽象(`tray-resident` / `dock-resident` / `no-persistence`),三平台共享同一组"最后窗口关闭确认" UI 组件。**这是 ADR-013。**

## 6 · Q&A 备弹(留 3–5 分钟)

预期被问 + 准备好的回答口径:

**Q1:跟 Wave Terminal 区别?**
> Wave 也做了 session 持久化,但它是"块状 AI 终端"的定位,组织维度还是 workspace + tab。Marina 把**路径**抬到一等公民,而且是**原生 Windows 优先**。Wave 是 Electron + AI-heavy;Marina 是 Electron + AI-agent-friendly,但本体上不做 AI。

**Q2:Electron?不重吗?**
> 接受。换 native 不会让 ConPTY 限制消失,也不会让"会话不死"自动出现。Electron 的成本买的是 7 天能出 alpha,18k 行 TS 跑遍三平台。性能瓶颈在 xterm.js 渲染,WebGL renderer 已开,Linux 上还专门做了 perf pass(看 commit `1dbc8bc`)。

**Q3:为什么不直接做 tmux GUI?**
> tmux 的 session 是"长期持有上下文的容器,用户给它取名"。Marina 的 session 是**任务实例,默认不命名**。哲学不一样,做出来不会是同一个东西。

**Q4:KI-004 不可怕吗?Marina 崩了 PTY 全死。**
> 怕。这是 V1 的明面债。V2 候选方案是 daemon 架构 —— 把主进程拆成 background daemon + UI viewer。但在 crash/upgrade 真的成为高频痛点之前不投入工程,先把哲学层稳住。

**Q5:为什么 Windows 优先?**
> 因为我自己在 Windows 上工作。也因为 Windows 上能打的对手最弱 —— WT 四年没做关窗不杀会话,这是个**洞**。macOS / Linux 上 iTerm2 / Wave 已经在了,做就是赤膊上阵,做不出差异化。

**Q6:开源协议?能商用吗?**
> MIT。能。

## 8 · 200 字摘要(用于活动页 / 转发文案 / 朋友圈)

> **Marina** 是一个以路径为中心、对 AI agent 友好的 Windows 桌面终端管理器。
>
> 它把传统终端"关窗即死"的设计前提整个推翻:会话跑在守护进程里,关掉所有窗口任务也继续,所有窗口完全平等且共享同一会话池;侧栏按项目路径(而不是 shell 进程)组织,用鼠标点击代替 `cd` 打字。底层判断是两条:**session 不是虚拟终端机,是 path 下的一个任务实例;CLI 是 UI 协议,不是命令接口。** 因此 Marina 不重造终端,只做上面一层 —— 任务的组织、持久化、状态可见性。
>
> 项目由一人在 **4 天 19 小时**内完成 alpha:**98 commits / 50,158 行净增代码 / 14 个测试文件 / 34 份设计文档 / Rust 子项目 / Win11 新右键菜单 COM 组件**,目前已发到 beta.4,开源 MIT。
