## 1. Marina 是什么

**Marina 是一个以"路径"为中心、对 AI agent 工作流友好的桌面终端管理器。**
当前 Windows 10/11 是 Tier 1,Ubuntu 22.04(GNOME)与 Fedora/CentOS 是 Tier 2,macOS 在路线图上等社区贡献。

它**不是**一个传统终端模拟器的迭代,而是对"终端会话应该如何被管理"这个问题的重新回答:

- 传统终端把"窗口生命周期"和"shell 进程生命周期"绑定 —— **关窗即死**
- 传统终端以 profile / tab 为组织维度 —— 用户需要靠**记忆和手动 `cd`** 来导航
- 传统终端假设用户是命令行专家 —— 重度依赖**快捷键和配置文件**

Marina 反其道而行:

- ✅ **会话独立于 UI 存活** —— 关掉所有窗口,会话仍在守护进程里运行,直到用户主动结束
- ✅ **路径作为一等公民** —— 所有终端按"工作位置"组织,不再按 profile 或 project
- ✅ **GUI 优先** —— 用户从不需要手敲路径,从不需要记忆快捷键,所有操作都可以鼠标完成

> 如果你要一句话告诉同事:**Marina 就是 Wave 的会话持久化 + tmux 的 session 独立 + 原生 Windows 的归属感,但组织维度从"shell"换成了"路径"。**

## 2. 为什么需要 Marina:三个真实痛点

### 2.1 痛点一 · 关错窗口 = 前功尽弃

你大概率遇到过:

- Project A 里 `npm run dev`,Project B 里 Claude Code 改 bug,Project C 里跑 pytest
- 切显示器、整理桌面、关错一个窗口 —— **Claude 半小时的对话上下文没了 / dev server 停了 / 测试结果丢了**
- 想用 tmux 防一手,但 tmux 学习成本高,而且你 90% 的活在本地、不想为这 10% 学一套快捷键

这不是 Windows Terminal / Tabby / Hyper 做得不好,这是它们的**设计前提**决定的 —— 在它们的世界里,**终端就是"一个 shell 进程加一块屏幕",关窗它就死**。

### 2.2 痛点二 · 你根本不记得哪个 tab 是哪个项目

打开 Windows Terminal 的 tab 列表,看到 "PowerShell"、"PowerShell"、"Bash"、"PowerShell"。

你能立刻分清吗?**因为它们都叫 shell 的名字,而不是项目的名字。**

主流终端的组织维度是 **profile / tab**,不是 **project / path**。这是 1970 年代物理终端时代留下的隐喻。

### 2.3 痛点三 · AI agent 把这个痛放大了一个数量级

如果只是 dev server,重启一下也就几秒钟。但现在终端里跑的可能是:

| 跑的是什么 | 关错窗口的代价 |
|---|---|
| Claude Code / Codex / Aider 长对话 | **整段上下文丢失**,要重新告诉它项目背景 |
| 半小时的回归 / 集成测试 | **全部重来**,半小时白干 |
| docker compose 起的一堆服务 | **所有客户端重连**,环境重新热起来 |
| 5 个 agent 在 5 个不同项目里干活 | **你根本不记得哪个在等你回话** |

> AI agent 的工作方式天然是"一个项目、一条长任务、跑几十分钟",而传统终端的设计前提"shell 跟着窗口生" —— **跟这套工作方式是冲突的**。

Marina 就是为这个冲突生的。

## 3. 核心功能全景

### 3.1 会话生命周期(关窗不杀任务)

- 任意窗口都是"任务的查看器",任务跑在 **Electron 主进程(守护进程)** 里
- 关掉所有窗口 → 应用沉到系统托盘,任务继续跑、scrollback 完整保留
- 重开窗口(单击托盘 / 双击桌面图标)→ 所有 session 还在原位
- 已退出的 session 灯显灰色 ⚫,scrollback 仍可见,**无时限自动过期**(详见 ADR-008)
- 退出码区分:`exitCode=0` 叠 ✓ / 非零叠 ✗ / 强杀只灰底

### 3.2 路径管理(三栏侧栏)

- **收藏** —— 用户主动加入,永久保留,可重命名、拖拽改序、设默认模板
- **临时** —— 当前有 session 在跑、且不在收藏的路径,**完全自动**
- **最近** —— 历史上开过 session、当前不在收藏也不在临时的路径,**自动 LRU 30 个**
- **系统(beta.4 新增)** —— 桌面 / 主目录 / 临时目录,默认显示,可关
- 加入方式:文件夹选择器、Explorer 拖拽到侧栏、Explorer 右键、`+` 按钮 —— **不允许手敲路径**
- 同名末级智能去重:`proj1/src` 与 `proj2/src` 自动补父目录区分

### 3.3 启动模板

- 内置:**Shell**(系统默认)、**Claude Code**、**Codex**、**OpenCode**
- 用户自定义:命令、参数、环境变量(默认 blur 遮罩)、退出后行为
- 每个收藏路径可指定**默认模板**,双击直接启动跳过选择

### 3.4 多窗口共享会话池

- 任意时刻可开任意多个窗口(包括 0 个)
- **所有窗口完全平等**,不存在"主窗口"
- 任一 session 同一时刻最多 1 个 Owner Window
- 非 Owner 窗口看见同样的 session 列表,但显示为灰色 + ↗ 图标
- 点击灰色 tab → 系统聚焦到真正持有它的窗口
- 关掉任意窗口 → 完全不影响其他窗口和任何 session

### 3.5 CWD 实时跟踪(OSC 1337)

- PowerShell 和 cmd.exe 走 shell hook 注入,**不污染用户 dotfiles**
- session 内 `cd` 不会让 session 在 UI 上换 path,而是在标签上显示 ⚠️ + 真实 cwd tooltip(详见 ADR-008)
- 这条设计的本质:**path 是 session 创建时的地点,是用户对"它属于哪个项目"的判断,不该被 `cd` 偷走**

### 3.6 终端体验

- **xterm.js** 渲染,WebGL renderer 默认开启,DOM 回退
- 支持 ANSI 颜色、Unicode 11、URL 自动识别可点击
- **Ctrl+F 搜索** —— 显示 `current/matches` 命中数,Enter 下一个 / Shift+Enter 上一个 / Esc 关
- **多行粘贴确认** —— 含 `\n` 时弹原生 confirm,显示行数 + 200 字预览,避免误把脚本喂给 shell
- **Bracketed paste + ANSI 注入警告** —— 防止恶意剪贴板内容执行命令
- **Ctrl + 鼠标滚轮**调字号
- 字体下拉**枚举系统真实安装字体**(走 `window.queryLocalFonts()`),顶部置顶推荐字体(Cascadia Mono / JetBrains Mono / Consolas / 霞鹜文楷 Mono)
- 终端工具栏(beta.4 新增):复制全部 scrollback / 清屏 / 搜索 / 简易模式切换 4 个按钮

### 3.7 主题与外观

- 内置 **11 套主题**(beta.4 新增 4 套):
  - 深色:Rose Pine(默认)、Rose Pine Moon、Cutie、Business、Ubuntu、Windows Terminal、One Dark Pro、Dracula、Tokyo Night、Catppuccin Mocha
  - 浅色:Rose Pine Dawn
- CSS variables 驱动,**14 个语义变量**,主题切换即时生效
- xterm.js 主题与 UI 主题同步切换
- 双标题栏样式:**Windows**(控件在右)或 **macOS**(红绿灯在左)
- UI 系统图标统一 lucide-react(用户数据中的 emoji 保留)
- 所有浅色主题 ANSI bright 集对比度 ≥ 4.5:1

### 3.8 设置系统

- **实时应用,无保存按钮** —— 改完立即生效,所有窗口同步
- 导出 / 导入 JSON 配置(导出可选公开/含敏感)
- 导入是 **in-memory replace**,不重启应用、不杀 session(ADR-009)
- 致谢页 + 关于页含版本号 / git hash / 构建时间

### 3.9 系统集成

- **系统托盘** —— 常驻,动态图标(active/default 两态),菜单含"正在运行的会话"子菜单
- **Explorer 经典右键** —— "在 Marina 终端中打开",HKCU 注册表写入
- **Win11 新右键菜单** —— Rust + IExplorerCommand COM + Sparse Package(MSIX),装/卸载提示重启
- **数据目录** —— `%APPDATA%\Marina\`(Windows)/ `~/.config/Marina/`(Linux)
- **持久化日志** —— daily + size 滚动,设置页一键打开日志目录
- **单实例锁** —— 双击桌面图标聚焦已有窗口,不无脑新开

### 3.10 AI 助手(beta.4 新增)

- 设置页新分类(Brain icon),支持 **Anthropic / OpenAI** 两个 provider
- apiKey 输入(显示/隐藏切换)、model 输入、测试连接按钮
- **LLM 状态复核**:`active→idle` 跃迁前可让 LLM 看一眼 scrollback 复核,避免 Vite 等长输出工具被误判 idle

### 3.11 国际化(beta.4 新增)

- **中英双语 i18n**,~80 个 key 覆盖关键 UI
- 设置可切"跟随系统 / 中文 / English"
- 自写轻量 i18n 框架(`src/shared/i18n.ts`),不引第三方依赖

## 4. 它跟 Windows Terminal / Tabby / Wave / Warp 有什么不一样

| 维度 | Windows Terminal | Tabby | Wave | Warp | **Marina** |
|---|:---:|:---:|:---:|:---:|:---:|
| 关窗不杀会话 | ❌ | ❌ | ✅ | ❌ | ✅ |
| 路径为中心组织 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 自动 CWD 跟踪 + 漂移告警 | ❌ | ❌ | ✅ | ✅ | ✅ |
| 多窗口共享会话池 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 关窗时不杀任何 session | ❌ | ❌ | ✅ | ❌ | ✅ |
| 为 AI agent 工作流专门设计 | ❌ | ❌ | ❌ | ❌ | ✅ |
| Windows 原生优先 | ✅ | ✅ | ❌ | ❌ | ✅ |
| 鼠标优先,无强制快捷键 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 开源 MIT | ✅ | ✅ | 部分 | ❌ | ✅ |

**和最像的两个对手的关键差异:**

- **vs Wave**:Wave 是"块状 AI 终端",组织维度还是 workspace + tab,内嵌 AI 是核心卖点。Marina 把**路径**抬到一等公民,本体不做 AI,只做对 AI agent 友好的容器。
- **vs tmux**:tmux 让 session 不死,但 session 还是"长期持有上下文的容器",用户给它取名、害怕丢失。**Marina 让 session 不重要** —— 真正重要的是 path,session 用完就关、要再跑就开新的。

## 5. 设计哲学:两条洞见,四条原则

> 这一节解释**为什么 Marina 的某些选择和传统终端不一样** —— 它们不是漏的,是故意的取舍。完整论证见 `docs/设计哲学-核心洞见-20260514.md`。

### 5.1 底层洞见一 · Session 不是"虚拟终端机",是"在 path 下要做的一件事"

主流终端沿用 1970 年代 VT100 物理终端的隐喻:

> Session = 一个 shell 进程 + 一块屏幕缓冲区 = 一台"虚拟终端机"

这把 session 当成**容器**:容器丢了等于丢一堆上下文,所以 tmux / screen / tmux-resurrect 一整套工具都在帮用户**护住 session**。

**Marina 重新定义 Session 是 path 下的一个任务实例:**

- 它的身份不是"第 3 个 shell",是 "my-blog 下跑着的 dev server"
- 它的归属是 path,不是窗口、不是 profile
- 它的生命周期由用户决定,跟窗口、UI 无关
- 它是**廉价的** —— 做完一件事就关掉,下次再开

这跟浏览器 tab 的心智更接近 —— 不是"我有一台机器",而是"我要做一件事,做完关掉"。

### 5.2 底层洞见二 · CLI 是 UI 协议,不是命令接口

主流认知里,CLI 是给程序员"输入命令"的接口。

但实际打开任何现代开发者的终端,真正"敲命令"的时间占比正在快速下降。更多的时间在:

- 看 `npm run dev` 的输出
- 跟 Claude Code 对话
- 看 `docker compose up` 的状态行
- 看 `pytest` / `vitest` 的进度
- 回应 `apt` / `rm -i` 的 y/n 提示

**这些不是命令输入,是使用文本 UI 应用。** CLI 工具本质上是用文本(加 ANSI、加 OSC)做渲染的 UI 应用。

**这条洞见对 Marina 是反向的** —— 它告诉我们**不该做什么**:

- ❌ 不自己实现 prompt(像 Warp 那样)
- ❌ 不内嵌 AI 补全
- ❌ 不拦截命令做 magic
- ❌ 不重做 shell

> Marina 的位置是**所有 CLI UI 应用之下的容器编排器**,不是其中任意一个。
> xterm.js 已经把"渲染 CLI 这种 UI 协议"做到工业级,Marina 没必要也没能力重做。

### 5.3 四条工程原则(从两条洞见推出)

| 原则 | 一句话 | 它推导出 |
|---|---|---|
| **1. Path 稳定 / Session 廉价 / UI 临时** | 收藏 path、别珍惜 session、别怕关窗口 | 三栏侧栏、关窗不杀任务、不维护 session 跨会话上下文 |
| **2. 不让用户输入路径,只让用户点路径** | `cd` 是 1971 年的设计,打错就报错 | 所有路径只能通过选择器/拖拽加入,UI 上路径永远可见 |
| **3. 用户决策最少化** | 工具应自动处理一切可自动的事 | 路径分类流转、cwd 跟踪、终端 resize 全自动 |
| **4. 窗口与应用解耦** | 应用 ≠ 窗口,关窗免费 | 守护进程持有数据,窗口是临时观察器 |

> **Linux 例外(ADR-013):** 第 4 条在 Linux 上不完全成立。GNOME 3.x 起官方移除 system tray,Marina 在 Linux 上不做"沉托盘"行为,关闭最后一个窗口 = 应用退出 + session 全销毁,配套阻塞 modal 在"最后窗口 + 仍有非 exited session"时弹二次确认。Windows/macOS 行为不变。

### 5.4 一句话归纳

> **Path 是项目。Session 是项目里的一件事。Window 是看这件事的临时镜头。**
> **CLI 工具自己负责呈现,Marina 负责让它们好好活着、好好被找到。**

## 6. 技术架构(TL;DR)

**Marina = Electron 31 + TypeScript + React 18 + node-pty + xterm.js**

### 6.1 进程模型

```
┌────────────────────────────────────────────────────────────────┐
│              Electron Main Process(守护进程)                  │
│                                                                │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐    │
│  │ SessionMgr   │  │ PathManager  │  │ SettingsManager  │    │
│  │ (所有 PTY)   │  │ (三栏数据)   │  │ (持久化配置)     │    │
│  └──────────────┘  └──────────────┘  └──────────────────┘    │
│                                                                │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐    │
│  │ WindowMgr    │  │ Tray         │  │ PlatformAdapter  │    │
│  │ (N 个窗口)   │  │ (常驻托盘)   │  │ (win/linux/macos)│    │
│  └──────────────┘  └──────────────┘  └──────────────────┘    │
└────────────────────────────────────────────────────────────────┘
            ↕ IPC(typed,strict)
┌────────────────────────────────────────────────────────────────┐
│  Renderer 1   │   Renderer 2   │   Renderer 3   │   ...       │
│  (React UI)   │   (React UI)   │   (React UI)   │             │
│   纯观察者     │   纯观察者      │   纯观察者     │             │
└────────────────────────────────────────────────────────────────┘
```

- **主进程 = 守护进程**:持有所有 PTY、所有数据、系统托盘
- **每个窗口 = 独立的 renderer 进程**,React UI,**纯观察者**
- **窗口关闭永远不影响 session**
- **通信走 Electron IPC**,strict typed,协议文档化在 `docs/ipc-protocol.md`(1,622 行)

### 6.2 关键技术决策

| 决策 | 理由 |
|---|---|
| Electron 而非 native | 7 天能出 alpha,18k 行 TS 跑遍三平台,性能瓶颈在 xterm 渲染而非框架 |
| node-pty + ConPTY | 唯一靠谱的 Windows PTY 方案 |
| xterm.js + WebGL renderer | 工业级文本渲染,Linux 上专门做了 perf pass |
| React 18 + Vite + electron-vite | 现代前端工具链,HMR、tsc 三段(node/preload/web)分别校验 |
| Vitest | 14 个测试文件 / 3,691 行 / **320 个测试全过** |
| 平台适配层 `src/main/platform/` | 抽象 `lifecycleModel`、`getProcessCwd`、shell 探测等,**跨平台不是 ifdef 复刻** |
| Rust + IExplorerCommand 子项目 | Win11 新右键菜单必须走 COM + Sparse Package,这是唯一能让 Marina 进入新菜单的路 |

### 6.3 已知大限制

- **KI-004**:Windows ConPTY 强约束 —— Marina 主进程崩溃则所有 PTY 必死。V1 阶段接受,长任务建议挂 tmux / screen / nohup。V2 候选方案是 daemon 架构(主进程拆 background daemon + UI viewer)。
- **代码签名未做**:Windows SmartScreen 首次启动会拦截,点"更多信息 → 仍要运行"即可。证书买完会签。
- **自动更新未做**:同样等代码签名。新版要去 GitHub Releases 翻。

完整已知问题清单:`docs/known-issues.md`

## 7. 安装与上手(5 分钟跑通)

### 7.1 下载

去 [GitHub Releases](https://github.com/suzhouwhl1025/marina-my/releases) 拿对应平台的包:

| 平台 | 包 | 适合谁 |
|---|---|---|

### 7.2 第一次跑

1. 启动,看到自绘标题栏 + 左侧三栏侧栏(全部为空) + 右侧 xterm 区
2. 点收藏分类右边的 `+`,选一个常用项目目录;或从 Explorer **拖**文件夹到侧栏
3. **双击**那个路径 → 默认起一个 shell session
4. 关掉整个窗口(× 按钮)
5. 任务管理器看 `Marina.exe` 还在
6. 单击托盘图标 → 窗口回来 → 刚才的终端还活着

**第 6 步如果终端死了 / 应用没了 → 这是 bug,告诉作者。**

## 8. 明确不做的事

为了不让用户对产品产生错误预期,以下功能**主动不做**(理由见 §5):

- ❌ **不是终端模拟器替代品** —— 用 xterm.js,跟所有人一样
- ❌ **不是 tmux 竞品** —— tmux 是 TUI,Marina 是 GUI,受众不同
- ❌ **不是项目管理工具** —— 没有 kanban、没有团队功能、没有 workspace 概念
- ❌ **不是 SSH 客户端** —— 仅本地 session
- ❌ **不是文件编辑器** —— 需要就 `code .`
- ❌ **不做 prompt 增强 / AI 补全 / 命令拦截** —— 这是 shell 和 CLI 工具的领域(违反 §5.2)
- ❌ **不做"贴砖式"功能** —— 不会因为 vim 用户喜欢就加 vim keybindings,违反"鼠标优先"的核心定位

## 11. 常见问题(FAQ)

**Q1:跟 Wave Terminal 区别?**
Wave 是"块状 AI 终端",组织维度是 workspace + tab,内嵌 AI 是核心卖点。Marina 把**路径**抬到一等公民,本体不做 AI,只做对 AI agent 友好的容器。Wave 是 Electron + AI-heavy;Marina 是 Electron + AI-agent-friendly。

**Q2:Electron?不重吗?**
接受。换 native 不会让 ConPTY 限制消失,也不会让"会话不死"自动出现。Electron 的成本买的是 7 天能出 alpha、18k 行 TS 跑遍三平台。性能瓶颈在 xterm.js 渲染,WebGL renderer 已默认开启,Linux 上还专门做了 perf pass。

**Q3:为什么不直接做 tmux GUI?**
tmux 的 session 是"长期持有上下文的容器,用户给它取名"。Marina 的 session 是**任务实例,默认不命名**。哲学不一样,做出来不会是同一个东西。

**Q4:KI-004 不可怕吗?Marina 崩了 PTY 全死。**
怕。这是 V1 的明面债。V2 候选方案是 daemon 架构 —— 把主进程拆成 background daemon + UI viewer。但在 crash/upgrade 真的成为高频痛点之前不投入工程,先把哲学层稳住。

**Q5:为什么 Windows 优先?**
因为作者自己在 Windows 上工作。也因为 Windows 上能打的对手最弱 —— WT 四年没做关窗不杀会话,这是个**洞**。macOS / Linux 上 iTerm2 / Wave 已经在了,做就是赤膊上阵,做不出差异化。

**Q6:开源协议?能商用吗?**
MIT。能。

**Q7:支持 SSH 吗?**
不专门支持。在本地 SSH client 起一个 session 是可以的,但 Marina 不会为 SSH 体验做优化。远程管理请用 Windows Terminal + SSH 或专用客户端。

**Q8:有自动更新吗?**
beta 阶段没有。等代码签名证书买完再做。新版请去 GitHub Releases 翻。

**Q9:数据存在哪?能备份吗?**
Windows 在 `%APPDATA%\Marina\`,Linux 在 `~/.config/Marina/`。设置页有"导出配置"(可选含敏感字段擦除),JSON 文件,人类可读,可手动备份。

**Q10:可以加我自己的快捷键吗?**
V1 不可配置。这是哲学约束 —— Marina 是鼠标优先,内置的快捷键最少化(Ctrl+F 搜索、Ctrl+鼠标滚轮调字号)。如果你严重依赖快捷键,Marina 不是为你设计的。

**Q11:支持 AI agent 之外的普通命令行任务吗?**
当然。AI agent 只是 Marina 设计的**触发场景**,任何"在多个路径下并发运行长任务"都受益 —— 编译、测试、dev server、docker compose、长 ssh 隧道、自动化脚本……

**Q12:在哪里反馈 bug / 提建议?**
GitHub Issues:`https://github.com/suzhouwhl1025/marina-my/issues`
带上:在做什么(路径、模板、命令)/ 期望 / 实际 / 能否复现 / 日志(设置页一键打开日志目录)。
