## I. 定位与核心不变式

### I.1 总体定位

**Marina = 终端管理器(本地 + SSH 远程终端),不做文件管理**

注意"终端管理器"的边界 — Marina 管的是**终端会话**(本地 shell / SSH shell / 远程 tmux),**不管文件传输 / 文件浏览 / 文件编辑**。SFTP / Explorer 等都属于"文件管理"范畴,**默认不做**(详见 I.2)。

### I.2 文件管理 vs 终端管理分离不变式 (新增 v2.1)

**Marina 的产品边界画在"终端"上,不延展到"文件管理"**:
- ✅ 在做:本地终端、SSH 远程终端、远程 tmux、终端搜索、终端复制粘贴
- ❌ 不做(默认):SFTP 文件浏览、远程文件编辑、文件拖拽上传、端口转发(本质是网络工具,不是终端)

**理由**:
1. 文件管理是另一个产品形态(WinSCP / FileZilla / Termius 的文件面板),做了等于让 Marina 兼任两类产品职责
2. "路径中心" 概念在 Marina 里指**会话**,不指**文件浏览**;加 SFTP 会让"路径"变成"会话 + 文件浏览"双中心
3. 用户已有成熟的文件管理工具,Marina 不做不影响 SSH 终端的核心可用性

**如果未来真要做(V2 可选)**:
- 必须走**独立窗口**,跟 RemoteTerminalView **物理隔离**,不共享 viewport
- 本地用户从不打开就完全不接触
- 在 spec §14.3 / §14.6 中明确标记"可选,非主线"

### I.3 本地用户感受不变式

**不开 SSH 的用户体验与今天 100% 一致**。

具体表现:
- sidebar 默认 segmented control 在"本地",filter 后看不到任何 SSH 节点
- 设置页 SSH 分类条件渲染(无 profile 且无 advanced 启用时不显示),本地用户进设置永远 8 个分类
- 启动时间、内存占用无变化(SSH 相关重型模块如 keytar / ssh2 lazy load,SFTP/Monaco 走 V2 可选不进 V1 bundle)
- 错误提示 / 文档 / onboarding 不含 SSH 内容

这个不变式由 **UI 层 segmented + filter + 条件渲染 + invariant 测试** 共同保障,**不依赖**后端代码分离。

## II. 设计约束

### II.1 类型系统强约束 — discriminated union

```ts
// src/shared/types.ts
export type Path = LocalPath | RemotePath;

interface LocalPath {
  kind: 'local';      // 必填,不允许 optional
  path: string;       // 本地绝对路径
  // 其他本地字段
}

interface RemotePath {
  kind: 'ssh';        // 必填
  profileId: string;
  remotePath: string;
  // 其他远程字段
}

// 类似地,Bookmark / RecentEntry / Session 都是 discriminated union
```

**强制规则**:
- `kind` 字段必填,不允许 optional(PR #2 当前是 `kind?`,要改成 required)
- 使用 Path 的函数必须 `switch on kind` 覆盖所有 case,TS exhaustiveness check 强制
- 增加新 PathKind 时(例如未来的 `'wsl'`、`'docker'`),编译器自动找出所有需要补 case 的位置

**收益**:
- 替代"物理边界"的运行时保护 → 编译期保护,更强
- 任何忘记处理 ssh kind 的代码立刻编译失败
- 添加新 kind 时无需重构所有 manager,只需在 switch 里加 case

### II.2 后端架构 — 共用 + SSH 专属

**共用 Manager(单实例,内部分支处理 kind)**:
```
src/main/
  path-manager.ts        统一管理 LocalPath + RemotePath
  session-manager.ts     统一管理 Local + Remote session,接 sshProfile? 参数
  bookmark-manager.ts    统一管理 Local + Remote bookmark
```

**SSH 专属 Manager(独立,因为没有"本地版本")**:
```
src/main/
  ssh-profile-manager.ts
  ssh-config-parser.ts
  known-hosts-manager.ts
  control-master.ts
  sftp-manager.ts
  port-forward-manager.ts
```

**设计原则**:**通用概念共用,专属概念独立**。
- Path/Session/Bookmark 是通用概念 → 共用,内部 switch on kind
- Profile/KnownHost/ControlMaster/SFTP/PortForward 是 SSH 专属概念,没有"本地版本" → 独立模块自然
- **这才是真正的 DRY** — 该共用的共用,该独立的独立

### II.3 前端 UI 层强分离

```
src/renderer/components/
  Sidebar.tsx                壳:顶部 segmented control [本地] [远程]
                             根据选择 filter 出对应 kind 的 path 数据
  SidebarLocalSection.tsx    本地节点列表(只渲染 kind === 'local')
  SidebarRemoteSection.tsx   远程节点列表(只渲染 kind === 'ssh')
  TerminalView.tsx           单一组件,根据 session.kind 渲染差异:
                              - kind === 'local': 现有 statusbar
                              - kind === 'ssh': + 重连按钮 + 连接状态指示
                              - 共用 xterm 装配 / 主题 / IME / SCROLL-1 / KBD-1
  SettingsView.tsx           SSH 分类**条件渲染**:
                              - 有 SshProfile 或 advanced.enableRemote=true 时显示
                              - 否则隐藏(本地用户视野 = 8 个分类)
  SftpPanel.tsx              独立窗口,本地用户从不打开
  PortForwardMonitor.tsx     独立窗口,同上
  HostKeyPromptModal.tsx     SSH 首次连接专用 modal(KBD-1 IME 守护延伸)
  ReconnectBanner.tsx        SSH session exit 后显示在 TerminalView 内
```

**UI 强分离三件套**:
1. Sidebar 顶部 segmented control + filter
2. 设置页 SSH 分类条件渲染
3. SFTP / 端口转发走独立窗口,不挤本地终端 viewport

**本地用户视觉感受 = 今天一样**。

### II.4 IPC 通道命名

**通用通道(含 kind 字段)**:
```
cmd:path:*              含 PathKind 分支
cmd:session:*           session 通道,接 sshProfile? 参数
cmd:bookmark:*
```

**SSH 专属通道**:
```
cmd:ssh-profile:*
cmd:ssh-config:*
cmd:known-hosts:*
cmd:sftp:*
cmd:port-forward:*
```

**不再**为本地额外加 `cmd:local-*` 前缀 — 沿用现有命名,只为 SSH 专属概念新增 `ssh-*` / `sftp-*` / `port-forward-*` 等前缀。

### II.5 持久化文件

**通用文件(含 kind 字段)**:
```
bookmarks.json     entries 含 kind: 'local' | 'ssh'
recent.json        同上
```

**SSH 专属文件**:
```
ssh-profiles.json
known-hosts.json
port-forwards.json
```

**Archive 导出**:本地用户的 archive 序列化时 filter 出 `kind === 'local'` 的项,不含 SSH 数据;SSH 用户的 archive 含完整数据。

### II.6 设置页结构

8 个分类不变。SSH 区作为**条件渲染的额外分类**(位置在"高级"之前),仅在以下任一条件下显示:
- 存在 SshProfile(已经加过 profile)
- 用户在"高级 → 启用远程功能"勾选

本地用户从不创建 profile 也不勾选 → 设置页永远是 8 个分类。

SSH 分类内部:
- SSH 服务器(profile CRUD,含 edit 入口)
- 远程文件夹(RemoteBookmark 管理)
- 已知主机(known_hosts UI)
- 端口转发规则
- SFTP 默认编辑器
- ssh_config 集成开关
- 性能 / 安全(ControlMaster / keepalive / 密钥强度检查等)

## VII. spec §14 草案

**§14.1 定位**

Marina 是本地终端管理器,可选启用 SSH 模式提供完整 SSH 客户端能力。SSH 不是默认产品形态。

**§14.2 本地用户感受不变式**

1. UI 层 segmented control 默认"本地",本地用户视野与无 SSH 时完全一致
2. Sidebar 在 segmented="本地"时,渲染数据 filter 后 `kind === 'local'`,不含任何 SSH 节点
3. 设置页"远程"分类条件渲染:无 SshProfile 且 `advanced.enableRemote=false` 时不显示,本地用户视野永远是 8 个分类
4. 启动时间 / 内存占用基线无变化:SSH 相关重型模块(Monaco / ssh2-sftp-client)必须 lazy load,验证手段在 CI Gate 中
5. CI Gate 拦截破坏上述任一不变式的 PR(invariant 测试 + lazy load 验证 + bundle size + 启动时间基线)

**§14.3 SSH 模式范围**

**V1 主线(M1 里程碑 = 0.2.0 GA)**:
- profile 管理(CRUD + ssh_config 集成)
- 连接(密钥 / agent / 密码 / MFA / passphrase / ProxyJump 跳板)
- known_hosts UX(首次提示 + 变化警告 + 指纹历史)
- 远程 tmux(per-launch + attach 检测 + session 列表)
- ControlMaster 连接复用
- 自动重连 + 网络变化检测

**V2 可选(视 V1 用户反馈决定,默认不做)**:
- SFTP 文件管理 + 远程编辑 — **属于文件管理范畴**(见 §14.6)
- 端口转发(本地 / 远程 / 动态 SOCKS5)— **属于网络工具范畴**

**永远不做**:
- X11 forwarding(罕用,跨平台 X server 噩梦)
- VPN 集成
- GUI 转发

**§14.4 类型系统约束**

1. PathKind 必须是 discriminated union,`kind` 字段必填(不允许 optional)
2. 每个使用 Path 的函数必须 exhaustive `switch on kind`,TS 编译器强制
3. ESLint 配 `@typescript-eslint/switch-exhaustiveness-check` rule 兜底
4. 任何破坏 exhaustiveness 的代码被 CI 拒绝

**§14.5 共用 vs 独立的划分原则**

- 通用概念(Path / Session / Bookmark / Recent)→ 共用 Manager,内部 switch on kind
- SSH 专属概念(Profile / KnownHost / ControlMaster / SFTP / PortForward)→ 独立 Manager,没有"本地版本"
- UI 层:本地与远程入口走 segmented + filter 分离,组件级别按需共用或拆分

**§14.6 文件管理 vs 终端管理分离原则(新增 v2.1)**

Marina 的产品边界画在"终端"上,**不延展到"文件管理"**:

1. **在做的**(终端管理):本地终端、SSH 远程终端、远程 tmux、终端搜索、终端复制粘贴
2. **不做的**(文件管理):SFTP 文件浏览 / 远程文件编辑 / 文件拖拽上传 / Explorer 集成

理由:
- 文件管理是另一个产品形态(WinSCP / FileZilla / Termius 的文件面板),做了等于让 Marina 兼任两类产品职责
- "路径中心" 概念在 Marina 里指**会话**,不指**文件浏览**;加 SFTP 会让"路径"变成双中心
- 用户已有成熟的文件管理工具,Marina 不做不影响 SSH 终端的核心可用性

**例外条款**:
- 端口转发不算"文件管理",但属于"网络工具",同样不在 V1 主线,降为 V2 可选
- 如果 V2 真要做 SFTP,**必须走独立窗口**,跟 RemoteTerminalView 物理隔离,不共享 viewport;本地用户从不打开就不接触

**§14.7 里程碑划分**

- **M1(= 0.2.0 GA)**:阶段 0-3 完成 = "Marina SSH 终端模式正式就位"。此时 SSH 用户能完成"管理远程服务器 + 进 shell 干活 + 用 tmux 保持会话"完整工作流。
- **V2(= 0.3.0,可选)**:阶段 4-5 完成 = "Marina 加 SFTP + 端口转发"。**只在 V1 上线后明确收到高频用户反馈或竞品压力时启动**。

## 附录 A:讨论过程关键认知(v1.0 → v2.0 思路转变)

本方案 v1.0 走的是"严格分离架构",经过用户挑战后大幅修订。关键认知转变:

### A.1 v1.0 的核心论据是哲学,不是技术

v1.0 反对"共用架构"的主要论据是:
- "重力方向":共用架构会让代码 1 年后自然漂向"双中心 / 通用化"
- "故障传染面":共用代码改一行可能两边都炸
- "扩张决策成本":加 SFTP 时"应该加哪个 manager"决策困难
- "产品定位锐度":共用暗示两边都一等公民,定位被稀释

**这些都是基于"代码给人读 / 给人改"的传统软件工程哲学**,延伸自 1990s OOP 文化的"边界优先 / DRY 跨域时损害清晰度"思潮。

### A.2 AI 时代这些哲学论据权重大幅下降

现实:
- 代码是 AI 写的,不是人写的
- 代码不会有人逐行读,maintenance 是 AI 重读整体 codebase 后改
- "重力方向"在 AI 节奏下不存在 — AI 一次性 reason 全局,不会渐进漂移
- "扩张决策成本"对 AI 不是成本

剩下的真技术问题只有两个:
- **故障传染面(变更安全)**:即便 AI 能 reason 全局,共用代码改坏两边可能都炸是物理事实
- **本地用户 UI 视野**:sidebar 看不看到 SSH 节点是用户感受问题

### A.3 关键洞察:剩下的两个真问题都不需要靠"架构分离"解决

- **故障传染**:用 discriminated union 编译期保护 + invariant 测试 + 共用 manager 分支独立单测 即可补救
- **UI 视野**:UI 层 segmented + filter + 设置页条件渲染即可保护

**这两个保护手段都不要求物理分离后端**。所以严格分离的代价(代码量 +30-40% / 维护成本 +30-50%)是不必要的。

### A.4 最终方案的核心洞察

**通用概念共用,专属概念独立**,这才是真正的 DRY:
- Path/Session/Bookmark 是通用概念 → 共用 Manager,discriminated union 内部分支
- SshProfile/KnownHost/ControlMaster/SFTP/PortForward 是 SSH 专属概念,没有"本地版本" → 独立 Manager 自然

**UI 层强分离**:不是因为后端要分,而是因为本地用户视野要保护。这是个 UI 决策,跟后端架构正交。

### A.5 PR #2 不需要重做

PR #2 走的就是共用方向,70% 走对了。剩下 30%(UI 分离 + 类型强化 + polish)1-2 turn 补完。

工作量从 60 天压缩到 38 天(37% 缩减),架构改造代价大幅降低,作者贡献得到尊重。

### A.6 范围分层 — V1 主线 vs V2 可选(v2.1 决策)

v2.0 把 5 个阶段全列为"主线",但这隐藏了一个产品判断问题:**SFTP / 端口转发跟 Marina 核心定位的距离比"基础 SSH 终端"远得多**。

- "基础 SSH 终端"(阶段 1-3)= 把"在远程服务器进 shell 干活"做完整 = 终端管理的自然延伸
- SFTP = 文件管理,**另一种产品形态**
- 端口转发 = 网络工具,**第三种产品形态**

把这三件事并列做意味着 Marina 隐式承诺成为"完整 SSH 客户端"。这跟 §I.1 "终端管理器" 的定位**不匹配**。

更诚实的做法:
- V1 主线只做"基础 SSH 终端"(阶段 0-3),到 M1 里程碑 = 0.2.0 GA
- SFTP / 端口转发降为 V2 可选,默认不做,视 V1 上线后用户反馈决定
- spec §14.6 新增"文件管理 vs 终端管理分离"原则,明确 Marina 的产品边界

**收益**:
- V1 主线工作量从 38 天降到 16 天(58% 缩减),AI solo 3-4 turn 完成
- 产品定位更聚焦,跟 §I.1 一致
- 避免 V1 期"什么都做但什么都不深"的陷阱
- 给 V1 上线后真实用户反馈留窗口,**用反馈决定 V2 做不做,而不是预先承诺**
