## 0. TL;DR

软件定义书已经列了四条原则。这份备忘录把其中两条往下挖一层,把"为什么是这样而不是那样"写清楚,免得过几个月自己回看时也忘了根据。

两条核心洞见:

1. **Session 不是"虚拟终端机",是"在某 path 下要做的一件事"。** Session 是任务实例,不是 shell 容器 —— 这决定了它廉价、可大量并行、可独立于 UI 存活。
2. **CLI 是 UI 协议,不是命令接口。** 现代终端里"输入命令"的比例在下降,用户更多在跟 CLI 工具(npm、docker、claude code、git log)呈现的文本 UI 打交道。这决定了 Marina 不需要重造终端 —— xterm.js 已经把"渲染 CLI 这种 UI 协议"做得足够好,Marina 只需要做好"任务的组织和持久化"。

下面分别展开。其余的(三层稳定性、窗口与应用解耦)软件定义书 §2 已经写清楚,这里不重复。

## 1. Session 的重新定义

### 1.1 旧定义

主流终端(iTerm2、Windows Terminal、Hyper、Warp)沿用的是 1970 年代 VT100 物理终端的隐喻:

> Session = 一台虚拟终端机 = 一个 shell 进程 + 一块屏幕缓冲区

这个定义把 session 当成**容器**:容器里装着 cwd、env、history、运行中的进程。容器丢了等于丢一堆上下文,所以 tmux / screen / tmux-resurrect 这一系列工具都在帮用户**护住 session**。

### 1.2 Marina 的定义

Session 是 **path 下的一个任务实例**,不是 shell 容器。

- **它的身份**不是"第 3 个 shell",而是"my-blog 下跑着的 dev server"、"my-app 下的 claude code"。
- **它的归属**是 path,不是窗口、不是 profile。
- **它的生命周期**由用户决定何时终止,跟窗口、UI、应用前台/后台无关。
- **它是廉价的**:做完一件事就关掉,下次需要再开,不需要珍惜。

这跟浏览器 tab 的心智更接近 —— 不是"我有一台机器",而是"我要做一件事,做完关掉"。

### 1.3 这个定义推导出哪些设计

| 设计选择 | 来自这个定义 |
|---------|------------|
| 三栏侧栏按 path 分类,不按 session 分类 | path 是 session 的家 |
| Session 不维护跨会话累积上下文 | session 是一次性的任务实例 |
| 关窗不杀 session,但用户主动右键"关闭"才销毁 | 任务由用户决定,不由 UI 决定 |
| 已退出 session 不重启(右键关闭即销毁) | 任务是一次性的,要重跑就开新的 |
| Path 进入"临时"的条件是"有 session 在跑" | 任务依附 path,反过来定义 path 的活跃度 |

### 1.4 跟 tmux 的区别

tmux 也实现了"session 独立于 UI"。但 tmux 的 session 还是旧定义里的"虚拟终端机"——一个长期持有上下文的容器,用户给它取名字(`work`、`server-1`),害怕丢失。

Marina 的 session 是**任务实例**,默认不命名(显示进程名或 cwd 末段),用户**不应该**对单个 session 投入心智。组织维度被推到了 path 这一层 —— path 才是用户会命名、收藏、记忆的东西。

简单说:tmux 让 session 不死,Marina 让 session 不重要。**真正重要的是 path 和它下面的任务集合**。

## 2. CLI 是 UI 协议

### 2.1 一个被忽视的事实

主流认知里,CLI 是给程序员"输入命令"的接口。

但实际打开一个现代开发者的终端,真正"输入命令"的时间占比正在快速下降。更多的时间是在:

- 看 `npm run dev` 的输出
- 跟 `claude code` 对话
- 看 `docker compose up` 的状态行
- 看 `pytest` / `vitest` 的进度
- 回应 `apt` / `rm -i` 的 y/n 提示

这些都不是"输入命令",是**使用文本 UI 应用**。

CLI 工具本质上是用文本(加 ANSI 序列、加 OSC 序列)做渲染的 UI 应用。终端窗口是渲染器,不是 REPL。

### 2.2 为什么 CLI 是 UI 协议而不只是字符流

- **标准化**:任何符合 VT100 / xterm 的终端都能渲染,跨 OS、跨远程、跨容器
- **可组合**:pipe、重定向、`tee` —— 一种 UI 能被另一个程序消费
- **可脚本化**:同一个 UI 既能给人看,也能被自动化驱动
- **开发成本极低**:不需要 designer、不需要 UI 框架、不需要每个 OS 适配
- **天然持久化**:文本流可以缓存、重放、attach/detach

这套属性的组合,正是 AI agent 时代爆发的 CLI 工具(Claude Code、Codex CLI、Aider、aichat...)选 CLI 作为载体的原因 —— 不是"复古",是 CLI 仍然是开发"任务为中心"的轻量工具的最优 UI 协议。

### 2.3 这个观察对 Marina 的直接含义

**Marina 不重造终端。**

xterm.js 已经把"渲染 CLI 这种文本 UI 协议"做到了工业级水平 —— ANSI、OSC、bracketed paste、WebGL、scrollback,这些都是别人花了十几年解决的问题。Marina 没必要也没能力重做。

Marina 要做的、也是 xterm.js 不做的,是上面一层:

- **任务的组织**:path → session 的归属关系
- **任务的持久化**:守护进程持有 session,关窗不死
- **任务的状态可见性**:active / idle / exited / (未来 waiting)
- **任务的导航**:三栏侧栏、跨窗口共享会话池、托盘

换句话说:**让用户用鼠标管理一堆运行中的 CLI UI 应用,而不是用键盘操作一台虚拟终端机。**

### 2.4 为什么这条洞见值得明文写下来

很容易在某个时刻被诱惑去"做点终端层的事":
- 自己实现 prompt(像 Warp 那样)
- 内嵌 AI 补全
- 拦截命令做 magic
- 重做 shell

这条洞见的意义是**反向**的 —— 它告诉我们**不该做这些**。这些事要么是 shell 的领域(zsh、fish、nushell 在做),要么是 CLI 工具自己的领域(Claude Code 自己呈现对话 UI)。Marina 越过这层界限,就会变成"又一个 Warp",而不是"基础设施"。

Marina 的位置是**所有 CLI UI 应用之下的容器编排器**,不是其中任意一个。

## 3. 跟现有 §2 设计哲学的关系

软件定义书 §2 的四条原则可以重新理解为本备忘录两条洞见的具体推论:

| §2 原则 | 来自本备忘录的哪条洞见 |
|---------|---------------------|
| 原则 1:Path 稳定、Session 廉价、UI 临时 | §1 Session 是任务实例(廉价、可大量) |
| 原则 2:不让用户输入路径,只让用户点击路径 | §1 Path 是 session 的家,是稳定的组织维度 |
| 原则 3:用户决策最少化 | §2 CLI UI 应用自己负责呈现,Marina 不介入 |
| 原则 4:窗口与应用解耦 | §1 Session 是任务,不是窗口 |

四条原则是工程化的设计规则,本备忘录两条是更底层的本体论判断。规则可以演化,本体论的判断如果变,产品就不是 Marina 了。

## 4. 自我边界

为了避免本文档被未来的自己当成"宣言"过度引用,明确两点:

1. **这不是普适理论,是 Marina 的设计取舍。** Vim 用户、tmux 重度用户、SSH 管理员的工作流跟这套哲学是冲突的,这是预期内的 —— 软件定义书 §3"非目标用户"已经写明。
2. **这不是"重新定义终端"。** Marina 是一个具体场景下(本地、多 path、多长任务、AI agent 友好)的更趁手的工具。终端这个生态有它自己的演化,Marina 只是在一个特定切片上下注。

写下这两条,是为了下次有人(包括自己)拿着这份文档去做过度宏大的叙事时,有个明确的刹车。

---

## 5. 一句话归纳

> Path 是项目。Session 是项目里的一件事。Window 是看这件事的临时镜头。CLI 工具自己负责呈现,Marina 负责让它们好好活着、好好被找到。
