## 1. 正确性 / 状态机 / 异步 (STM / CON)

### STM-1 [H] · handlePtyExit 不 flush pendingEmit,造成"出口"与"最后一段输出"乱序

**位置**:`src/main/session-manager.ts:992-1024`(handlePtyExit)、对照 `src/main/session-manager.ts:1026-1066`(destroySession)

**现象**:用户在 PTY 退出瞬间能看到的最末几行输出会**晚于** `evt:session:exited` 到达 renderer,极端情况下完全丢失(若 owner 窗口在此期间被关掉,sessionOutput 不再有归属)。

**根因**:
- handlePtyExit 调 `clearTimers(managed)`(L1014),但 `clearTimers`(L1178-1191)只清 idleTimer / cwdGraceTimer / cwdPollTimer,**不**清 pendingEmitTimer。
- PER-2 / F1 引入的 8ms 聚合窗口可能正含尚未 emit 的 bytes(`managed.pendingEmit` 非 null)。
- handlePtyExit 已先 emit `sessionExited` + `sessionStateChanged({state:'exited'})`,**之后** pendingEmitTimer 才 fire → flushPendingEmit emit `sessionOutput`。renderer 看到"先 exited,后又来一段 output"。
- destroySession(L1035-1041)里有 flush pendingEmit 的代码,但 destroySession 只在用户右键关闭 / shutdown 时触发,handlePtyExit → destroySession 之间可能间隔无限长(ADR-008 砍墓地后 exited 不再自动销毁)。

**影响**:用户感知"刚退出的 session 又冒出一段乱码"或"最后一条 error message 看不到";renderer dataHandler 收到 exited session 的 output,根据现有逻辑可能被丢弃。

**修复方向**:
```ts
private handlePtyExit(managed, exitCode, signal): void {
  if (!this.sessions.has(managed.info.id)) return;
  if (managed.info.state === 'exited') return;
  // 新增:flush 最后一段输出,保证 exited 之前 owner 已看到全部字节
  if (managed.pendingEmitTimer) {
    clearTimeout(managed.pendingEmitTimer);
    managed.pendingEmitTimer = null;
  }
  if (managed.pendingEmit) this.flushPendingEmit(managed);
  // ... 原状态转移 + emit sessionExited
}
```

### STM-2 [M] · clearTimers 漏清 pendingEmitTimer,exited session 仍可能 emit output

**位置**:`src/main/session-manager.ts:1178-1191`

**现象**:即使 STM-1 不被触发,destroySession 之外的路径(handlePtyExit 单独走)留下了一个会自激发的 setTimeout 引用。session 已进入 exited 状态,但 timer 还在 8ms 后 fire,emit `sessionOutput`。

**根因**:`pendingEmitTimer` 与 `pendingEmit` 这对状态的清理在 destroySession(L1035-1041)与 handlePtyExit 之间不一致。

**修复方向**:把 pendingEmit 系列纳入 clearTimers,或在 clearTimers 之外加一个 `flushAndStopEmit(managed)` 统一调用点:
```ts
private clearTimers(managed): void {
  if (managed.idleTimer) ...;
  if (managed.cwdGraceTimer) ...;
  if (managed.cwdPollTimer) ...;
  // 新增
  if (managed.pendingEmitTimer) {
    clearTimeout(managed.pendingEmitTimer);
    managed.pendingEmitTimer = null;
  }
}
```
(STM-1 的修复仍要在 clearTimers **之前** flush pendingEmit;STM-1 / STM-2 是同源问题,但 STM-2 是更稳的兜底。)

### STM-3 [M] · attachSession 内 removeRecent + touchRecentTimestamp 重置 useCount

**位置**:`src/main/path-manager.ts:328-331`(attachSession)、`src/main/path-manager.ts:454-466`(touchRecentTimestamp)

**现象**:用户在某 recent 路径上反复"开终端 → 关终端",useCount 在 1 ↔ 2 之间循环,不真实增长。该字段虽未在 V1 UI 显示,但已持久化到 `recent.json`,V1.1 排序或筛选若启用会失真。

**根因**:
- attachSession L329-330 顺序:`removeRecentInternal(path)` → `touchRecentTimestamp(path)`。
- removeRecentInternal 已把 recent 中该路径的 entry 删除,然后 touchRecentTimestamp.find 返回 undefined → 走 unshift 分支 → 新 entry 的 useCount=1。
- 之前 entry 的 useCount(任意值)被无声丢弃。
- detachSession 路径的 touchRecent 才会增 useCount,但每轮"attach 重置 1 + detach +1"刚好压成 2。

**修复方向**:
1. 简洁版:attach 时不 removeRecent + 重新 unshift。只要 sortRecent 重排即可;getTree 已经过滤掉 sessionPaths,不需要也不应该让该 path 在 recent 数组中消失/重生。
2. 等价版:把 touchRecentTimestamp 改为"存在则更新,不存在则创建",并去掉 attach 中的 removeRecentInternal 调用(留 touchRecentTimestamp + persistRecent + sortRecent)。

### CON-1 [M] · JsonStore.flushInternal 尾调递归,极端高频 set 下栈溢出

**位置**:`src/main/persistence.ts:136-157`

**现象**:flush 期间又 set 进新值,代码路径:flushInternal → atomicWrite → finally → `if (this.pendingValue !== null) await this.flushInternal()`。理论上 set() 节奏快于磁盘写,递归深度无界。

**根因**:`await` 表达式不在 JS 中复用栈帧(尾调用未优化),N 层递归 = N 层调用栈。daily-driver 场景下 settings 高频改(Ctrl+滚轮调字号、主题变化)有概率触发。

**修复方向**:把尾递归改 while 循环:
```ts
private async flushInternal(): Promise<void> {
  if (this.writeInFlight) await this.writeInFlight.catch(() => {});
  while (this.pendingValue !== null) {
    const value = this.pendingValue;
    this.pendingValue = null;
    this.writeInFlight = this.atomicWrite(value);
    try { await this.writeInFlight; }
    finally { this.writeInFlight = null; }
  }
}
```

### CON-2 [L] · sessionShellsCache 返回引用,caller 可意外污染

**位置**:`src/main/session-manager.ts:1209-1228`(getShells / listAvailableShells)

**现象**:`listAvailableShells()` 返回 `this.cachedShells` 引用(同一数组对象)。IPC handler `SETTINGS_LIST_SHELLS`(`src/main/ipc.ts:513-521`)再 `.map(s => ({...}))` 得到副本送往 renderer,目前没人 mutate。但 API 表面不安全。

**修复方向**:listAvailableShells 返回 `this.cachedShells.slice()`。

### CON-3 [L] · main/index.ts:window-all-closed 处理器主体为 no-op,意图与实现不一致

**位置**:`src/main/index.ts:183-187`

**现象**:
```ts
app.on('window-all-closed', () => {
  if (isQuitting) return;
});
```
注释说"isQuitting=true 时让默认行为执行"。实际上注册了 `window-all-closed` 监听器即抑制 Electron 默认 quit;handler 内若不显式调 `app.quit()`,默认行为永不执行,无论 isQuitting 是 true 还是 false。`if (isQuitting) return;` 是 dead code。

**影响**:无功能 bug — quit 由 cmd:app:quit / tray.quitApp 主动 `app.quit()` 推动,此 handler 仅作为"窗口都关了但应用要继续(进入纯托盘)"的标志。但注释误导读者。

**修复方向**:删除 `if (isQuitting) return` 或换成空体 + 注释 `// no-op: 注册即抑制默认 quit;真退出走 app.quit() 主动调用`。

## 2. IPC 协议与三层一致性 (IPC / TYP)

### IPC-1 [H] · SESSION_CREATE 在 `takeOwnership=false` 时抛 NotOwner,该路径根本走不通

**位置**:`src/main/ipc.ts:260-275`、`src/main/session-manager.ts:489`(createSession 创建 info)、`src/main/session-manager.ts:671-689`(releaseOwner)

**现象**:
```ts
// ipc.ts
const session = await sessionManager.createSession({
  ...
  ownerWindowId: takeOwnership ? envelope.windowId : '',  // false 时传空串
});
if (!takeOwnership) {
  sessionManager.releaseOwner(session.id, envelope.windowId);  // 抛 NotOwner
}
```
createSession 把空串通过 `input.ownerWindowId || null` 落到 `info.ownerWindowId = null`。然后 releaseOwner 校验 `managed.info.ownerWindowId !== windowId` → `null !== uuid` → 抛 NotOwner。

**影响**:`CreateSessionPayload.takeOwnership=false` 路径在 IPC 层永远抛错。当前 renderer 是否走这条路径未在范围内审,但协议契约写了就该兑现。

**修复方向**:
- 简洁版:`if (!takeOwnership)` 分支整段删除,createSession 已经接受空 owner;不需要再做 release。
- 防御版:releaseOwner 接受 `windowId === null` 视为"释放无主"(语义上 no-op),返回 true 而非 throw。

### IPC-2 [M] · broadcastEvent 用 BrowserWindow.getAllWindows() 而非 windowManager.list()

**位置**:`src/main/ipc.ts:141-144`

**现象**:广播事件遍历 `BrowserWindow.getAllWindows()`,任何 Electron 创建的 BrowserWindow(理论上的 dialog window、第三方插件创建的窗口、未来某个独立 UI)都会收到 marina 内部事件。

**根因**:写法早于 WindowManager.list() 完整化。

**修复方向**:改用 `windowManager.list()` + 通过 windowId 取 webContents。或保持 getAllWindows 但加 `if (managed.has(webContents.id))` 过滤。

### IPC-3 [M] · preload windowId 缺省回退 'bootstrap',误中无关窗口的隐患

**位置**:`src/preload/index.ts:33-44`

**现象**:URL 缺 windowId 参数时(理论上不该发生,但保护性兜底)windowId='bootstrap'。renderer 用此 ID 发 `WINDOW_CLOSE_SELF` / `WINDOW_FOCUS` / `SESSION_CLAIM`,main 端找不到该 windowId,效果是静默失败或行为未定义。

**根因**:'bootstrap' 是 magic string,WindowManager 也用 UUID,理论上不冲突 — 但 IPC 层用 envelope.windowId 直接查 WindowManager,得 `null`,handler 视情况静默 / 抛 WindowNotFound。

**修复方向**:
- 严格版:缺 windowId 时 `throw new Error('preload: window query missing windowId, refusing to start')`。
- 妥协版:把 'bootstrap' 改成更醒目的 'UNINITIALIZED-windowId-MISSING',让 IPC 层日志能立刻定位。

### IPC-5 [L] · `explorer-integration` 不在 domain 白名单,protocol.test.ts evt 正则也不允许

### IPC-6 [L] · IPC envelope.windowId 由 renderer 自报,无服务端反向校验

**位置**:`src/main/ipc.ts:201-251`(WINDOW_CLOSE_SELF / WINDOW_MINIMIZE / WINDOW_TOGGLE_MAXIMIZE)、`src/main/ipc.ts:362-391`(SESSION_SEND_INPUT)

**现象**:所有 handler 直接信任 `envelope.windowId`。如果一个被攻陷的 renderer 伪造 envelope,可关闭/聚焦/最小化任意窗口、声称是其他窗口在写 PTY(走 SEND_INPUT 的 owner 校验,见 `ipc.ts:382`)。

**根因**:Electron 的 IPC 信任模型默认"renderer 是可信代码"。但 marina 接受用户从 Explorer 拖入路径、读 OSC 数据等外部输入,renderer 一旦被攻陷,IPC 是次级防线。

**修复方向**:用 `BrowserWindow.fromWebContents(_e.sender)` + `windowManager.list()` 反查真实 windowId,与 envelope.windowId 对比不一致时拒绝(或仅警告日志)。Electron 推荐做法。Daily-driver 单用户场景下优先级低,但开源前应当补。

### TYP-1 [M] · SessionRuntimeShape 放在 shared/types.ts 但内含 `Buffer`,renderer 不可用

**位置**:`src/shared/types.ts:340-346`

**现象**:`interface SessionRuntimeShape { info: SessionInfo; scrollback: Buffer | null; }` 放在 shared 命名空间。`Buffer` 是 Node 全局类型,renderer 上下文(电子化的 web)不可用。

**根因**:接口注释承认"Main 内部使用,这里只为文档化暴露 schema"。但 shared/* 的语义就是两端共用,放此处违背单一职责。

**影响**:若 renderer 不慎 import 此 interface(类型层面无错),编译期不报错但运行时 web `Buffer` 名字未定义。当前 renderer 没 import 它,所以无故障。

**修复方向**:删除该 interface(它本就是给开发者看的文档),或把它移到 `src/main/session-manager.ts` 与 ManagedSession 旁。

### TYP-2 [M] · applyArchiveInMemory 不真事务化,replaceAll 不校验 bookmarks/recent shape

**位置**:`src/main/ipc.ts:901-951`(applyArchiveInMemory)、`src/main/ipc.ts:862-863`(buildArchive 强转)、`src/main/path-manager.ts:506-513`(replaceAll)

**现象**:
1. `buildArchive` 把磁盘读出的 `bookmarks.json` / `recent.json` 强转 `as SettingsArchiveV1['bookmarks']`(注释 L862:"持久化文件不通过 IPC 不需要严格 schema 校验") — 但导入时 archive 是用户从外部带回的,所以下游 applyArchiveInMemory 接收时必须校验。
2. `validateArchive`(L868-886)只校验顶层四个字段存在,**不**校验 bookmarks[i].path / addedAt / id 等字段格式。
3. `PathManager.replaceAll(input)` 完全信任 input,bookmarks/recent 数组元素只做 slice 不做形状校验。
4. applyArchiveInMemory L905 注释"无法在不 commit 的前提下单独 validate ... 这一层事务化主要是日志清楚而不是真原子"。承认是半事务。

**影响**:用户导入恶意 JSON(或老格式 / 半坏 JSON)后,bookmarks 数组里可能存在 `{ id: null, path: undefined, ... }` 类型的伪 bookmark;后续 `findBookmarkByPath(...).path === normalized` 比较恒 false / 抛 undefined.endsWith。

**修复方向**:
- PathManager.replaceAll 顶部加 type guard:每条 bookmark / recent 必须 `typeof b.id === 'string' && typeof b.path === 'string' && typeof b.addedAt === 'number'`,有任一不符的整段拒绝并 throw。
- buildArchive 改为不依赖磁盘文件,直接调 `pathManager.listBookmarks() / listRecent()`,保证导出形状受类型约束。

### TYP-3 [L] · renameSession 空名复用 SessionNotFound 错误码

**位置**:`src/main/session-manager.ts:560-566`

**现象**:`if (!trimmed) throw new SessionManagerError('SessionNotFound', '新名不能为空');`。注释承认"复用错误码:不至于因为校验另开一个枚举"。

## 3. 资源管理 / 错误处理 (RES / ERR)

### RES-1 [M] · logger.flush 不真等待 fs flush,quit 前最后几行日志可能丢失

**位置**:`src/main/logger.ts:176-185`

**现象**:
```ts
async flush(): Promise<void> {
  await new Promise<void>((resolve) => {
    if (!stream) { resolve(); return; }
    setImmediate(() => resolve());
  });
}
```
注释承认"WriteStream.write 是异步的,end 之后才能保证完全 flush,但我们不能 end(下次写需要重建),改用 setImmediate"。**这只是让出一个事件循环 tick,与 fs flush 无关**。

**影响**:`main/index.ts:386-391` 在 will-quit 里 `Promise.race([flushAll, 1s])`,logger.flush() 在 setImmediate 后立即 resolve,Promise.race 不会等真正落盘 → quit 前最后一波日志(尤其 shutdown 关键诊断)可能在 stream internal buffer 中随 process.exit 一起消失。

**修复方向**:
- 简单版:`flush()` 内 `await new Promise(r => stream.write('', r))`,write 的 callback 在 buffer flush 后触发。
- 精确版:维护 `pendingWrites: Promise<void>[]`,每次 `stream.write(line)` 推一个 Promise(用 cb 包装),flush 等待全部 resolve。
- 妥协版:flush 时 `stream.end()` 强行落盘,下次 ensureStream 重建。end 后 stream null,下次 write 走重建路径。

### RES-2 [M] · logger.write 内部 await 错误被 `void` 吞

**位置**:`src/main/logger.ts:142-143`

**现象**:
```ts
function log(level, module, message, ...extra): void {
  void write({ level, module, message, ts: Date.now(), extra });
}
```
`write` 是 async,内部 `await ensureStream()` 可能因 `fs.mkdir` 失败抛错(权限、磁盘满、路径长度等)。`void` 把 Promise 抛丢,失败不可见。

**影响**:磁盘异常时日志静默丢,但 main 进程感知不到、用户看不到。

**修复方向**:
```ts
function log(level, module, message, ...extra): void {
  void write({...}).catch((err) => {
    try { console.error('[logger] write failed:', err); } catch {}
  });
}
```

### RES-3 [M] · stream.write 不检 backpressure,高速 DEBUG 日志内存膨胀

**位置**:`src/main/logger.ts:138`

**现象**:`stream.write(line)` 返回 false 时 Node 已把数据存在内部 buffer,等 drain 才落盘。当前代码不检返回值、不监听 drain。

**影响**:`settings.advanced.logLevel='DEBUG'` 下,session-manager 每次 PTY chunk + IPC 通讯都落盘 → 高速 PTY 输出场景日志写入速度 < fs throughput,内存使用涨。Daily driver 极少开 DEBUG,但 troubleshooting 期会触发。

**修复方向**:
- 简单版:write 返回 false 时,drop 后续 N 条 + 写一条 "[logger] backpressure drop XXX entries"。
- 完整版:维护 pending queue,drain 后 flush queue;同时把 pending 上限 PENDING_LIMIT 也用于已绑定 logDir 后的 backpressure 期。

### RES-4 [L] · main/index.ts Promise.race 的 setTimeout 不 clearTimeout

**位置**:`src/main/index.ts:386-391`

**现象**:
```ts
await Promise.race([
  flushAll,
  new Promise<void>((resolve) => setTimeout(resolve, 1000)),
]);
```
flushAll 1s 内完成时,setTimeout 仍在跑,留 dangling handle。Quit 阶段无影响,但 idiom 不严谨。

**修复方向**:
```ts
const timeoutId = setTimeout(() => {}, 1000);
try {
  await Promise.race([flushAll, new Promise(r => setTimeout(r, 1000))]);
} finally {
  clearTimeout(timeoutId);
}
```
或用 AbortController。

### RES-5 [L] · createSession 错误后 PTY 已 spawn 但未释放(部分路径)

## 4. 安全 / 边界 / 用户数据 (SEC)

### SEC-1 [L] · IPC 命令信任 envelope.windowId,与"renderer 防御"愿景不符

详见 IPC-6。同一问题从安全视角看一次。在 Marina 当前"单一受信 renderer"模型下风险低,但纳入"daily driver + 开源"语境后,Electron 推荐做法是用 `_e.sender` 服务端校验,而非 renderer 自报。

### SEC-2 [L] · cmd quoting 缺 `%` / `!` / `\n` 等元字符

**位置**:`src/main/platform/windows.ts:393-396`(quoteCmd)

**现象**:`if (!/[\s"&|<>^]/.test(s)) return s;` 漏了:
- `%` — cmd.exe 环境变量展开(`%PATH%` 在 PROMPT / argv 内会被解释)
- `!` — delayed expansion(`setlocal enabledelayedexpansion` 下 `!var!` 展开)

- `\n` `\r` — 命令换行
- `\t` — 与 space 等效但未列

**影响**:用户自定义 template.command/args 含上述字符 → cmd.exe 解释错;若用户在 args 里有 `%USERNAME%` 期望 literal,会被展开。低概率攻击面 — 是用户自己写的模板,危害对自身。

**修复方向**:扩展正则到 `[\s"&|<>^%!()\r\n\t]`。或更稳:用 PowerShell `-Command` 路径作主路径(已经如此),cmd.exe 模板由用户自担风险并加注释 + UI 警告。

### SEC-3 [L] · SPAWN_ENV_SKIP 名单不全,Electron 私有 env 泄露到子 shell

**位置**:`src/main/session-manager.ts:59`

**现象**:仅 `['ELECTRON_RUN_AS_NODE', 'ELECTRON_RENDERER_URL']`。Electron 还有:
- `ELECTRON_NO_ASAR` / `ELECTRON_DEFAULT_ERROR_MODE` / `ELECTRON_DISABLE_SECURITY_WARNINGS` / `ELECTRON_ENABLE_LOGGING` 等
- `NODE_OPTIONS` / `NODE_DEBUG` — 子 shell 用 node 起子进程会感知,可能跑出意料路径
- `PORTABLE_EXECUTABLE_DIR`(build-type.ts 用它,但子 shell 继承也会感知 Marina 是 portable)

**影响**:信息泄露层面 — shell 进程能从 env 嗅探 marina 上下文。低危,但对"零信任"是负面项。

**修复方向**:扩展 SPAWN_ENV_SKIP,或反向思路 — 列白名单 PATH / TEMP / SystemRoot / USERPROFILE 等常用 env,其余全 drop。

### SEC-4 [L] · settings 导入 / PathManager.replaceAll 不做 path normalization 校验

**位置**:`src/main/path-manager.ts:506-513`(replaceAll)、`src/main/ipc.ts:937-940`(applyArchiveInMemory)

**现象**:用户导入归档,bookmarks 中可能有 `path: "../../../etc/passwd"` 或 `path: "\\\\?\\GLOBALROOT\\..."` 等 path traversal / 非常规路径。replaceAll 不 normalize、不校验存在性。后续 attachSession / addBookmark 会 normalize,但 replaceAll 路径直接绕过。

### SEC-5 [L] · explorer-integration 的 PowerShell 调用 hardcode 安全,但模式可推广

**位置**:`src/main/explorer-integration.ts:91, 114-115, 312`(runPowerShell / execFile)

**现象**:当前所有 PowerShell 调用的参数都是硬编码常量(MSIX_PACKAGE_NAME / CERT_SUBJECT / 资源文件路径),无注入风险。但 `runPowerShell` 函数本身接受任意 script string,以后若有人传 user input 进来易踩坑。

**修复方向**:

- 或拆成 `runPowerShellScriptByName(scriptName, args[])` + `runPowerShellInlineHardcoded(staticScript)`,后者类型上排斥变量插值。

### SEC-6 [M] · 导出归档"含敏感凭据"选项与文件名后缀,但归档内容本身不加密

**位置**:`src/main/ipc.ts:707-758`(SETTINGS_EXPORT)

## 5. 性能 / 内存 (PER)

### PER-1 [M] · scrollback 仍用 Buffer.concat,O(N) 拷贝未根除

**位置**:`src/main/session-manager.ts:965-989`(appendScrollback)

**现象**:每次 flushPendingEmit 调用 appendScrollback → `Buffer.concat([scrollback, bytes])`。即便 PER-2 / F1 8ms 聚合把 IPC 调用频率压到 ~125 次/秒,每次仍要 O(scrollback.length) 拷贝。2MB scrollback × 125 = 250MB/s memcpy。

**与终端渲染审计的关系**:OSC-8 在终端渲染审计标记为已知问题,本审计再次确认未修。

**修复方向**:
- ring buffer:预分配固定大小 Buffer(例如 4MB),用 read/write 双指针,环形写入。Read 时取连续段(可能需要一次拼接,但读频率远低于写频率)。
- 妥协:scrollback 上限调小到 512KB,O(N) 拷贝量减少 4 倍,daily-driver 影响可接受。

### PER-2 [M] · SESSION_CREATE 用 JSON.stringify(getTree()) 两次判 pathTreeChanged

**位置**:`src/main/ipc.ts:261, 277`

**现象**:
```ts
const oldTreeJson = JSON.stringify(pathManager.getTree());
... await createSession(...);
const pathTreeChanged = JSON.stringify(pathManager.getTree()) !== oldTreeJson;
```
getTree 是 O(N+M+K)(三栏遍历 + sessionsForPath O(N²) 内层);JSON.stringify 又是 O(树大小)。每次 createSession 走两遍。Daily-driver 100+ session 的极端场景下浪费明显。

**修复方向**:
- 加 PathManager 内部"tree 变更计数器" / 事件标记,createSession 前后对比 counter。
- 或:createSession 已经知道是否触发 attachSession 让 path 进入临时分类,直接传 boolean 不算计。
- 或:接受 false positive,renderer 总用 `evt:path:tree-updated` 增量更新,createSession 的 pathTreeChanged 字段废弃。

### PER-3 [L] · path-manager.getTree 对 recent 重新 normalizePath 

**位置**:`src/main/path-manager.ts:385-395`

**现象**:`this.recent.filter(r => { const n = normalizePath(r.path); ... }).map(r => ({ id: normalizePath(r.path), path: normalizePath(r.path), ... }))`。recent 数组里的 path 应当已是 normalized(addBookmark / touchRecent 入口已 normalize),这里 3 次重复 normalize 是无意义的 CPU 浪费。

**修复方向**:`const n = r.path; const id = r.path; ...` 直接用。

### PER-4 [L] · PathManager 多处 find / findIndex O(N) 查询

**位置**:`src/main/path-manager.ts`(findBookmarkByPath / sessionsForPath / 等)

**现象**:bookmarks 数组上的 findIndex 等查询,在 30+ bookmark 时无碍;若用户导入数千条 recent,getTree 内 `sessionsForPath` 在每个 bookmark 节点都走 O(sessionToPath.size) 遍历,总 O(B×S)。

**修复方向**:维护 `pathToBookmarkMap: Map<string, Bookmark>` 与 `pathToSessionIds: Map<string, Set<string>>` 反向索引。优先级低 — V1 不会有大数据量。

**修复方向**:移到 `src/main/platform/noop.ts`,session-manager.ts 不再含此函数;import 关系更清。或者放在 test helper 文件,SessionManager 接受可选 platformAdapter 注入(目前已是)。

## 7. 测试质量 (TST)

### TST-1 [M] · 大量 `as unknown as XxxManager` 强转 stub,运行时失败而非编译期发现

**位置**:`src/main/session-manager.test.ts:109-115, 122-136, 138-161`、`src/main/path-manager.test.ts:67-71`、`src/main/templates-manager.test.ts:17-50`

**现象**:stub 只实现 SUT 实际调用的几个方法,通过 `as unknown as RealManager` 满足类型。一旦 SUT 调用 stub 上未实现的方法,运行时 `undefined is not a function`。

**影响**:重构 SUT(SessionManager 等)新增依赖方法时,stub 不会被 TS 提示需要更新,只在测试运行时炸。

**修复方向**:
- 用 `vi.mocked()` 或 `vitest-mock-extended` 自动 mock 全部方法,默认 throw "not mocked"。
- 或显式声明 `interface IPathManager` 在 src/main/ 内,PathManager 实现该接口,stubs 也实现该接口 — TS 期能检测漏方法。

### TST-2 [M] · ipc.ts 1120 行代码无独立测试,IPC-1 这类 bug 仅靠运行时复现

**位置**:`src/main/ipc.ts` 全文件

**现象**:registerCommandHandlers / wireEventBroadcasts / applyArchiveInMemory / buildArchive / validateArchive 没有 .test.ts。整个 IPC 调度逻辑只靠 SessionManager / SettingsManager 等的单测**间接**覆盖,IPC-1(takeOwnership=false 抛错)就是这个间接覆盖的盲区。

**影响**:任何 envelope 解包错误、handler 调用错 manager 方法、handler 顺序依赖等,都得用 smoke / 手测发现。

**修复方向**:
- 给 ipc.ts 加 `ipc.test.ts`,用 ipcMain.emit 模拟 invoke 调用,验证关键 handler 的:
  1. envelope.windowId 透传到 manager
  2. payload 解析正确
  3. 错误从 manager 透出
  4. takeOwnership=true / false 两条 SESSION_CREATE 分支
- 至少覆盖 SESSION_CREATE / SESSION_CLAIM / SESSION_SEND_INPUT / WINDOW_CREATE / SETTINGS_UPDATE 几个高频路径。

### TST-3 [M] · settings export/import 工具函数(buildArchive / validateArchive / applyArchiveInMemory)无测试

**位置**:`src/main/ipc.ts:831-951`

**现象**:这三个函数处理用户外部输入(JSON 归档),是 corruption / 兼容性 / 半事务化风险的集中点。代码内注释承认事务化不完整(L905-916)、format 字段接受新旧两值(L869-882)、env 字段清空(L750-754)等业务规则,但都没有测试。

**修复方向**:加 `archive.test.ts`,覆盖:
- 合法归档 round-trip(export → import → 状态等同)
- 缺字段 / 多字段 / format 不对 / version 不对的拒绝
- 部分失败时的状态(settings 替换成功但 templates 失败,是否能从快照恢复)
- env "仅公开" 模式

### TST-4 [M] · logger.ts 无测试

**位置**:`src/main/logger.ts`

**现象**:有 `_resetForTest()` 接口表明本应可测试,但目录里没有 `logger.test.ts`。pending 缓存上限(PENDING_LIMIT=500)、日切换文件(currentDate 变化)、size 切换(LOG_FILE_SIZE_LIMIT=5MB)、purgeOld 7 天清理逻辑都无覆盖。

**修复方向**:加 `logger.test.ts`,覆盖:
- setLogDir 前的 pending 缓存与 flush
- 日切(mock Date.now / currentDate)
- size 切(连写 > 5MB,断言文件序号 +1)
- purgeOld(mock fs.readdir / stat)
- minLevel 过滤(DEBUG 关闭时 debug 不进文件 + 不进 console)

### TST-5 [M] · explorer-integration.ts 无测试

**位置**:`src/main/explorer-integration.ts`

**现象**:340+ 行无测试。`getWindowsBuildNumber` 是纯函数可直接测;runPowerShell / queryMsixPackage / setClassicIntegration 需要 mock execFile / runReg。

**修复方向**:至少给 `getWindowsBuildNumber` / `checkModernSupport` / `checkClassicSupport` 三个纯函数加测试,这些是策略层的关键判定点。

### TST-6 [L] · ipc-protocol.md 与代码 channel 列表无对账测试

**位置**:`src/shared/protocol.test.ts`

**现象**:protocol.test.ts 只验证命名格式与去重。但 ipc-protocol.md 与 shared/protocol.ts 是否同步,无自动化检查。

**修复方向**:写个测试 case 读 ipc-protocol.md 的命令表,与 COMMAND_CHANNELS 对账。或在 CI 加 markdownlint 规则。短期可接受 IPC-4 的人工补齐。

### TST-7 [L] · session-manager.test.ts 用 console.warn spy 抑制,但 session-manager 还在用 console

**位置**:`src/main/session-manager.test.ts` 各处 / `src/main/session-manager.ts:743, 787, 1055`

**现象**:SessionManager 内 `console.warn` 在测试运行时会打到 vitest stderr。测试目前没有 spy 抑制,测试输出含未抑制的 warn 噪音。

**修复方向**:与 ERR-1 / MNT-2 联动:全部 console.* 切到 logger 后,测试只需要 mock logger 静默即可。或在 vitest setup 里 spy console.warn 全局抑制(但会掩盖真正的 warn)。
