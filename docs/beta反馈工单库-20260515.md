## BETA-001 · 新装软件后无法被新终端识别

**状态**:待办 | **分类**:Bug-已知根因 | **优先级**:P0 | **实施批次**:1

### 用户反馈

> 安装新软件后,新建的终端检测不到该软件,必须重启整个 Marina 才能生效。

### 代码调研

**位置**:`src/main/session-manager.ts:450`

```ts
const env = buildSpawnEnv(process.env, SPAWN_ENV_SKIP);
```

**根因**:Marina 主进程启动时,Node 把当时的 `process.env`(含 PATH)快照固定。之后 Windows 安装新软件向 HKLM/HKCU 注册表写入新 PATH,但**已运行进程不会收到广播**。每个新 PTY 都从这个陈旧 PATH spawn,新装的 `python.exe` / `node.exe` 找不到。同样影响通过 `setx` 设的任何环境变量。

### 决策(2026-05-15)

每次 spawn 前从注册表重读 PATH。

### 实施要点

1. 在 `session-manager.ts:450` 的 `buildSpawnEnv` 前增加一层 Windows 平台 PATH 刷新:
   - `reg query 'HKCU\Environment' /v Path` + `reg query 'HKLM\System\CurrentControlSet\Control\Session Manager\Environment' /v Path`
   - 合并:HKLM 在前,HKCU 在后(与 Windows 系统 PATH 语义一致)
   - 覆盖 `env.PATH`
2. 失败时回退到 `process.env.PATH`,记录 `log.warn`
3. Linux/macOS 平台不需要此修(走标准 `process.env`)
4. 建议在 `PlatformAdapter` 加 `getRefreshedPath(): string` 方法,Windows 实现 reg query,其他平台返回 `process.env.PATH`
5. 测试:`src/main/session-manager.test.ts` 加用例,验证 Windows mock 下 reg query 失败时仍能 spawn

### 关联代价

- 每次 spawn 多几十毫秒(reg query 启动 + 解析),acceptable
- 不影响 Linux/macOS

## BETA-002 · 后端崩溃会拖垮所有终端

**状态**:待办 | **分类**:大规模演进 | **优先级**:P1 | **实施批次**:3

### 用户反馈

> 一旦后端进程异常退出,所有终端会话全部丢失。需要调研后端的健壮性与崩溃恢复机制,确保后端不会轻易崩溃。

### 代码调研

**现状**:
- `src/main/index.ts` 顶部已有 `process.on('uncaughtException', …)`,只 log 不 restart
- `src/main/session-manager.ts` 已为 `pty.write` race condition 加 try/catch(K3/PER-2)
- 持久化层(`persistence.ts`)有 `.bak` 备份、原子写
- Renderer 崩溃已自动 reload(CRA-1),但 **main 崩溃没有 supervisor**

**架构约束**:Windows ConPTY 的 PTY 是 main 进程直接子进程,父死则子必死(Windows job object 默认)。**"main 死但 session 续命"在 Windows 上做不到**,除非把 PTY 改成独立 sidecar(重写主进程)。

### 决策(2026-05-15)

强化兜底(方案 A),接受 PTY 同死作为已知限制写进文档。**不做 watchdog、不做 PTY sidecar**。

### 实施要点

1. 全面 audit IPC handler 的 try-catch 缺口:
   - `src/main/ipc.ts` 各 handler 顶层 try-catch,失败时返回错误 + log,不让异常冒泡到 ipcMain
   - 重点查 cmd:session:* / cmd:path:* / cmd:bookmark:* 三组
2. 持久化层 / logger / pty 缺口查漏补缺
3. `docs/known-issues.md` 加一节 **KI-NNN · Windows ConPTY 强约束 → main 死则 PTY 必死**(格式参考现有 KI 条目)
4. 工作量估 1-2 人天

### 关联约束

- 工单 BETA-002 完成后,可在 README / 试用说明中明确告知用户"Marina 主进程崩溃会丢失所有 session,定期 export 备份是好习惯"

## BETA-003 · Linux 支持(方案 A:无托盘普通桌面 app)

**状态**:待办(已重写) | **分类**:大规模演进 | **优先级**:P1 | **实施批次**:3(拆 4 子工单)

### 用户反馈

> 后续有一批"一生一芯"的小可爱们要评估本软件,Ubuntu 支持必须自己补上。动工前需先审计当前托盘模型如何迁移到 Ubuntu。环境预期是 Ubuntu 22.04。

### 决策演化(2026-05-15 → 2026-05-16)

**初版(2026-05-15)** 计划"完整复刻 Windows 托盘心智":AppIndicator 扩展引导 + nautilus-python 写 Nautilus 扩展 + Linux 端伪装 tray 行为。

**修订(2026-05-16)** 经哲学讨论后推翻初版,采用**方案 A**:Linux 上 Marina 作为普通桌面 app 实现,**不做托盘 / 不做后台常驻**。关闭最后一个窗口 = 应用退出,session 全销毁。配套阻塞 modal 在"最后窗口 + 仍有非 exited session"时弹二次确认。文件管理器集成走 freedesktop 标准(`.desktop` + `Categories=TerminalEmulator` + gsettings + update-alternatives),**完全不写 Nautilus 扩展**。

**理由摘要**(完整论证见软件定义书 ADR-013):
- GNOME 自 3.x 起官方移除 system tray,强推 AppIndicator 扩展是把代价转嫁用户
- nautilus-python 扩展依赖重 + GTK 版本兼容性差;Linux 早有"默认终端"标准入口,根本不需要扩展
- gnome-terminal / kitty / alacritty / wezterm / Foot 等所有主流 Linux 终端都不驻托盘 —— 这才是 Linux 终端 30 年的既有习惯
- 工作量从 2-3 周降至 ~1 周(砍掉 AppIndicator / nautilus-python / 引导弹窗 / 伪装托盘那些"为 Linux 仿 Windows"的代码)

### 跨发行版策略

代码 0 行差异 —— PlatformAdapter `linux.ts` 用的全是 freedesktop / GNOME / 内核标准,跟发行版无关。差异仅在打包脚本:

| Tier | 发行版 | 测试承诺 | 包格式 |
|---|---|---|---|
| **1(必测)** | Ubuntu 22.04 GNOME | 阻塞 release | `.deb` + AppImage |
| **2(best-effort)** | Fedora / CentOS Stream 9 / RHEL 9 | "应该能跑",不阻塞 release | `.rpm` + AppImage |
| **3(不承诺)** | CentOS 7(EOL,glibc 太旧)、KDE/XFCE 桌面、Wayland-only 极端配置 | — | AppImage 自试 |

## BETA-005 · 终端 statusbar 的状态点 idle/exited 时不显示

**状态**:待办 | **分类**:Bug-已知根因 | **优先级**:P1 | **实施批次**:1

### 用户反馈

> 标签页下方标题栏中、标题前面的状态指示器经常不显示。

**澄清(2026-05-16)**:此处"标签页下方标题栏"指 tab bar 下方的 **terminal statusbar**(`<div class="terminal-statusbar">`),不是 Tab 自身。状态点 DOM 一直存在(`<span class="status-dot idle" />`),只是色块没渲染出来。

### 代码调研

**位置**:
- DOM:`src/renderer/components/TerminalView.tsx:1217-1227` 已渲染 `<span className={statusDotClass} />`,classList 正确切换 `status-dot active/idle/exited`
- CSS:`src/renderer/styles/global.css:1036-1045` 只定义:
  ```css
  .status-dot { width: 8px; height: 8px; border-radius: 50%; flex: 0 0 auto; }
  .status-dot.active { background-color: var(--pine, #f0f); }
  ```
  **`.status-dot.idle` 与 `.status-dot.exited` 两条规则完全缺失**,导致 background-color 留空,色块透明不可见。

**触发频率**:配合 BETA-008(新 session 初始即 idle),绝大多数时间 statusbar 的状态点都看不见 → 用户感知为"经常不显示"。Sidebar 的 SessionItem 走自己 inline style(`Sidebar.tsx:724`),不受此 bug 影响。

### 决策(2026-05-16)

在 `global.css` 补两条 CSS 规则,色值与 `Sidebar.tsx:53-57` `STATE_DOT_COLOR` 对齐(idle = `--gold`,exited = `--muted`),不动 TSX。

### 实施要点

1. `global.css:1045` 之后加:
   ```css
   .status-dot.idle {
     background-color: var(--gold, #f0f);
   }

   .status-dot.exited {
     background-color: var(--muted, #f0f);
   }
   ```
2. **不**改 TSX(`statusDotClass` 切换逻辑已正确)
3. **不**抽 STATE_DOT_COLOR 到共享文件:Sidebar 用 inline style 是历史选择,statusbar 走 CSS class 更轻量,两处保持各自实现(避免无谓重构,见 AGENTS.md 第 7 章封箱规则)
4. 人工验证:新建终端瞬间(idle,BETA-008 后)→ 金色点;输入命令 → 绿点;命令完成回到 idle → 金色点;进程退出 → 灰点

### 关联

- BETA-007(完成/失败 ✓/✗ 区分)在此 statusbar 状态点基础上叠加 icon,不再涉及 Tab
- BETA-008 的状态机改动后,初始 idle 金点会立即可见(本工单的直接受益场景)

## BETA-006 · Vite 等长输出工具的状态判定不准

**状态**:待办 | **分类**:功能增强 | **优先级**:P2 | **实施批次**:3

### 用户反馈

> 对 Vite 等长输出工具的状态判定不准确,需要在 Running 转暂停时引入 LLM 复核。

### 代码调研

`src/main/session-manager.ts:878-904` 的 idle 检测阈值 2s。Vite `vite dev` 启动后进入"watching files"无输出,阈值后被误判 idle。

### 决策(2026-05-15)

**仅在 active→idle 跃迁时 LLM 复核 + 设置开关默认关**(用户主动开 + 填 API key 才生效)。LLM 成本不考虑。

### 实施要点

1. `session-manager.ts` markIdle 触发前判断:
   ```ts
   if (this.settings.ai?.statusRecheckEnabled && this.aiClient) {
     const recent = managed.scrollback.slice(-2048).toString('utf8');
     const verdict = await this.aiClient.recheckIdle(recent);
     if (verdict === 'keep-active') return;  // 阻止 markIdle
   }
   ```
2. **设置加 AI 助手面板**(关联 BETA-031):
   - `settings.ai.statusRecheckEnabled: boolean` 默认 false
   - `settings.ai.provider: 'anthropic' | 'openai'`
   - `settings.ai.apiKey: string`(env 字段,默认遮罩)

## BETA-019 · Claude Code 自绘终端运行后闪烁输入光标

**状态**:Workaround 已实施(根因未定位) | **分类**:Bug-未知 | **优先级**:P1 | **实施批次**:3

### 用户反馈

> 对于 Claude code 这种自绘终端的软件,不断闪动的字符后面经常出现闪烁的输入光标,这是非预期行为,而且这是一个真的输入光标闪动,因为他闪动的时候,输入法的 UI 位置也会同步闪动,不是一直都有这个问题,一般是运行一段时间之后才出现的,这是一个严重问题。

**2026-05-16 用户补充**:
- 截图显示光标出现在 Claude Code 任务列表中正在重绘的文字后(`Synt` 后面)
- 复现条件:**运行一段时间才出现 + 重启窗口后消失**

### 决策(2026-05-16)

**根因未定位前实施 workaround 兜底**,业界通行做法:**alt-screen buffer 内禁用 cursorBlink,normal buffer 内启用**。Claude Code / vim / htop 等 TUI 应用一律走 alt buffer,该期间光标即使可见也不闪烁,消除用户报告的视觉困扰。Windows Terminal / iTerm2 / kitty 同此策略。

实现:`TerminalView.tsx`
1. mount effect 内挂 `term.buffer.onBufferChange` listener,buffer.type 切换时 set `cursorBlink = (type === 'normal')`
2. FLK-10 effect 在 not-exited 分支也读 `term.buffer.active.type`,避免 session.state idle↔active 切换覆盖掉 alt-buffer 关闭
3. `cursorStyle === 'underline'` 用作"session.exited 标识位",listener 据此跳过(否则 PTY 已死时仍可能改 blink)

副作用:用户在 vim / nano 等编辑器内 insert mode 也看不到光标闪烁,但光标仍**静态显示**(`cursorBlink: false` 只关闪烁不隐藏),业界用户普遍接受这一 trade-off。

### 调试痕迹(根因排查 — 未结案)

实施 workaround 前做了完整静态分析 + 运行时 HUD 注入抓数据,**仍未定位根因**。痕迹保留在此供下次接手参考。

**已排除的假设**(均有代码级证据):

- ❌ (a) WebGL atlas 把光标残影烤进 cell — 光标走 `RectangleRenderer.renderCursor()` 单独叠加层(`node_modules/@xterm/addon-webgl/src/WebglRenderer.ts:359-361`),`_model.cursor` 每帧先 `undefined` 再按位置重设(line 408),不进 glyph atlas
- ❌ (b) Claude Code 本身行为问题(漏发 `?25l` 等) — 与"重启窗口消失"约束矛盾;同一 Claude Code 二进制在新窗口照样运行,如有 bug 应同样复现
- ❌ (c) IME composition 竞态 — 用户语义是"光标真的在动,输入法 UI 跟着光标位置变",不是 IME 引起
- ❌ (d) `cursorBlink` 重复 set / timer 累积 — xterm `OptionsService.setValue`(`node_modules/@xterm/xterm/src/common/services/OptionsService.ts:132`)在新值等于旧值时不 fire event,FLK-10 effect 反复 set 同值是 no-op;`CursorBlinkStateManager._restartInterval` 也总是先 `clearInterval` 再 set,不会 leak
- ❌ (e) `coreService.isCursorHidden` 被 DECSTR / RIS 翻回 false — 部署运行时 HUD 监控 (见下),实测 bug 时段无任何异常翻转;`isCursorHidden` 翻转全部来自 `setModePrivate` / `resetModePrivate` 标准路径(`?25h` / `?25l`)

**运行时 HUD 注入实验(已删除)**:

为验证 (e),曾在 `src/renderer/debug/beta019-cursor-hud.ts` 用 `Object.defineProperty` 拦截 `coreService.isCursorHidden` 的 setter,翻转瞬间抓 stack trace 解析 InputHandler caller。标题栏挂展开式 HUD 实时显示 `flips` 计数 + 翻转日志。

实测发现:
- bug 出现时 HUD 显示 `flips:0` — 也就是说从 patch 装上后 `isCursorHidden` 没翻转过,仍是初始的 `false`,但光标在闪
- 切换 session 后切回该 session,**瞬间刷出几十条翻转记录** — 已确认是 `TerminalView` mount 时的 `scrollback replay` (line 939 `term.write(all.subarray(...))`) 重放历史字节流引起,**replay 期间的翻转不代表运行时实际状态**
- 排除 (e) 的方式:bug 时段实时翻转计数为 0,DECSTR / RIS / 异常 `?25h` 都没出现

**剩余怀疑**(未验证):

1. **scrollback replay 后 `isCursorHidden` 状态被卡死在 false**:Marina 切 session 时 `TerminalView` 用 `session.id` 作 React key 强制重建 xterm 实例(注释见 TerminalView.tsx:24-25),新实例 `isCursorHidden` 默认 false,replay 历史字节流时按字节顺序处理 `?25l/?25h`,**replay 结束后取决于最后一个 escape**。Claude Code 写一帧的协议是 `?25l → 字符 → ?25h`,所以 replay 结束后 `isCursorHidden=false` 是大概率结果。这与"运行一段时间(积攒 scrollback)+ 切回该 session(触发 replay)+ Claude Code 此时处于思考停顿(没有新字节)"匹配,光标暴露在可见态且 cursorBlink=true → 闪烁。
2. **某种 xterm.js 5.x 已知 bug**:WebGL renderer 长时间运行后的 model / atlas 状态机污染,与代码静态分析得到的结论冲突 — 需要 xterm.js issues 区搜索类似复现

## BETA-041 · 粘贴误报"多行粘贴意外复制"

**状态**:待办 | **分类**:Bug-已知根因 | **优先级**:P2 | **实施批次**:1

### 用户反馈

> 粘贴行为不够理想:粘贴到 Agent 时偶尔会提示"多行粘贴意外复制"。

### 代码调研

**位置**:`src/renderer/components/TerminalView.tsx:435-458`

```ts
const normalized = text.replace(/\r\n?/g, '\n').replace(/\n$/, '');
const lineCount = normalized.split('\n').length;
```

可能误判:单行命令带尾换行 → 应该 1 行,实际可能统计为 2 行(具体取决于输入是否含混合换行符)。

### 决策(2026-05-15)

**仅修多行判定逻辑**,不加设置开关。

### 实施要点

1. `TerminalView.tsx:435+` 改 normalize:
   ```ts
   const lines = text.split(/\r\n|\r|\n/);
   while (lines.length > 0 && lines[lines.length - 1] === '') lines.pop();
   const lineCount = lines.length;
   ```
2. 单测:加 `terminal-view.test.tsx`(若已有,补 case):
   - 单行 + 单尾换行 → 1 行
   - 单行 + 多尾换行 → 1 行
   - 双行 + 尾换行 → 2 行
   - 混合 `\r\n` 和 `\n` → 正确计数

## BETA-043 · 路径存在性没好好检查

**状态**:待办 | **分类**:Bug-已知根因+演进 | **优先级**:P2 | **实施批次**:2

### 用户反馈

> 路径存在性没有好好检查一下,没有考虑用户修改路径的情况。

### 代码调研

- `src/main/index.ts:430+ sanitizeOpenHerePath` 已对**冷启动入口**做 `existsSync` 检查
- **运行时**点击侧栏路径没有 re-check;`createSession` 时 spawn 失败才发现

### 决策(2026-05-15)

`createSession` 被动检查 + 启动时扫一轮(不做定期 60s 后台扫)。

### 实施要点

1. **createSession 被动检查**:
   - `session-manager.ts:createSession` spawn 前 `fs.statSync(cwd)`,失败 → SessionManagerError + renderer toast
2. **启动扫描**:
   - 主进程 `bootstrap` 末尾异步对 PathManager 内所有路径做 stat
   - 失败的路径在 PathNode 上标记 `invalid: true`(`src/shared/types.ts` PathNode 加字段)
   - 触发一次 `pathTreeUpdated` 让 renderer 重渲
3. **UI 表示**:
   - Sidebar 渲染时,`node.invalid === true` 加 ⚠️ 图标(`AlertTriangle` lucide) + opacity 0.6
   - 鼠标悬停显示 tooltip "路径不可访问"
   - **不自动删**(用户手动决定是否右键删除)
4. **不做定期 60s 后台扫**(资源考虑)

### 关联

- BETA-011 系统路径(`%TEMP%` 等)永远存在,无需扫,但走同样代码路径不报错即可
