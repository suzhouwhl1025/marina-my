## 1. 产品定位

Marina 是一个 **Windows 平台的桌面终端管理器**,服务于"在多个不同路径下并发运行多个长时间任务(包括 AI coding agent 和普通命令行任务)"的开发者。

它**不是**一个传统终端模拟器的迭代,而是对"终端会话应该如何被管理"这个问题的重新回答:

* 传统终端把"窗口生命周期"和"shell 进程生命周期"绑定 —— 关窗即死。
* 传统终端以"profile / tab"为组织维度 —— 用户需要靠记忆和手动 cd 来导航。
* 传统终端假设用户是命令行专家 —— 重度依赖快捷键和配置文件。

Marina 反其道而行:

* **会话独立于 UI 存活** —— 关掉所有窗口,会话仍在守护进程内运行,直到用户主动结束。
* **路径作为一等公民** —— 所有终端按"工作位置"组织,不再按 profile 或 project。
* **GUI 优先** —— 用户从不需要手敲路径,从不需要记忆快捷键,所有操作都可以用鼠标完成。

### 一句话定位

> "你的终端会话不应该因为关掉窗口就死掉。"

## 2. 设计哲学

Marina 的所有设计决策都源自四条核心原则:

### 原则 1:Path 是稳定的,Session 是廉价的,UI 是临时的

| 层级 | 性质 | 用户对它的态度 |
|------|------|--------------|
| Path(路径) | 长期、稳定、用户主动管理 | 收藏、命名、组织 |
| Session(会话) | 短期、用完即弃、自动产生与销毁 | 启动、使用、关闭 |
| UI 窗口 | 完全临时、可随时开关、彼此平等 | 打开看一眼,不用了关掉 |

**这条原则推导出**:
* 三栏侧栏分类(收藏/临时/最近)以 path 为单位
* session 不维护跨会话的累积上下文
* 关闭窗口完全不影响 session
* 守护进程持有所有 session 和数据,窗口只是"观察窗"

### 原则 2:不让用户输入路径,只让用户点击路径

`cd` 命令是一个 1971 年的设计,它打错就报错或静默进入错误位置,代价高昂。Marina 把"切换工作位置"重构为"点击一个已存在的路径条目",彻底绕过文本输入路径这个易错动作。

**这条原则推导出**:
* 所有路径必须通过文件夹选择器或 Explorer 拖入,不能手敲
* 路径在 UI 上永远可见,用户不需要 `pwd` 校对
* 切换路径 = 点击侧栏 = 启动新会话或聚焦已有会话,而不是 `cd`

### 原则 3:用户决策最少化

工具应该自动处理一切可以自动处理的事,只在用户必须做选择时才打扰用户。

**这条原则推导出**:
* 路径在三个分类之间的流转完全自动(状态机驱动)
* session 标签在路径间的归属由 cwd 自动决定(OSC 1337 hook)
* 终端切换时的 resize 自动发生
* 状态指示自动检测,不需要用户配置
* 所有窗口平等,没有"主次"概念,用户不需要思考"我该关哪个"

### 原则 4:窗口与应用解耦

应用 ≠ 窗口。窗口是临时的观察工具,应用是承载所有 session 的守护进程 + 托盘。

**这条原则推导出**:
* 关闭任何一个窗口,完全不影响其他窗口或任何 session
* 关闭所有窗口,应用进入"纯托盘模式",继续在后台运行
* 应用退出由用户主动从托盘触发,不和窗口生命周期绑定
* 所有窗口平等,不存在"主窗口"概念

> **Linux 平台例外(v1.6,ADR-013)**:此条对 Linux 不完全成立。GNOME 自 3.x 起官方移除 system tray,Marina 在 Linux 上不再做"沉托盘"行为(`lifecycleModel: 'no-persistence'`),关闭最后一个窗口 = 应用退出 = session 全销毁。配套阻塞 modal 在"最后一个窗口 + 仍有非 exited session"时弹出二次确认,作为无托盘环境的唯一 UX 妥协。Windows(`tray-resident`)和 macOS(`dock-resident`,Electron darwin 默认行为)保持原行为不变。"窗口零成本开关"在非最后窗口 / 全 exited session 关闭时三平台一致,只有"最后一个非空窗口关闭"在 Linux 上需要确认。详见 ADR-013 与第 12 章 `lifecycleModel`。

## 3. 目标用户

### 主要用户

* **Windows 平台的开发者**,日常工作中需要并发运行多个长时间命令行任务
* **同时运行多个 AI coding agent**(Claude Code、Codex、OpenCode 等)的开发者
* **不愿意学习 tmux / zsh / dotfiles 的用户** —— 他们需要工具开箱即用,而不是花一周配置环境
* **跨多个不相关项目工作的用户** —— 路径不一定是 git 仓库,可能是脚本目录、日志路径、网络盘等

### 典型场景

* RTL/EDA 工程师在多个 run 目录之间跳转,跑综合、仿真、回归测试
* 全栈开发者在前端项目 + 后端项目 + 脚本目录之间切换
* 同时让 5 个 agent 在 5 个不同项目里干活,不希望忘记任何一个的状态
* 跑长时间编译/部署/AI 任务,不希望因为关错窗口前功尽弃

### 非目标用户

* 重度 tmux / zsh 用户(他们已有满意工作流,Marina 不强迫他们改变)
* 团队协作场景中需要"项目级 session 共享"的用户(请用 Nimbalyst 等)
* 远程服务器管理为主的用户(请用 Windows Terminal + SSH 或专用 SSH 客户端)

## 4. 核心心智模型

### 三个一等公民,一种临时实体

Marina 的世界由四种实体构成:

```
应用(守护进程 + 托盘)—— 单例,长期运行,用户从托盘控制其生死
  │
  ├── Path(路径)—— 永久,用户管理,数据持久化
  │     │
  │     └── Session 1(会话)—— 临时,运行时产生,完成即销毁
  │     └── Session 2
  │
  └── Window(窗口)—— 临时观察工具,任意开关,彼此平等
```

**关键关系**:
* 应用持有 Path 数据和所有 Session
* Session 归属于 Path(由 cwd 决定)
* Window 是观察 Path 和 Session 的临时窗口,**不持有数据**
* 关闭 Window 不影响 Path,不影响 Session,不影响应用

### Path 的三种状态(由状态机自动管理)

```
[最近 Recent]
  ↑↓ (用户主动加入/移除收藏)
[收藏 Bookmarked]
  ↑↓ (有终端在跑/全部关闭)
[临时 Temporary]
```

**精确定义**:

| 分类 | 定义 | 是否自动 |
|------|------|---------|
| **收藏** | 用户主动加入的路径,永久保留 | 用户主动 |
| **临时** | 当前有终端在跑、且不在收藏区的路径 | 完全自动 |
| **最近** | 历史上开过终端、当前不在收藏也不在临时的路径 | 完全自动 |

**流转规则**:
* 在某路径新建终端 → 路径进入"临时"(若不在收藏)
* "临时"路径的最后一个终端关闭 → 路径离开"临时",进入"最近"
* "最近"或"临时"路径被用户加入收藏 → 进入"收藏",离开原分类
* "收藏"路径被用户移除收藏 → 回到"最近"(若有历史)或消失

**容量**:
* 收藏:无限
* 临时:无限(由当前活跃 session 数量决定,自然有限)
* 最近:最多 30 个,按最近活动时间排序,超出自动淘汰最旧的

### Session 的生命周期

```
[创建] → [活跃] ↔ [空闲] / [等待输入] → [已退出] → [销毁]
                                              ↑
                                       用户主动右键关闭 / 应用退出
                                       (无时限自动过期)
```

* **创建**:用户从某路径新建终端,守护进程启动 PTY 进程
* **活跃**:PTY 最近有字节输出
* **空闲**:PTY 一段时间无输出,但进程仍在
* **等待输入**:通过 OSC 1337 命令完成事件检测,处于 prompt 状态(V1.1)
* **已退出**:进程已结束(正常或异常),session 灯显灰色 ⚫,scrollback 仍可见;**不可重启**;无时限自动过期
* **销毁**:用户主动右键"关闭"、或应用完全退出时(批量销毁)

**核心承诺**:从 [创建] 到 [销毁] 的整个过程**与任何窗口的开关无关**。即使关闭所有窗口,守护进程依然管理这些 session。

**已退出 session 的语义**:像传统终端"crashed but tab still open"的体验。用户看到 ⚫ 灯 + 完整退出前的 scrollback;选项卡有右键"关闭"。**没有重启功能**,如果用户想"再跑一次",在同一 path 新建一个 session 即可。详见 ADR-008。

### Window 与 Session 的关系

```
N 个 Window(窗口,N ≥ 0)<──── 任意多对一/一对多 ────> M 个 Session(会话)
                                                            在守护进程内
                                                            持续运行
```

**核心约束 1:任一 Session 同一时刻只有 0 或 1 个 Owner Window**

* Owner Window 持有该 session 的输入/输出渲染权
* 非 Owner 窗口在自己的 UI 中**只能看到 session 存在**(灰显标签),不能直接显示其内容
* 用户在非 Owner 窗口点击灰显标签 → 系统聚焦真正的 Owner 窗口
* Session 可以无 Owner(所有窗口都关了,或最后一个窗口主动释放)

**核心约束 2:N 可以是 0**

应用启动后,可以保持 0 个窗口,只有托盘和守护进程运行。任何时候用户从托盘可以新开窗口。

**优雅之处**:这两个约束彻底消除了"多窗口同步同一 session"的所有复杂度,同时让窗口的开关变成纯粹的"观察方式选择",不绑定任何业务逻辑。

### 应用生命周期

```
[启动] → [运行(0~N 窗口)] → [完全退出]
              ↑↓
         窗口数量随时变化
         不影响应用本身
```

**应用何时退出**:
1. 用户从托盘菜单选"完全退出 Marina"
2. 系统关机/重启
3. 应用崩溃

**关闭窗口绝不等于关闭应用。**

## 5. 功能规格

### 5.1 必做(MVP V1)

#### 5.1.1 路径管理
* 添加收藏路径(通过文件夹选择器或拖拽 Explorer 文件夹)
* 移除收藏路径
* 重命名收藏路径(自定义显示名)
* 收藏路径排序(拖拽改变顺序)
* 临时路径自动出现/消失
* 最近路径自动记录,容量上限 30
* 用户右键最近路径可"加入收藏"或"从最近移除"

#### 5.1.2 Session 管理
* 在任一路径新建终端(默认 shell 或选择启动模板)
* 关闭终端:调用 `node-pty` 的 `kill()`,Windows ConPTY 立即终止子进程(无 SIGTERM/grace 周期 — 这是 ConPTY 模型决定的,不是软件设计选择)
* 已退出 session 保留在 UI 上(灯显灰色),用户可右键"关闭"销毁;**不可重启**
* 列出所有 session,按路径分组
* 状态指示:活跃 / 空闲 / 已退出(MVP 第一版只这三种)
* **session 与 path 的归属**:session 创建时记录的 path **永久绑定**,生命周期内不变。session 内 cwd 变化(cd 等)不会让 session 在 UI 上换 path,而是在该 session 的标签上显示 ⚠️ 提示真实 cwd(详见 ADR-008)

#### 5.1.3 启动模板
* 内置:Shell(系统默认)、Claude Code、Codex、OpenCode
* 用户自定义模板(命令、参数、环境变量、退出后行为)
* 每个收藏路径可设置"默认模板"(双击直接启动,跳过模板选择)

#### 5.1.4 终端体验
* xterm.js 渲染,支持 ANSI 颜色、Unicode 11、URL 自动识别可点击
* 终端搜索(Ctrl+F):弹出搜索栏,显示 `current/matches` 命中数;Enter 跳下一个 / Shift+Enter 上一个 / Esc 关闭。Ctrl+F、Esc 通过 `term.attachCustomKeyEventHandler` 在 xterm 把它们转成 PTY 字节流之前拦截,避免把 ^F / 0x1B 误送到 shell
* 复制(选中即可,可在设置关)、粘贴(右键 → 弹菜单 / 直接粘贴,设置可选)
* **多行粘贴确认**:剪贴板内容含换行(`\n`)时弹原生 `confirm`,显示行数 + 200 字预览;用户确认后再写入 PTY,避免误把脚本喂给 shell 立即执行
* 字体可配置 — 终端字体下拉枚举系统已安装等宽字体(走 `window.queryLocalFonts()`,主进程已自动放行 `local-fonts` 权限),并在顶部置顶推荐字体(Cascadia Mono / JetBrains Mono / Consolas / 霞鹜文楷 Mono 等);未装的推荐字体仍显示并标注"(未装)";不在推荐里的系统字体按字母升序展示;额外提供"自定义"输入框直接写 CSS `font-family`
* 字号 / 行高 / UI 字体 / UI 缩放 均可配置(数值范围见 6.6.2)

#### 5.1.5 多窗口支持
* 任意时刻可开任意多个窗口(包括 0 个)
* 所有窗口完全平等,不存在"主窗口"概念
* 关闭任意窗口不影响应用、其他窗口或任何 session
* 关闭所有窗口 → 应用进入"纯托盘模式",可随时从托盘新开窗口
* 每个窗口独立维护:当前选中路径、当前选中 session、侧栏展开状态
* 全局共享:收藏数据、最近数据、所有 session 列表、临时分类、设置
* 窗口标签:Window 1 / Window 2 / ... 自动编号(运行时分配,不持久化)
* 灰显标签 → 点击聚焦 Owner 窗口
* Owner 切换:用户在非 Owner 窗口点击同路径下的灰显 session(可选,V1 优先级低)

#### 5.1.6 守护进程与持久化
* 守护进程 = Electron Main Process,生命周期独立于任何窗口
* 第一个 Marina 实例启动时拉起守护进程(单例锁保护,通过 `app.requestSingleInstanceLock()`)
* 第二次启动 Marina.exe → 转发到已运行的实例,新开一个窗口
* 任何窗口关闭都不退出应用
* 用户从托盘"完全退出"→ 守护进程退出
* 数据持久化:收藏、最近、模板、设置 → JSON 文件,原子写

#### 5.1.7 系统托盘
* 常驻系统托盘,应用运行期间永不消失(除非"完全退出")
* 图标动态:有 session 在跑 / 全空闲(V1.1 加"有 session 等待用户")
* 单击托盘图标:聚焦最近活动的窗口;若无窗口,新开一个
* 右键菜单详细规格见 [6.5 系统托盘](#65-系统托盘)

#### 5.1.8 CWD 实时跟踪
* 通过 OSC 1337 序列追踪 session 内 cwd 变化
* **cwd 仅用于 UI 提示,不驱动 path 迁移**(ADR-008):session.currentCwd 与 session.pathId 对应的目录不一致时,该 session 的标签上显示 ⚠️ 图标 + tooltip 真实 cwd
* shell hook 注入透明完成,不污染用户 dotfiles(PowerShell 包装 prompt 函数;cmd.exe 嵌入 OSC 序列到 PROMPT 环境变量)
* 兜底策略:启动后首 5 秒内若未收到任何 OSC 1337,启动 5 秒间隔的进程 cwd 轮询(`PlatformAdapter.getProcessCwd`);收到首条 OSC 后立即关闭轮询。
  * **V1 实际状态**:`WindowsAdapter.getProcessCwd` 始终返回 `null` — 真正实现需要 `NtQueryInformationProcess` + 读 PEB,得引入 `ffi-napi` 原生包,违反 AGENTS.md 1.2 边界 2。SessionManager 内的轮询调度结构已就位,未来加原生绑定即生效。所以 V1 阶段 OSC 1337 hook 是 cwd 跟踪的**唯一**可靠路径。

#### 5.1.9 主题系统
* 内置 **7 套主题**:Rose Pine(默认)、Rose Pine Dawn、Rose Pine Moon、Cutie、Business、Ubuntu、Windows Terminal
* CSS variables 驱动,主题切换即时生效;14 个语义变量(`--base` / `--surface` / `--overlay` / `--highlight-{low,med,high}` / `--muted` / `--subtle` / `--text` / `--love` / `--gold` / `--rose` / `--pine` / `--foam` / `--iris`)由 `[data-theme="..."]` 切换
* xterm.js 主题与 UI 主题同步切换(`XTERM_THEMES` 在 `TerminalView.tsx` 中与 `global.css` 同源维护)
* 所有 `var()` 调用必须带 `#f0f` fallback,防止未定义变量污染
* stylelint 静态检查,确保所有 var() 引用的变量在所有主题里都有定义
* UI 系统图标使用 **lucide-react** 图标库(CP-4 勘误 #11 引入,替代原 Emoji);用户数据中的 emoji(`Template.icon`)保持原样,因为这是用户自行编辑的内容

## 7. 交互规格

### 7.1 不做的交互

Marina 明确**不做**以下交互:

* **应用内功能键**:任何不在 §7.2.2 终端键位清单 / §7.2.1 鼠标交互之外的快捷键(无 Ctrl+T 新 tab、无 Ctrl+W 关 tab、无 Ctrl+Tab 切换 session、无 Alt+1..9 跳 tab、无菜单加速键、无任何 vim/emacs 风格)
* **全局快捷键**:不注册任何全局快捷键
* **命令输入框**:UI 中无任何"输入路径"或"输入命令"的文本框,所有路径必须通过文件夹选择器或拖拽

理由:这些交互假设用户是"愿意学习快捷键的命令行专家",而 Marina 的目标用户是"愿意用鼠标的普通开发者"。坚持鼠标优先工作流,产品才能保持自洽。

**§7.2.2 清单是唯一权威来源**:任何不在该清单内的键位必须透传给 PTY,renderer 层不得消费、不得弹 UI、不得改 state。任何 PR 加新键位必须先改 §7.2.2 并同步更新 `docs/键盘交互规范.md` 与 `src/renderer/terminal-keybindings.ts`。

### 7.2 必做的交互

#### 7.2.1 鼠标操作
* 点击侧栏路径:选中路径,在右侧显示该路径标签页 + 第一个终端(或加号)
* 点击侧栏路径展开箭头:展开/收起该路径下的终端列表
* 双击侧栏路径:在该路径用默认模板新建终端
* 点击侧栏终端条目:切换到该 session(本窗口)或聚焦 Owner 窗口
* 拖拽 Explorer 文件夹到侧栏收藏区:加入收藏
* 拖拽 Explorer 文件夹到终端区:在该路径新建终端
* 右键各处:上下文菜单

#### 7.2.2 终端内交互
* 选中文本即复制到剪贴板(类 Linux 风格,可在设置里关闭)
* 右键终端区:在设置中可选"弹菜单"(默认)或"直接粘贴"。弹菜单含:复制(选中时)/ 粘贴 / 清屏 / 搜索
* **多行粘贴确认**:剪贴板含换行时弹原生 `confirm`,显示行数 + 200 字预览;用户确认后才送入 PTY(防误粘脚本立即执行)。单行粘贴不打扰
* 滚轮:滚动 scrollback
* Ctrl+滚轮 / Cmd+滚轮:调整字号

##### 终端键位清单(唯一权威 — 此清单之外的所有键位透传 PTY)

| 功能 | Windows / Linux | macOS | 备注 |
|------|-----------------|-------|------|
| 复制 | `Ctrl+C`(有选区时)/ `Ctrl+Shift+C` / `Ctrl+Insert` | `Cmd+C` / `Ctrl+Shift+C` | 无选区时 `Ctrl+C` 透传 SIGINT |
| 粘贴 | `Ctrl+V` / `Ctrl+Shift+V` / `Shift+Insert` | `Cmd+V` / `Ctrl+Shift+V` | 所有路径走同一个 `handlePaste`(bracketed paste / 大粘贴警告 / ESC 检测) |
| 搜索 | `Ctrl+F` | `Cmd+F` | 唤出搜索栏,Enter=下一个,Shift+Enter=上一个,Esc=关 |
| 字号 | `Ctrl+Wheel` | `Cmd+Wheel` | 范围 8-24,本地立即生效,跨窗口 100ms debounce 广播 |
| 关搜索栏 | `Esc`(仅搜索栏可见时) | `Esc`(同) | 其他时刻 Esc 透传给 PTY(vim 等需要) |

**实现不变式**:
1. 上述键位通过 `term.attachCustomKeyEventHandler` + capture-phase `paste` listener 拦截,在 xterm 把它们转成 PTY 字节 / 让 native paste 进 PTY 之前接管
2. IME composition 期间(`isComposing` 或 `keyCode===229`)所有键位透传,renderer 不消费 — 避免打断中文 / 日文 / 韩文 IME 状态机
3. Esc 优先级走 `UiOverlayStack`:栈顶 overlay 吃 Esc,无 overlay 时透传给 PTY
4. 复制三套(`Ctrl+C 有选区` / `Ctrl+Shift+C` / `Ctrl+Insert`)统一 `copyAndClearSelection` — 否则下次 `Ctrl+C` 因残留选区无法发 SIGINT(CPB-C3)
5. session 切换 scrollback replay 期间 terminal-host `visibility:hidden + inert`,replay 完才 reveal — 按键不会进错位 focus(SCROLL-1)
6. 任何 PR 新增 / 改 / 删键位必须同时改:本节清单 + `docs/键盘交互规范.md` + `src/renderer/terminal-keybindings.ts` + 设置页快捷键速查卡片

### 7.3 拖拽行为

| 拖拽源 | 拖拽目标 | 行为 |
|--------|---------|------|
| Explorer 文件夹 | 侧栏收藏区 | 加入收藏 |
| Explorer 文件夹 | 终端区 / 加号区 | 在该路径新建终端(默认模板) |
| 侧栏路径条目 | 收藏分类内 | 改变收藏顺序 |
| 标签页 | 标签栏内 | 改变顺序(V1.2) |
| 标签页 | 窗口外 | 拆分到新窗口(V1.2) |

### 7.4 关闭与退出的精确语义

这是 Marina 与传统应用最大的不同之一,必须精确定义:

#### 7.4.1 关闭单个窗口
* 触发方式:点窗口右上角 × / Alt+F4 / 系统菜单关闭 / 托盘菜单"关闭所有窗口"中之一
* 行为:
  1. 该窗口立即消失
  2. 它持有的所有 session 的 `owner_window_id` 变为 `null`
  3. 那些 session 在守护进程中继续运行
  4. 其他窗口完全不受影响
  5. 应用继续运行
  6. **绝不弹任何确认对话框**

#### 7.4.2 关闭所有窗口
* 等价于逐个关闭每个窗口
* 关掉最后一个后,应用进入"纯托盘模式":
  - 没有任何窗口可见
  - 托盘图标继续显示
  - 守护进程继续管理 session
  - 用户从托盘可随时新开窗口
* **不弹任何确认**

#### 7.4.3 完全退出应用
* 触发方式:仅托盘菜单"完全退出 Marina"
* 行为:
  1. 若有 session 在跑且"完全退出前确认"开启 → 弹确认对话框
  2. 用户确认 → 给所有 PTY 发 SIGTERM,等待 5 秒
  3. 强制 kill 仍未退出的 PTY
  4. 持久化所有数据
  5. 守护进程退出
  6. 托盘图标消失
  7. 应用进程结束

#### 7.4.4 系统级关机/重启
* Windows 发送系统关机消息时,守护进程响应:
  1. 持久化数据
  2. 给所有 PTY 发 SIGTERM
  3. 退出
* 不阻止系统关机

### 8.1 应用生命周期状态机

```
              ┌──────────────────────┐
              │  未运行 NotRunning  │
              └──────────┬───────────┘
                         │ 用户启动 Marina.exe

                         ↓
              ┌──────────────────────┐
              │  启动中 Starting    │
              │  - 守护进程初始化   │
              │  - 加载持久化数据   │
              │  - 创建托盘图标     │
              └──────────┬───────────┘
                         │ 根据"启动时行为"设置
              ┌──────────┴──────────┐
              ↓                     ↓
    ┌──────────────────┐   ┌──────────────────┐
    │ 运行中(有窗口)│   │ 运行中(纯托盘)│

    └────────┬─────────┘   └────────┬─────────┘

             │  ←── 关闭最后一个窗口  │
             │                       │
             │  ── 从托盘新建窗口 ──> │
             │                       │
             ↓                       ↓
             └──────────┬────────────┘
                        │ 用户从托盘"完全退出"
                        ↓
              ┌──────────────────────┐
              │  退出中 Exiting     │
              │  - SIGTERM 所有 PTY │
              │  - 持久化数据       │
              │  - 托盘图标消失     │
              └──────────┬───────────┘
                         ↓
              ┌──────────────────────┐
              │  未运行 NotRunning  │
              └──────────────────────┘
```

**注意**:
* "运行中"两个子状态可以反复切换,不影响应用本身
* 用户可以通过启动 Marina.exe 第二次来"新开窗口"(单例锁会转发给已运行的实例)

### 8.2 Path 状态机

```
            ┌──────────────────────────────────┐
            │         (用户主动加入收藏)       │
            ↓                                  │
       ┌────────┐  (有终端在跑)         ┌────────┐
       │ 最近   │ ────────────────────> │ 临时   │
       │Recent  │                       │Temp    │
       │        │ <──────────────────── │        │
       └────────┘  (最后一个终端关闭)    └────────┘
            ↑                                  ↑
            │         (用户主动加入收藏)       │
            │                                  │
            │  ┌────────┐                     │
            └─ │ 收藏   │ ←───────────────────┘
   (移除收藏) │Bookmark│  (用户主动加入收藏)
              │   ed   │
              └────────┘
```

**注意**:
* 收藏路径即使没有终端在跑也不会进入临时(收藏优先级高)
* 收藏路径有终端在跑时,仍只显示在收藏分类(不重复显示)
* 路径首次出现(从未开过终端、从未被收藏)只能通过"加入收藏"或"在此新建终端"两种方式

### 8.3 Session 状态机

```
                      ┌─────────────┐
                      │  Created    │ (PTY 启动)
                      └──────┬──────┘
                             │ (持续接收 PTY 字节流)
                             ↓
        ┌───────────┐  有输出 ┌──────────┐  无输出  ┌──────────┐
        │ Idle      │ ←────── │ Active   │ ──────→ │ Idle     │
        │ (空闲)    │         │ (活跃)   │ N秒     │          │
        └─────┬─────┘         └──────────┘         └─────┬────┘
              │  (PTY 进程退出)                           │
              ↓                                           ↓
              └────────────────┬─────────────────────────┘
                               ↓
                      ┌─────────────────┐
                      │  Exited         │ (灯显灰色 ⚫,无时限保留)
                      │  (已退出)       │ scrollback 完整保留
                      └─────────┬───────┘
                                │ (用户右键"关闭" / 应用完全退出)
                                ↓
                          ┌─────────┐
                          │Destroyed│
                          └─────────┘
```

**Active / Idle 阈值**:
* Active → Idle:无字节输出超过 2 秒(可在设置调整)
* Idle → Active:有任何字节输出

**Exited 的特性**:
* 进入即"挂起",**不再消耗任何资源**(PTY 句柄已释放,无定时器)
* `exitCode` 字段记录进程退出码;`scrollback` 仍持有
* **不能从 Exited 回到 Active**(无重启路径)。用户想"再跑一次" → 在同一 path 新建一个 session
* 销毁来源:用户主动右键"关闭" / 应用完全退出时批量销毁

### 8.4 Window 与 Session 的所有权状态

每个 Session 有一个 `owner_window_id` 字段,可能值:
* 某个 Window 的 ID — 该 Window 当前持有此 session
* `null` — 此 session 当前无 owner

**所有权转移规则**:
* Window A 持有 Session X,用户在 Window A 切换到 Session Y → A 释放 X(变 null),A 持有 Y
* Window A 持有 Session X,Window A 关闭 → X 的 owner 变 null
* 用户在 Window B 点击 X(此时 X 的 owner 是 A)→ 守护进程聚焦 A,所有权不变
* Session X 无 owner,用户在 Window B 点击它 → B 接管,X.owner = B
* 应用进入"纯托盘模式"(无窗口)→ 所有 session 的 owner 都是 null

## 9. 技术架构

### 9.1 总体架构

```
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│  UI Window 1     │  │  UI Window 2     │  │  UI Window N     │

         │   Electron IPC      │                     │
         └──────────┬──────────┴─────────────────────┘
                    ↓
         ┌──────────────────────────────┐
         │   Main Process (守护进程)    │
         │  ┌────────────────────────┐  │

         │  └────────────────────────┘  │
         │  ┌────────────────────────┐  │
         │  │ Settings Manager       │  │
         │  │  (设置数据 + 广播)     │  │
         │  └────────────────────────┘  │
         │  ┌────────────────────────┐  │
         │  │ Window Manager         │  │
         │  │  (窗口列表 + 编号)     │  │
         │  └────────────────────────┘  │
         │  ┌────────────────────────┐  │
         │  │ Persistence            │  │
         │  │  (JSON 文件 I/O)       │  │
         │  └────────────────────────┘  │
         │  ┌────────────────────────┐  │
         │  │ Tray                   │  │
         │  └────────────────────────┘  │
         │  ┌────────────────────────┐  │
         │  │ Platform Adapter       │  │
         │  │  (Windows-only V1)     │  │
         │  └────────────────────────┘  │
         └──────────────────────────────┘
                    ↓
         ┌──────────────────────────────┐
         │  System APIs                 │
         │  ├─ Tray (Electron)          │
         │  ├─ ConPTY (node-pty)        │
         │  ├─ NtQueryInformation       │
         │  │   Process (cwd 兜底)      │
         │  ├─ Registry (Explorer 集成) │
         │  └─ FileSystem               │
         └──────────────────────────────┘
```

**关键说明**:
* 守护进程 = Electron Main Process,不是独立的可执行文件
* 每个 UI Window = 一个 BrowserWindow + 一个 Renderer 进程
* Main 与 Renderer 之间用 Electron 标准 IPC(`ipcMain` / `ipcRenderer`)
* 多窗口共享同一个 Main 进程,自然共享所有数据
* Window 数量可以是 0:此时 Main 仍在运行,只有托盘可见

### 9.2 进程职责划分

#### 9.2.1 Main Process(守护进程)
* 创建和管理所有 PTY(node-pty)
* 维护 session 状态、scrollback buffer
* 维护 path 数据(收藏/临时/最近)
* 维护 window 列表与 owner 关系
* 维护设置数据 + 广播变更
* 持久化数据(JSON I/O)
* 系统托盘
* 系统 API 调用(注册表、cwd 查询、单实例锁)
* 单一可信数据源
* **生命周期独立于任何窗口**:监听 `app.on('window-all-closed')` 但不调用 `app.quit()`

#### 9.2.2 Renderer Process(每个窗口一个)
* 渲染 React UI(侧栏、标签页、终端区、设置 UI)
* 维护**本窗口独立的视图状态**:
  - 当前选中的路径(选中状态)
  - 当前选中的 session(显示哪一个)
  - 侧栏分类的展开/折叠状态
  - 路径展开状态
  - 当前是否在设置视图
* 处理用户交互,通过 IPC 把意图发给 Main
* 接收 Main 推送的 PTY 字节流,送给 xterm.js 渲染
* 接收 Main 推送的设置变更,实时应用

**关键原则**:Renderer 不存储任何全局状态。所有"全局可见"的数据都从 Main 拉取或订阅。

### 9.3 数据流示例

#### 用户在路径 A 新建一个 Claude Code 终端

```
1. Renderer (Window 1):用户点击 "+" 按钮,选 Claude Code 模板
2. Renderer → Main:IPC 调用 createSession({ pathId: 'path-a', templateId: 'claude-code' })
3. Main:
   a. 启动 node-pty,cwd = path A 的实际目录,command = "claude"
   b. 注入 OSC 1337 hook(根据所选 shell)
   c. 创建 Session 对象,owner_window_id = window1.id
   d. 把 Session 添加到 path A 的 sessions 列表
   e. 如果 path A 不在收藏,加入临时分类
4. Main → 所有 Renderer:广播 path 树更新事件
5. Main → Window 1:推送该 session 的字节流(只推给 owner)
6. Window 1 Renderer:收到字节流,送给对应的 xterm.js 实例
```

#### 用户在 Window 2 点击 Window 1 持有的 session

```
1. Window 2 Renderer:用户点击灰显 session
2. Renderer → Main:IPC focusSessionWindow({ sessionId })
3. Main:查 session.owner_window_id == window1.id
4. Main → Window 1:命令该窗口 focus()
5. Window 1 浮到前台,选中那个 session 的标签
```

#### Session 内 cwd 变化(自 v1.2 起,**不再驱动 path 迁移**)

```
1. PTY 内的 shell 执行用户的 `cd /new/path`
2. PTY 输出 OSC 1337 序列:\x1b]1337;CurrentDir=/new/path\x07
3. Main 解析字节流时识别到 OSC 1337,提取序列后透明转发剩余字节给 owner
4. Main 更新 session.currentCwd(session.pathId 不变)
5. Main → 所有 Renderer:广播 sessionStateChanged({ id, currentCwd })
6. Renderer 比较 session.currentCwd 与 session 所属 path 的 path 字段:
   - 一致 → 标签正常显示
   - 不一致 → 标签旁显示 ⚠️ 图标,tooltip 显示真实 cwd
7. **path 状态机不被触动**(session.pathId 永久绑定,见 ADR-008)
```

#### 用户关闭 Window 2(不影响其他)

```
1. 用户点击 Window 2 的 ×
2. Electron 触发 'close' 事件
3. Main 监听到 → 找到 Window 2 持有的所有 sessions
4. 对每个 session:owner_window_id 改为 null
5. Window 2 真正销毁(Renderer 进程退出)
6. Main → 所有剩余 Renderer:广播 window-list-updated + sessions-state-updated
7. Main 检查窗口数量:
   - 还有窗口 → 继续运行
   - 没有窗口 → 进入"纯托盘模式",不退出应用
```

#### 用户在设置 → 数据 → 导入归档(自 v1.3 起,**in-memory replace**,不重启应用)

```
1. Renderer:用户点"导入…"按钮
2. Renderer → Main:IPC SETTINGS_IMPORT
3. Main:dialog.showOpenDialog 选 JSON 文件
4. Main:fs.readFile + JSON.parse + validateArchive(format / version 校验)
5. Main:dialog.showMessageBox 二次确认("整体覆盖现有配置,继续?")
6. 用户确认 → applyArchiveInMemory:
   a. settingsManager.replaceAll(archive.settings)   // deep-merge + validate + emit
   b. templatesManager.replaceAll(archive.templates) // mergeBuiltins + persist + emit
   c. pathManager.replaceAll(archive.bookmarks/recent) // persist + emit
   d. await Promise.all([flush, flush, flush])      // 等磁盘 debounce 落盘
7. Main → 所有 Renderer:广播 settings:changed / templates:updated /
                          path:tree-updated / bookmarks:updated
8. 各窗口 Renderer:reducer 应用变更,UI 立即刷新
9. 运行中的 PTY session 不被打扰(导入只换数据,不动 session 树)
```

**为什么不再 `app.relaunch + app.quit`**:dev 模式下 `app.relaunch` 与 Vite HMR daemon 协作不稳,新进程拉起后 renderer URL 失效,用户看到"导入后无法正常渲染"。In-memory 路径在 dev / 打包模式都自然工作,且符合"窗口零成本开关 / session 与 UI 解耦"哲学。详见 ADR-009。

### 9.4 IPC 协议

完整 IPC 协议(命令通道、事件通道、信封 schema、错误码、字节流编码)已在 [`docs/ipc-protocol.md`](./ipc-protocol.md) 单独定稿,版本与 `src/shared/protocol.ts` 中的常量同源。

主要分组:
* **命令通道**(Renderer → Main,`ipcRenderer.invoke` + `ipcMain.handle`):
  - app:get-protocol-version / get-snapshot / quit
  - window:create / close-self / close-all / focus
  - session:create / close / claim / release / focus-owner / send-input / resize / get-scrollback
  - bookmark:add / remove / rename / reorder / set-default-template / pick-folder
  - path:remove-from-recent
  - settings:get / update / reset / list-shells / get-auto-start / export / import
  - template:add / update / delete / set-default
  - system:show-in-explorer / open-data-dir / open-logs-dir / open-external
* **事件通道**(Main → Renderer,广播):
  - path:tree-updated / bookmarks:updated
  - session:created / owner-changed / state-changed / exited / destroyed / output
  - templates:updated
  - window:list-updated / focus-requested
  - settings:changed
  - app:state-changed

**信封**:每条命令带 `{ windowId, requestId, payload }`,每条事件带 `{ eventId, timestamp, payload }`(见 ipc-protocol.md 第 2 章)。
**安全**:`open-external` 仅放行 `http(s)` / `mailto`;`add-bookmark` 强制目录存在校验。

### 10.2 关键决策记录

#### ADR-001:为什么选 Electron 不选 Tauri
* node-pty 是 Microsoft 官方维护的 Node 模块,VS Code 在用
* xterm.js 是 JavaScript 库,在 Electron 的 Chromium 上验证最充分
* 多窗口 + 共享主进程的架构与 Electron 模型完全对齐
* 开发迭代速度(JS 热重载 vs Rust 编译)对单人项目重要
* 包大小、启动速度等 Tauri 优势在本场景下不重要(常驻应用,启动一次)
* 学习曲线:作者熟悉 JS / TS,不熟悉 Rust

#### ADR-002:为什么 IPC 用 Electron 原生而不是独立守护进程 + Named Pipe
* Electron Main Process 本身就是 Node.js 守护进程,无需再开一层
* Renderer 与 Main 之间用 Electron IPC 极快
* 简化部署:一个 .exe 即所有
* 缺点:守护进程不能在 UI 全部退出后独立运行 — 通过"托盘常驻"+"窗口可独立关闭"+"`window-all-closed` 不退出"解决

#### ADR-003:为什么用 OSC 1337 而非进程 cwd 查询作为主方案
* OSC 1337 是事件驱动,毫秒级响应,远胜轮询
* iTerm2、VS Code、Wave、Warp 都用此机制,生态成熟
* 进程 cwd 查询(NtQueryInformationProcess)作为兜底:每 5 秒一次,**仅在启动后首 5 秒未收到任何 OSC 1337 时才启动**(收到首条 OSC 后立即关闭轮询,大多数 session 永远不消耗轮询开销)
* **(v1.2 调整)**:OSC 1337 不再驱动 path 迁移,仅用于在标签上提示真实 cwd。详见 ADR-008

#### ADR-004:为什么 session 用完即弃,不维护跨会话上下文
* AI agent 上下文消耗 token,跨 session 累积消耗指数膨胀
* 真正需要持久化的知识应进 git / 文档 / CLAUDE.md
* agent 自己有 `claude -c` 等机制处理"continue 上次"
* 简化数据模型,降低实现复杂度

#### ADR-005:为什么一个 session 同时只有一个 owner window
* 多 owner = 多 client 字节流广播 + scrollback 同步 + 输入冲突协调,复杂度 5x
* "关 UI 不死"的核心价值不需要"多窗口同时显示同一 session"
* "多屏幕同时看"的场景由屏幕共享 / 投屏解决,不应是终端工具的责任
* 协议层留出多 owner 扩展点(owner 字段设计为 list),但 UI/服务端实现都按单 owner

#### ADR-006:为什么所有窗口完全平等,无主窗口概念
* 主窗口模型继承自 Win32 时代,与"应用 = 数据 + 多个观察窗"的现代心智不符
* 主窗口模型导致用户必须思考"我能不能关这个窗口"的决策疲劳
* 平等模型下用户的心智简化为"窗口是观察工具,关掉再开就好"
* 实现上反而更简单:无需为"主窗口"加特殊保护逻辑
* 用户可以零顾虑地新建/关闭窗口,匹配多屏多场景使用

#### ADR-007:为什么设置即改即生效,无保存按钮
* 用户改完不知道要点保存的体验糟糕
* 主题、字体等"调整型"设置必须实时反映才能边调边看
* 配置存盘代价低(JSON < 10KB),debounced 写入即可
* 仅"重置所有设置"等破坏性操作需要二次确认

#### ADR-008:为什么 path 与 cwd 解耦,且砍掉 5 分钟墓地+重启(v1.2 启动 CP-3 前确立)

**变更内容**:
1. session.pathId 在创建时确定,**生命周期内永远不变**;session 内 cd 不再让 session 在 UI 上迁移 path 分类
2. cwd 变化仅触发标签上的 ⚠️ 提示(currentCwd ≠ originalPath 时),不动 path 树
3. 砍掉 Tombstoned 的"5 分钟自动过期"和"用户重启" — 进程退出后 session 进入 Exited 灯显态,无时限保留,直到用户主动关闭

**变更理由**:

**关于 path 与 cwd 解耦**:
* 原方案让 session 跟随 cwd 在侧栏分类间"跳",用户心智上很费力 — "我刚在 ~/projects/auth 起的 claude,怎么突然出现在 ~/Downloads 了?"
* path 是"工作位置"的稳定索引,session 是该位置上的临时活动,cd 只是 session 内部的工具行为,不应该污染上层组织
* 新规则更稳:**"我在 path X 下打开的终端,就属于 path X,无论里面发生什么"**
* 实现复杂度大幅降低 — PathManager 不再需要"detach 旧 path / attach 新 path / 重新计算分类",cwd 跟踪只是 UI 装饰
* OSC 1337 仍然主推,因为我们仍需要知道真实 cwd(给 ⚠️ 提示、未来"在 Explorer 中显示"功能、可能的"在当前 cwd 新建终端"等)

**关于砍墓地**:
* 5 分钟墓地的初衷是让用户撤回误操作 / 看 claude 退出后的最终响应,但实测 5 分钟既太短(用户离开吃饭就过了)又太长(占用心智的标签灰显几分钟很烦)
* "重启"功能在 path 与 cwd 解耦后语义模糊 — 用什么 cwd 重启?用 originalCwd 用户体验和"新建一个"几乎一样
* 砍掉后:进程退出后 session 进入 Exited 状态,scrollback 完整保留,**无时限**等待用户;用户想看就看,看完右键关闭;想再跑一次就在同一 path 新建一个
* 这其实是更接近"传统终端 + 标签持久"的体验:tab 不会"自己消失"
* 数据模型更简单:删 `tombstonedAt` / `sessionTombstoneMinutes`;state 枚举是 `'active' | 'idle' | 'exited'` 三态终态可观察

#### ADR-009:为什么设置导入改 in-memory replace,移除 followSystemTheme(v1.3,CP-4 勘误)

**变更内容**:

1. **移除 `appearance.followSystemTheme`** 字段 + UI 开关 + main 端 `nativeTheme.on('updated')` wiring。
2. **设置导入路径重写** — 由"`fs.writeFile` 4 个 JSON + `app.relaunch + app.quit`"改为"各 Manager 的 `replaceAll()` + emit 事件 → 所有窗口实时刷新",**不再重启应用**。
3. **CSP 通过 `webRequest.onHeadersReceived` 在 main 进程注入**(dev 含 `unsafe-eval` 给 React Refresh,生产严格);不再用 index.html 的 meta CSP(避免 Vite HMR 与 meta CSP 冲突)。dev 模式同时 `process.env.ELECTRON_DISABLE_SECURITY_WARNINGS = 'true'`(unsafe-eval 是有意为之,Electron 仍会在控制台 warn,这是它的硬编码检查;打包后该模块不加载,无影响)。
4. **`window.queryLocalFonts()` 启用** — main 端 `setPermissionRequestHandler` 自动放行 `local-fonts` + `clipboard-read` 权限;字体下拉用真实安装列表而非写死白名单。
5. **状态机微调** — `SessionManager.createSession` 末尾立即 `scheduleIdleCheck()`,防止首波 PTY 数据是纯 OSC(passthrough 空)时永远卡在 `active`。

**变更理由**:

* **followSystemTheme 不可靠**:Windows 上 `nativeTheme.shouldUseDarkColors` 在多用户 / 远程会话 / 系统未配置颜色模式的机器上经常返回错误值,导致主题"自己跳",反而是 bug 来源。7 套主题让用户主动选,是更确定的体验。
* **`app.relaunch` 在 dev 失效**:electron-vite 的 dev daemon 拉起 main 进程后,`app.relaunch` 触发的二次启动与 Vite dev server 之间会失联,renderer URL 不可用,用户看到"导入后无法正常渲染"。In-memory replace 在 dev 与打包都自然工作。
* **In-memory replace 也更符合产品哲学**:导入是数据层面的事,运行中的 PTY session 不应被关 — 这与"窗口零成本开关 / session 与 UI 解耦"完全一致。
* **CSP via headers**:meta CSP 在 Vite dev 模式下与 HMR 注入的脚本互冲(`script-src 'self'` 会拒绝 Vite 内联);headers 路径让我们对 dev / prod 分别精准下发,避免双重 CSP 源冲突。
* **queryLocalFonts**:之前用 Canvas measureText probe 写死白名单是双输 — 既漏掉用户自己装的字体,又把未装的列出来。Local Font Access API 是 Chromium 原生方案,Electron 通过 permission handler 一键放行即可。

**实现要点**:
* 各 Manager 暴露 `replaceAll(input)` 方法(`SettingsManager` / `PathManager` / `TemplatesManager`),走的还是各自的"validate / merge / persist / emit"路径,与运行时 update 同源。
* `applyArchiveInMemory` 在 `ipc.ts` 顺序调用三个 `replaceAll` 后 `await Promise.all(flush)` 等磁盘 debounce 落盘。
* IPC 事件 `evt:settings:changed` / `evt:templates:updated` / `evt:path:tree-updated` / `evt:bookmarks:updated` 自然广播到所有窗口,reducer 合并即生效。

#### ADR-010:为什么自绘标题栏 + macOS / Windows 双布局(v1.4,Milestone 1)

**变更内容**(M1-A):
1. `BrowserWindow` 改为 `frame: false`,完全自绘标题栏(`WindowChrome.tsx`)。
2. 新增 `settings.appearance.windowStyle: 'windows' | 'macos'`,默认 `'windows'`。
3. Windows 布局:控制按钮在右侧(最小化 / 最大化-还原 / 关闭),lucide 图标,方形热区。
4. macOS 布局:三色 traffic light 在左侧(红/黄/绿圆形),hover 显示内部符号 ×/−/⤢;标题居中。
5. 新增 IPC:`cmd:window:minimize` / `cmd:window:toggle-maximize` / `cmd:window:get-max-state`;事件 `evt:window:max-state-changed`。
6. `Menu.setApplicationMenu(null)` — 彻底禁默认菜单条(File/Edit/View),与 spec 7.1"不做应用内快捷键"一致。

**变更理由**:
* 原 OS 默认 frame 在 7 套主题下视觉断裂(尤其深色主题:Rose Pine base #191724 vs 系统白色标题栏)。daily driver 每天看 8 小时,这种割裂是反复的视觉摩擦。
* 用户对"标题栏风格"的偏好是真实存在的二分:习惯 Win 风格的人讨厌 macOS 按钮位置,反之亦然。强制一种是错配。
* macOS 风格不仅是按钮位置,整体 chrome 更紧凑(32px vs 36px),适合"低视觉噪声"的开发者偏好;Windows 风格按钮更大,符合 Fitts's Law 的 Win11 设计语言。
* 自绘后免去了 OS 主题切换时的"标题栏抖一下"问题。
* `Menu.setApplicationMenu(null)` 是必须的同步动作,否则 Alt 还能弹出隐藏的菜单条,与自绘标题栏叠加视觉混乱。

**风险/边界**:
* Windows 7/10 没有 Window Controls Overlay(`titleBarOverlay`),但我们走纯自绘按钮路径,所有版本都一致。
* 全屏 / 最大化时窗口外圆角通过 `[data-maximized="true"]` 取消(代码已挂)。
* macOS 真机 V1 不测试(`mac:` 配置只为社区贡献者占位),`windowStyle: 'macos'` 是 Windows 上的"视觉风格选项"而非"模拟 macOS 行为"。

#### ADR-011:Milestone 1 内部上线的其他工程决策(v1.4)

打包成一个 ADR 是因为这些决策都来自同一时间点的"准备日常使用"评估,各自单写过细。

* **窗口位置记忆走 settings 而非独立文件**(M1-G):`windowDefaults` 是单一组而非 per-windowId,因为 windowId 是 UUID 不复用,而且"窗口零成本开关"哲学下,记每个窗口的位置毫无意义。"最后关闭的窗口"作为下次默认就够了。多显示器越界检测在 `resolveInitialBounds`,外接屏拔了自动回退主屏居中。
* **持久化日志自建,不引入 electron-log**(M1-D):AGENTS.md 1.2 边界 2。日均日志量 < 5 MB,日切 + 7 天保留,Node 内建 `createWriteStream` 完全够。一旦真要做远程上报再考虑。
* **崩溃兜底是 `keep alive` 而非"重启自愈"**(M1-D):未捕获异常多半是边角 bug,损坏的状态在下次操作时各 Manager 会重新校验。直接退出会丢所有 PTY,远比"带轻微 inconsistent state 继续跑"代价大。
* **托盘动态图标只两态**(M1-H):active / default;spec 6.5.1 提及的"waiting-input"是 V1.1 的状态识别扩展,本期不做。
* **导出归档敏感凭据弹三选一,不内置 keyring**(M1-F):正经凭据存储是 V1.x+ 的事;现在让用户每次导出明确决定"含/不含 env",已经比"默认明文导出"安全很多。
* **启动期 grace 1.5s**(M1-I):比 RESIZE_QUIET_MS 长(500ms),因为 PowerShell banner + profile 加载在慢机上能跑 1+ 秒;太短会闪,太长会让"刚开就跑命令"的 session 状态点延迟变 active(但 idle timer 仍正常启,所以最坏情况是首 1.5s 状态点稳定显示 active 而非闪烁)。
* **Ctrl+滚轮调字号通过 settings.update**(M1-I):全局生效,所有窗口同步。不做"per-session 字号"(违反"session 廉价 / settings 全局"哲学)。
* **env 字段 blur 遮罩而非真 password input**(M1-I):textarea 不支持 type=password。CSS `filter: blur(4px)` 实现视觉遮罩,hover/focus 自动解除,既保护偷瞄又不影响日常编辑。
* **第三方 prompt(starship/OMP)兼容性**(P1-5)**推后到上线后验证**:暂时不在 M1 内修。我们的 `pwsh.ps1` hook 先加载 `$PROFILE` 再 wrap prompt,理论上后注入的 OMP 会覆盖我们 → cwd 跟踪失效。日常验证发现失效再加 PSReadLine hook 路径(避免在装了 OMP 的开发者真机上才发现,从而进入 M2 勘误)。
* **`build/icon.ico` 离线生成**(M1-E):AGENTS.md 1.2 边界 2 不装 electron-icon-builder。SVG 源 + README 指引(在线工具/ImageMagick/PNG fallback),实际 .ico 由作者一次性生成放入,这是"零依赖 + 一次性手工"的合理折衷。

#### ADR-012:为什么 alpha 末改名 EasyTerm → Marina(v1.5)

**变更**:产品 alpha 阶段(CP-1 ~ Milestone 1)的开发代号 **EasyTerm** 在 v1.5 改为正式产品名 **Marina**。

**触点**:
- `package.json` `name` / `description` / `homepage`
- `electron-builder.yml` `appId` / `productName` / `nsis.shortcutName`
- 窗口标题、托盘 tooltip、托盘菜单文案、退出对话框、设置页"关于"、字符串中所有 brand 出现
- shell hook 内部标识符(`__easyterm_emit_cwd` → `__marina_emit_cwd`)与日志前缀
- 用户数据目录 `%APPDATA%\Marina\`(Electron 由 `productName` 自动派生)
- 归档格式 `marina-archive`(读侧仍接受 `easyterm-archive` 旧值)

**变更理由**:
- "EasyTerm" 是描述性的功能名(easy + terminal),没有产品感,不易在 GitHub / 工具市场上让人记住。
- "Marina" 是隐喻名:**码头 / 船坞**。多艘船(session)同泊;船关港不影响港湾本身(关窗不杀 session);船在港之间游弋(session 跨 path 切换);这套意象与产品哲学"path 是稳定的、session 是廉价的、UI 是临时的"高度同源。
- 名字软、容易记、4-6 字母、域名虽然主域被抢但 `.so` / `.sh` / `marinaterm.com` 可注 — 改名时机在 alpha 末 / 正式上线前,代价最低。

**风险与缓解**:
- **现存用户数据**:Electron 由 `productName` 派生 `app.getPath('userData')`,改名后**数据目录从 `%APPDATA%\EasyTerm\` 切到 `%APPDATA%\Marina\`**,老数据会"看起来丢了"。**作者本人需在改名后启动新版前手动 copy 一次**(`Copy-Item -Recurse $env:APPDATA\EasyTerm\* $env:APPDATA\Marina\`)。M1-G 的 `windowDefaults` 也走这一步迁移。
- **归档格式**:读侧同时接受新旧两个 `format` 字段,所以旧 EasyTerm 导出的备份仍能在新 Marina 里导入。导出统一用新名。
- **历史文档**:`docs/checkpoints/`、cp2/cp3/cp4 勘误、各 milestone 工作记录 等历史档保留 EasyTerm 字样作为时间点快照(在它们的头部加一行简短迁移说明)。
- **shell hook 标识符**:`__easyterm_emit_cwd` 等是注入到用户 shell 进程的函数名;每个 session spawn 时重新注入,改名后立即生效,不影响"已在跑"的 session(它们的 hook 函数在用户 shell 里已加载,session 结束就消失)。

#### ADR-013:为什么 Linux 上"最后窗口关闭"允许阻塞 modal(v1.6,beta 反馈回合)

**背景**:Marina 第 2 章原则 4"窗口与应用解耦"在 Windows 上由系统托盘承载(关窗 → 沉托盘 → app 继续跑),在 macOS 上由 Dock 承载(关窗 → 留在 Dock,Electron 默认 `window-all-closed` darwin 分支不退出 = 原生 HIG)。两者共有"app 不死于窗口"的持久化位置。

**Linux 的根本问题**:GNOME 自 3.x 起官方移除 system tray,只能用社区 AppIndicator 扩展(用户侧负担、Wayland 下脆弱);若退而求其次走 GNOME Background Apps API,则仅 GNOME 44+ 支持,KDE/XFCE 没有。无论选哪个,都是"为支持 Marina 把代价转嫁给 Linux 用户"—— 违反 Linux"软件融入桌面环境"的传统。

**决策**:Linux 上 Marina 不做托盘 / 不做 Background Apps,**作为普通桌面 app 实现**(`lifecycleModel: 'no-persistence'`)—— 关闭最后一个窗口 = 应用退出,session 全销毁。配套**阻塞 modal 二次确认**作为唯一 UX 妥协:当最后一个窗口关闭且仍有 `state !== 'exited'` 的 session 时,弹 `确定关闭?N 个会话将终止 [取消/关闭]`;全灰则静默退出。

**为什么允许 modal,不是哲学倒退**:
- "窗口零成本开关"哲学的前提是"app 在别处常驻、关窗确实零代价"。Linux 无托盘 = 前提消失 = 最后窗口关闭不再零代价。
- modal 是**信息披露 + 确认门禁**,只在"会损失工作"时触发。非最后窗口关闭 / 全 exited session 时仍零成本,精神上的"廉价开关"对非最后窗口仍然成立。
- 三大主流 Linux 终端(gnome-terminal / konsole / xterm 系)都在有子进程时弹相似的确认 —— 这是 Linux 用户在终端语境下的既有期待。

**Windows / macOS 是否对称引入**:Windows 托盘菜单"完全退出"和 macOS `Cmd+Q` / App Menu Quit 已有(或应有)同语义二次确认。三平台共享同一 modal 组件(`<LastSessionConfirm />`),仅触发位置不同:
- Windows:托盘菜单"完全退出"按钮
- macOS:Cmd+Q / `before-quit` 事件
- Linux:最后一个窗口的 close 事件 + 上述非空条件

**为什么不走 daemon 架构彻底消灭这个 UX 妥协**:
- daemon(独立后台进程持有 PTY,Electron UI 仅是观察窗)能让 Linux 上 session 自然跨窗口存活,无需 modal。
- 但 daemon 要求主进程拆分、IPC 层重写、electron-builder 双产物、升级/崩溃恢复流程重设计 —— 工作量 2-3 倍 v1 现有架构。
- 收益评估:Windows / macOS 现有托盘 / Dock 模型已覆盖 daemon 的关窗存活收益,daemon 在两个平台上只剩"崩溃存活 / 升级存活"边缘改善(BETA-002 半解);Linux 上才是真正的架构升级。
- 结论:三平台架构对称比"为 Linux 单独 daemon"更重要,后者会演变成两套心智 / 两套代码。**v0.x / v1 不走 daemon**,留作 v2 评估窗口(详见第 15 章"演进路线 · daemon 候选")。

**对设计哲学的影响**:第 2 章原则 4 增加"Linux 平台例外"脚注;AGENTS.md 附录 C"关闭弹确认"硬规则增加同一例外脚注。两处都引回 ADR-013 作为论据来源。

**对实现的影响**:
- PlatformAdapter 新增 `lifecycleModel: 'tray-resident' | 'dock-resident' | 'no-persistence'` 字段(详见 12.2)。
- 主进程关窗事件处理按 `lifecycleModel` 分支:`tray-resident` 沉托盘、`dock-resident` 留 Dock(Electron 默认)、`no-persistence` 在最后窗口 + 非空 session 时弹 modal。
- modal 组件跨平台单一实现,触发条件三平台各异。

### 11.2 内存数据(守护进程)

为避免重复维护两套定义,主进程内的"对外可见"数据结构与 `src/shared/types.ts` 同源,以下为简化展示;`ManagedSession` 是 Main 内部 PTY/scrollback 句柄的容器,**不通过 IPC 暴露**,真实定义见 `src/main/session-manager.ts`。

```typescript
interface AppState {
  windows: Map<string, ManagedWindow>;     // windowId → window info(WindowManager)
  sessions: Map<string, ManagedSession>;   // sessionId → managed session(SessionManager)
  pathTree: {
    bookmarks: PathNode[];                 // 用户主动管理,持久化到 bookmarks.json
    temporary: PathNode[];                 // 自动:有 session 在跑且非收藏
    recent: PathNode[];                    // 自动:历史开过、当前不在临时也不在收藏(<= 30)
  };
  nextWindowNumber: number;                // 下一个窗口编号(单调递增,运行时分配,不持久化)
}

// ── IPC 暴露的部分 (shared/types.ts) ──────────────────────

interface WindowInfo {
  id: string;                              // UUID,持久且不复用
  number: number;                          // 显示给用户的编号 Window 1/2/...,关闭后不复用
  electronWindowId: number;                // BrowserWindow.webContents.id
}

interface SessionInfo {
  id: string;
  pathId: string;                          // session 创建时确定,**永不变**(ADR-008)
  templateId: string;
  originalCwd: string;                     // session 创建时的 cwd,永不变(ADR-008)
  currentCwd: string;                      // 实时 cwd(随 OSC 1337 更新);仅 UI ⚠️ 提示
  cols: number;
  rows: number;
  pid: number;                             // PTY 子进程 PID;-1 = 未 spawn 或已退出
  displayName: string;                     // 默认 = 模板名 / shell 名,后续可由用户改
  ownerWindowId: string | null;
  state: 'active' | 'idle' | 'exited';     // v1.2 起去 tombstoned(ADR-008)
  exitCode?: number;                       // exited 时
  exitedAt?: number;                       // exited 时,Unix ms
  createdAt: number;                       // Unix ms
}

interface PathNode {
  id: string;
  path: string;
  displayName?: string;
  category: 'bookmarked' | 'temporary' | 'recent';
  sessionIds: string[];                    // 创建顺序,push 追加,任何 owner 切换不会 reorder
  defaultTemplateId?: string;              // 仅收藏:双击新建终端的默认模板
}

// ── 主进程内部细节(不通过 IPC,见 session-manager.ts) ──

interface ManagedSession {
  info: SessionInfo;
  pty: IPty | null;                        // exited 后置 null
  outputSeq: number;                       // evt:session:output 单调序号
  scrollback: Buffer;                      // ring buffer,2 MB 上限,尾部裁切
  scrollbackLastSeq: number;               // scrollback 末尾对应的 outputSeq;给新 owner 拉历史 + 去重 evt
  parser: Osc1337Parser;                   // 每 session 一份,持有未完结 stash
  idleTimer: NodeJS.Timeout | null;        // active → idle
  cwdGraceTimer: NodeJS.Timeout | null;    // 启动 5s grace,到期未收 OSC 才进 polling
  cwdPollTimer: NodeJS.Timeout | null;
  oscReceived: boolean;                    // 一旦 true,所有 cwd 兜底永久关
  resizeQuietUntil: number;                // ConPTY 重绘 quiet 窗口截止 ts(CP-3 勘误 #3 v2)
  disposables: IDisposable[];              // PTY 监听句柄
}
```

**SessionInfo 与 ManagedSession 的边界**:`SessionInfo` 是经由 IPC 在 main / renderer 间传输的"外观",必须 JSON-safe;`ManagedSession` 是 main 进程内部对 PTY 句柄、ring buffer、计时器的容器,renderer 永远不接触。

### 11.3 数据持久化策略

* 写策略:任何状态变更触发 debounced 写入(500ms 节流)
* 写方法:写入临时文件 → fsync → rename(原子操作)
* 备份:每次成功写入后,保留前一份为 `.bak`
* 损坏恢复:启动时若主文件 JSON 解析失败,尝试 `.bak`;都失败则用默认值
* 版本迁移:settings.json 中的 `version` 字段控制迁移逻辑,未识别的版本拒绝加载

### 12.1 策略:Windows-first,but cross-platform-ready

Marina V1 只发布 Windows 版,作者只测试 Windows。但代码架构上为未来的跨平台扩展留好接口,让开源贡献者能在不修改核心代码的前提下添加 macOS / Linux 支持。

#### 抽象接口

```typescript
export interface PlatformAdapter {
  /**
   * 平台生命周期模型(v1.6,ADR-013)
   * - 'tray-resident'(Windows):关掉所有窗口后,app 仍跑在系统托盘,session 由托盘里的主进程持有。
   *   关窗事件:沉到托盘,不退出。完全退出走托盘菜单"完全退出"按钮(带二次确认 modal)。
   * - 'dock-resident'(macOS):关掉所有窗口后,app 仍在 Dock(Electron `window-all-closed`
   *   darwin 分支默认不退出 = macOS HIG)。完全退出走 Cmd+Q / App Menu Quit(带二次确认 modal)。
   * - 'no-persistence'(Linux):无托盘 / 无 Dock,关掉最后一个窗口 = 应用退出。
   *   当最后一个窗口关闭且仍有 state !== 'exited' 的 session 时,弹同一 modal 二次确认;
   *   全 exited 则静默退出。
   *
   * 三平台共享同一个 <LastSessionConfirm /> 组件,仅触发位置不同。
   */
  readonly lifecycleModel: 'tray-resident' | 'dock-resident' | 'no-persistence';

  /**
   * 检测系统中可用的 shell
   * Windows: pwsh.exe / powershell.exe / cmd.exe / Git Bash
   * macOS: zsh / bash / fish
   * Linux: 读 /etc/shells
   */
  detectShells(): Promise<ShellInfo[]>;

  /**
   * 给定一个 shell,返回启动它时需要的额外参数和环境变量,
   * 以便注入 OSC 1337 cwd hook
   */
  buildShellLaunchParams(
    shell: ShellInfo,
    hookFilePath: string
  ): { args: string[]; env: Record<string, string> };

  /**
   * 注册 / 注销文件管理器右键集成
   * Windows: 注册表 HKCU\Software\Classes\Directory\shell\... + MSIX sparse
   * macOS: Finder Sync Extension(Swift,需独立子项目)
   * Linux: **不写 Nautilus 扩展**(nautilus-python 依赖重 + GTK 版本兼容性差)。
   *   走 freedesktop 标准:安装 .desktop 文件 + Categories=TerminalEmulator;
   *   用户在 Marina 设置→系统集成点"设为默认终端"时,跑 gsettings
   *   org.gnome.desktop.default-applications.terminal exec + update-alternatives
   *   (或 RHEL/CentOS 的 alternatives)。所有 GTK 文件管理器(Nautilus / Nemo / Caja)
   *   的"在终端中打开"会自动用 Marina。详见 ADR-013 与 BETA-003 实施计划。
   */
  registerFileManagerIntegration(appExePath: string): Promise<void>;
  unregisterFileManagerIntegration(): Promise<void>;

  /**
   * 查询进程的当前工作目录(OSC 1337 hook 失败时的兜底)
   * Windows: NtQueryInformationProcess + 读 PEB
   * macOS: proc_pidinfo
   * Linux: readlink /proc/<pid>/cwd
   */
  getProcessCwd(pid: number): Promise<string | null>;

  /**
   * 设置开机启动
   */
  setAutoStart(enabled: boolean): Promise<void>;
  isAutoStartEnabled(): Promise<boolean>;
}
```

### 12.3 跨平台礼仪

为了让未来贡献者方便,V1 代码必须遵守:

1. **不硬编码平台特定 API** — 通过 PlatformAdapter
2. **不硬编码路径分隔符** — 用 `path.sep` / `path.join`
3. **不硬编码行结束符** — 用 `os.EOL`
4. **不硬编码用户目录** — 用 `app.getPath('userData')`
5. **不硬编码 socket 路径** — 通过 PlatformAdapter
6. **平台特定逻辑都加注释** — 写清"其他平台思路"
7. **`window-all-closed` 按 `lifecycleModel` 分支处理**(v1.6,ADR-013):
   - `tray-resident`(Windows):不退出,沉到托盘
   - `dock-resident`(macOS):不退出,Electron darwin 默认行为(留在 Dock)
   - `no-persistence`(Linux):若仍有 `state !== 'exited'` 的 session,弹 `<LastSessionConfirm />` modal;用户确认后 `app.quit()`;全 exited 则直接 `app.quit()`

为各 shell 准备的 hook 文件全部在 `src/shell-hooks/` 提交:

* `pwsh.ps1` — PowerShell(V1 实现并测试)
* `cmd.bat` — cmd.exe(V1 实现并测试)
* `bash.sh` — bash(V1 已写好但不主动启用,留给 Mac/Linux 贡献者)

每个 hook 文件做两件事:
1. 加载用户原本的配置(rc 文件 / profile)
2. 注入 OSC 1337 prompt hook

## 13. 非目标(明确不做的事)

### 13.1 V1 不做(为 V2+ 留)
* 多 UI 同步同一 session(只做单 owner 模型)
* WSL session 支持
* 标签页拖拽改顺序 / 拖出新窗口
* 自动更新(用户手动下载新版)
* 应用打开时恢复上次窗口布局
* 状态识别中的"等待输入"和"错误"(只做活跃/空闲/已退出)
* 系统通知
* 已退出 session 重启(若需要"再跑一次",在同一 path 新建一个 session,见 ADR-008)

### 13.2 永远不做(违反产品哲学)
* 应用内功能键(任何不在 §7.2.2 终端键位清单内的快捷键)— 违反"鼠标优先";例:Ctrl+T 新 tab / Ctrl+W 关 tab / Ctrl+Tab 切 session / Alt+1..9 跳 tab / 任何菜单加速键
* 全局快捷键 — 同上
* 命令输入框输入路径 — 违反"路径只能点击"
* "主窗口"概念或任何窗口的特殊地位 — 违反"窗口平等"
* 关闭单个窗口的确认对话框 — 违反"窗口平等",窗口应是零成本开关
* Project / Workspace / Repo 等容器层级 — 违反"路径中心"
* 跨 session 上下文累积 / kanban / workstream — 违反"session 用完即弃"
* 内置 AI 助手或 chat 界面 — 让 agent 自己做 agent
* 远程 SSH session 管理 — 不是产品定位
* 内置文件编辑器 — 不是产品定位
* 设置的"保存"按钮 — 违反"即改即生效"

### 13.3 不会自己做但欢迎社区贡献
* macOS 实现
* Linux 实现
* WSL 集成
* 更多 shell hook(fish, nushell, xonsh, ...)
* 主题方案扩展(目前 7 套已足够)
* i18n 国际化(V1 中英双语已包含)

### Phase 1:个人内部工具(0 → 自用可用)
**时间:4-6 周(业余时间)**

目标:作者本人和公司同事日常可用,不在乎打磨。

* Week 1-2:技术骨架

* Week 3-4:核心功能

  - Path 状态机自动管理

  - 终端右键菜单

### Phase 4:V2.0 跨平台(部分由社区贡献)
* macOS 支持(贡献者实现 `macos.ts` + Finder Sync Extension,`lifecycleModel='dock-resident'`)
* **Linux 支持**(v1.6 beta 反馈回合启动,作者主导实现 `linux.ts` + GNOME 默认终端机制,`lifecycleModel='no-persistence'`,见 ADR-013 + BETA-003)
* WSL session 一等公民

### 演进路线 · daemon 候选(评估窗口:v2.0+)

**触发条件**:任一成立时考虑立项
- BETA-002 类崩溃导致用户实际损失工作的频率 > 月 1 次
- 升级 Marina 时丢 session 成为社区反馈高频项
- Linux 用户对"最后窗口 modal"持续不满,要求 session 跨窗口存活

**架构变更概要**(详细评估留给真正立项时):
- 拆 Electron 主进程为 **daemon(后台常驻,持有所有 PTY)** + **UI(Electron 窗口,通过 named pipe / 本地 socket 连 daemon)** 两份产物
- Windows:daemon 跑成 Windows Service(或退而求其次 detached process,登录态活)
- macOS:daemon 跑成 launchd LaunchAgent
- Linux:daemon 跑成 systemd --user service —— 顺带消灭 ADR-013 的"最后窗口 modal"
- IPC 协议升级支持 UI 端断连重连、session 接管

**为什么 v1 不做**:
- 工作量 2-3 倍于现有主进程架构,需重写 IPC、持久化、单实例、升级流程
- Windows / macOS 现有 tray / Dock 模型已覆盖 daemon 大半收益(关窗存活),边缘改善(崩溃存活 / 升级存活)不足以驱动整体架构重做
- 三平台同步演进比"仅 Linux daemon"更重要 —— 后者会变成两套架构维护两套心智

**为什么不彻底否决**:
- daemon 是 Linux 优等公民方案(对齐 tmux/screen 心智),长期价值在
- 真正解决 BETA-002 系列崩溃丢 session
- 留作 v2 候选,而不是"永远不做"

## 附录 A:术语表

| 术语 | 定义 |
|------|------|
| Path | 用户的"工作位置",一个文件系统目录 |
| Session | 一个 PTY 进程及其状态,包括 shell + 可选的子命令(如 claude code) |
| Window | Marina 的一个 UI 窗口实例,临时的观察工具,可任意开关,所有窗口完全平等 |
| 应用 / 守护进程 | Marina 的本体,持有所有 PTY 和数据;Electron Main Process;生命周期独立于任何窗口 |
| 纯托盘模式 | 应用在运行但 0 窗口的状态 |
| Owner Window | 当前持有某 session 字节流推送权的窗口,任一时刻最多一个 |
| 无主 Session | 当前无 owner window 的 session,在守护进程内继续运行 |
| OSC 1337 | iTerm2 定义的 ANSI 序列扩展,用于 shell 与终端的元数据通信 |
| Scrollback | 终端的历史输出缓冲区,新 owner 切换时需要重放 |
| Hook | 注入到 shell 启动流程中的脚本,负责发出 OSC 1337 序列 |
| Exited Session | Session 进程已退出但 UI 上仍可见的状态,无时限保留直到用户右键关闭(自 v1.2 起取代原"墓地期"概念,见 ADR-008) |
| originalCwd | Session 创建时的 cwd,等于该 session 所属 path 的 path 字段,生命周期内不变 |
| currentCwd | Session 内 OSC 1337 实时报告的工作目录,与 originalCwd 不一致时标签显示 ⚠️ |
| `replaceAll()` | 各 Manager(Settings / Path / Templates)v1.3 新增的方法,用于设置导入时整体替换内存数据 + 持久化 + emit 事件,**不重启应用**(见 ADR-009) |
| Local Font Access API | Chromium 提供的 `window.queryLocalFonts()`,枚举系统已安装字体;Electron 通过 `setPermissionRequestHandler` 自动放行 `local-fonts` 权限(v1.3 启用) |
| Resize Quiet 窗口 | `pty.resize` 后 500 ms 内 PTY 输出视为 ConPTY 重绘回声,**不触发 markActive**(只更新 scrollback / emit output);防 idle session 切窗口时 tab 闪绿(CP-3 勘误 #3 v2) |

## 附录 C:决策历史

本文档是经过约 25 轮深度讨论后的产物,关键设计决策的演化:

1. 起点:解决"多 agent 并发跑会忘记某个"的实际痛点
2. 拓展:意识到痛点不限于 agent,普通终端也有
3. 关键转向 1:从"多 UI 同步 session"砍成"单 owner + 跨窗口聚焦",架构复杂度降至 1/3
4. 关键转向 2:从 project 中心改为 path 中心,匹配真实工作流
5. 关键转向 3:确立"session 用完即弃"哲学,不维护跨会话上下文
6. 关键发现:Windows 平台的此类组合无对手,微软自己都没解决基础需求(close to tray)
7. 跨平台策略:从"做跨平台"调整为"架构 ready,实现交给社区"
8. 主题系统:复用作者已有的 Rose Pine 5 主题方案,带 #f0f fallback 机制
9. 不做应用内快捷键:保持鼠标优先工作流,避免向"键盘原教旨主义"妥协
10. **(v1.1 修订)** 修正"主窗口"概念错误,确立"所有窗口完全平等"模型;关闭窗口不影响应用,应用退出仅来自托盘菜单
11. **(v1.1 新增)** 设置页面完整规格;即改即生效无保存按钮
12. **(v1.2 修订,ADR-008)** path 与 cwd 解耦 — session.pathId 创建后永不变;cd 仅在标签上加 ⚠️ 提示,不再驱动 path 在分类间迁移
13. **(v1.2 修订,ADR-008)** 砍掉 5 分钟墓地自动过期 + 重启功能;session 进程退出后进入 Exited 灯显态,无时限保留,直到用户主动关闭
14. **(v1.3 修订,ADR-009)** CP-4 勘误回合一次性确立 6 项与实际工程的对齐:主题 5→7 套(加 Ubuntu / Windows Terminal);移除 followSystemTheme(`nativeTheme.shouldUseDarkColors` 在 Windows 不可靠);字体下拉走 `queryLocalFonts`;UI 系统图标改 lucide-react;设置导入改 in-memory replace 不重启应用(原 `app.relaunch` 在 dev 模式失效);CSP via webRequest header(取代 meta CSP)。
15. **(v1.4 修订,ADR-010/011)** Milestone 1 内部上线大优化 — `frame:false` + 自绘标题栏 + macOS/Windows 双布局选项;14 项 daily driver 必需的工程改进(右键菜单完整化 / 拖文件夹到终端区 / 主进程 uncaughtException 兜底 / 持久化日志 / 应用图标 / 导出归档隐私 / 窗口位置记忆 / 托盘菜单完整化 + 动态图标 / 启动期 grace / Ctrl+滚轮字号 / env 遮罩 / Toast 通知 / 不可达路径处理 / 单实例聚焦)。
16. **(v1.5 修订,ADR-012)** 品牌改名 — alpha 开发代号 **EasyTerm** 改为正式产品名 **Marina**。"路径中心 + session 不灭 + 多窗口平等"的港湾隐喻 vs 描述性功能名,产品感更强。Windows 数据目录从 `%APPDATA%\EasyTerm\` 切到 `%APPDATA%\Marina\`,归档格式读侧向后兼容。
17. **(v1.6 修订,ADR-013)** beta 反馈回合 · Linux 跨平台方向定稿 — 三平台生命周期模型显式分裂:Windows `tray-resident` / macOS `dock-resident` / Linux `no-persistence`,共享同一个最后会话二次确认 modal。Linux 上原则 4"窗口与应用解耦"让位于"防止用户意外丢会话",ADR-013 论证这不是哲学倒退而是无托盘环境下的必要让步。Linux 文件管理器集成走 freedesktop 标准而非 Nautilus 扩展,与 BETA-003 实施计划重写同步。daemon 架构(独立后台进程持有 PTY,Electron UI 仅是观察窗)作为 v2 候选保留为演进路线伏笔,v1 不做。
