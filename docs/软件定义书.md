## 1. 产品定位

EasyTerm 是一个 **Windows 平台的桌面终端管理器**,服务于"在多个不同路径下并发运行多个长时间任务(包括 AI coding agent 和普通命令行任务)"的开发者。

它**不是**一个传统终端模拟器的迭代,而是对"终端会话应该如何被管理"这个问题的重新回答:

* 传统终端把"窗口生命周期"和"shell 进程生命周期"绑定 —— 关窗即死。
* 传统终端以"profile / tab"为组织维度 —— 用户需要靠记忆和手动 cd 来导航。
* 传统终端假设用户是命令行专家 —— 重度依赖快捷键和配置文件。

EasyTerm 反其道而行:

* **会话独立于 UI 存活** —— 关掉所有窗口,会话仍在守护进程内运行,直到用户主动结束。
* **路径作为一等公民** —— 所有终端按"工作位置"组织,不再按 profile 或 project。
* **GUI 优先** —— 用户从不需要手敲路径,从不需要记忆快捷键,所有操作都可以用鼠标完成。

### 一句话定位

> "你的终端会话不应该因为关掉窗口就死掉。"

## 2. 设计哲学

EasyTerm 的所有设计决策都源自四条核心原则:

### 原则 1:Path 是稳定的,Session 是廉价的,UI 是临时的

| 层级 | 性质 | 用户对它的态度 |
|------|------|--------------|
| Path(路径) | 长期、稳定、用户主动管理 | 收藏、命名、组织 |
| Session(会话) | 短期、用完即弃、自动产生与销毁 | 启动、使用、关闭 |
| UI 窗口 | 完全临时、可随时开关、彼此平等 | 打开看一眼,不用了关掉 |

**这条原则推导出**:
* 三栏侧栏分类(收藏/临时/最近)以 path 为单位
* session 不维护跨会话的累积上下文
* 关闭窗口完全不影响 session
* 守护进程持有所有 session 和数据,窗口只是"观察窗"

### 原则 2:不让用户输入路径,只让用户点击路径

`cd` 命令是一个 1971 年的设计,它打错就报错或静默进入错误位置,代价高昂。EasyTerm 把"切换工作位置"重构为"点击一个已存在的路径条目",彻底绕过文本输入路径这个易错动作。

**这条原则推导出**:
* 所有路径必须通过文件夹选择器或 Explorer 拖入,不能手敲
* 路径在 UI 上永远可见,用户不需要 `pwd` 校对
* 切换路径 = 点击侧栏 = 启动新会话或聚焦已有会话,而不是 `cd`

### 原则 3:用户决策最少化

工具应该自动处理一切可以自动处理的事,只在用户必须做选择时才打扰用户。

**这条原则推导出**:
* 路径在三个分类之间的流转完全自动(状态机驱动)
* session 标签在路径间的归属由 cwd 自动决定(OSC 1337 hook)
* 终端切换时的 resize 自动发生
* 状态指示自动检测,不需要用户配置
* 所有窗口平等,没有"主次"概念,用户不需要思考"我该关哪个"

### 原则 4:窗口与应用解耦

应用 ≠ 窗口。窗口是临时的观察工具,应用是承载所有 session 的守护进程 + 托盘。

**这条原则推导出**:
* 关闭任何一个窗口,完全不影响其他窗口或任何 session
* 关闭所有窗口,应用进入"纯托盘模式",继续在后台运行
* 应用退出由用户主动从托盘触发,不和窗口生命周期绑定
* 所有窗口平等,不存在"主窗口"概念

## 3. 目标用户

### 主要用户

* **Windows 平台的开发者**,日常工作中需要并发运行多个长时间命令行任务
* **同时运行多个 AI coding agent**(Claude Code、Codex、OpenCode 等)的开发者
* **不愿意学习 tmux / zsh / dotfiles 的用户** —— 他们需要工具开箱即用,而不是花一周配置环境
* **跨多个不相关项目工作的用户** —— 路径不一定是 git 仓库,可能是脚本目录、日志路径、网络盘等

### 典型场景

* RTL/EDA 工程师在多个 run 目录之间跳转,跑综合、仿真、回归测试
* 全栈开发者在前端项目 + 后端项目 + 脚本目录之间切换
* 同时让 5 个 agent 在 5 个不同项目里干活,不希望忘记任何一个的状态
* 跑长时间编译/部署/AI 任务,不希望因为关错窗口前功尽弃

### 非目标用户

* 重度 tmux / zsh 用户(他们已有满意工作流,EasyTerm 不强迫他们改变)
* 团队协作场景中需要"项目级 session 共享"的用户(请用 Nimbalyst 等)
* 远程服务器管理为主的用户(请用 Windows Terminal + SSH 或专用 SSH 客户端)

## 4. 核心心智模型

### 三个一等公民,一种临时实体

EasyTerm 的世界由四种实体构成:

```
应用(守护进程 + 托盘)—— 单例,长期运行,用户从托盘控制其生死
  │
  ├── Path(路径)—— 永久,用户管理,数据持久化
  │     │
  │     └── Session 1(会话)—— 临时,运行时产生,完成即销毁
  │     └── Session 2
  │
  └── Window(窗口)—— 临时观察工具,任意开关,彼此平等
```

**关键关系**:
* 应用持有 Path 数据和所有 Session
* Session 归属于 Path(由 cwd 决定)
* Window 是观察 Path 和 Session 的临时窗口,**不持有数据**
* 关闭 Window 不影响 Path,不影响 Session,不影响应用

### Path 的三种状态(由状态机自动管理)

```
[最近 Recent]
  ↑↓ (用户主动加入/移除收藏)
[收藏 Bookmarked]
  ↑↓ (有终端在跑/全部关闭)
[临时 Temporary]
```

**精确定义**:

| 分类 | 定义 | 是否自动 |
|------|------|---------|
| **收藏** | 用户主动加入的路径,永久保留 | 用户主动 |
| **临时** | 当前有终端在跑、且不在收藏区的路径 | 完全自动 |
| **最近** | 历史上开过终端、当前不在收藏也不在临时的路径 | 完全自动 |

**流转规则**:
* 在某路径新建终端 → 路径进入"临时"(若不在收藏)
* "临时"路径的最后一个终端关闭 → 路径离开"临时",进入"最近"
* "最近"或"临时"路径被用户加入收藏 → 进入"收藏",离开原分类
* "收藏"路径被用户移除收藏 → 回到"最近"(若有历史)或消失

**容量**:
* 收藏:无限
* 临时:无限(由当前活跃 session 数量决定,自然有限)
* 最近:最多 30 个,按最近活动时间排序,超出自动淘汰最旧的

### Session 的生命周期

```
[创建] → [活跃] ↔ [空闲] / [等待输入] → [已退出] → [销毁]
                                              ↑
                                       用户主动右键关闭 / 应用退出
                                       (无时限自动过期)
```

* **创建**:用户从某路径新建终端,守护进程启动 PTY 进程
* **活跃**:PTY 最近有字节输出
* **空闲**:PTY 一段时间无输出,但进程仍在
* **等待输入**:通过 OSC 1337 命令完成事件检测,处于 prompt 状态(V1.1)
* **已退出**:进程已结束(正常或异常),session 灯显灰色 ⚫,scrollback 仍可见;**不可重启**;无时限自动过期
* **销毁**:用户主动右键"关闭"、或应用完全退出时(批量销毁)

**核心承诺**:从 [创建] 到 [销毁] 的整个过程**与任何窗口的开关无关**。即使关闭所有窗口,守护进程依然管理这些 session。

**已退出 session 的语义**:像传统终端"crashed but tab still open"的体验。用户看到 ⚫ 灯 + 完整退出前的 scrollback;选项卡有右键"关闭"。**没有重启功能**,如果用户想"再跑一次",在同一 path 新建一个 session 即可。详见 ADR-008。

### Window 与 Session 的关系

```
N 个 Window(窗口,N ≥ 0)<──── 任意多对一/一对多 ────> M 个 Session(会话)
                                                            在守护进程内
                                                            持续运行
```

**核心约束 1:任一 Session 同一时刻只有 0 或 1 个 Owner Window**

* Owner Window 持有该 session 的输入/输出渲染权
* 非 Owner 窗口在自己的 UI 中**只能看到 session 存在**(灰显标签),不能直接显示其内容
* 用户在非 Owner 窗口点击灰显标签 → 系统聚焦真正的 Owner 窗口
* Session 可以无 Owner(所有窗口都关了,或最后一个窗口主动释放)

**核心约束 2:N 可以是 0**

应用启动后,可以保持 0 个窗口,只有托盘和守护进程运行。任何时候用户从托盘可以新开窗口。

**优雅之处**:这两个约束彻底消除了"多窗口同步同一 session"的所有复杂度,同时让窗口的开关变成纯粹的"观察方式选择",不绑定任何业务逻辑。

### 应用生命周期

```
[启动] → [运行(0~N 窗口)] → [完全退出]
              ↑↓
         窗口数量随时变化
         不影响应用本身
```

**应用何时退出**:
1. 用户从托盘菜单选"完全退出 EasyTerm"
2. 系统关机/重启
3. 应用崩溃

**关闭窗口绝不等于关闭应用。**

## 5. 功能规格

### 5.1 必做(MVP V1)

#### 5.1.1 路径管理
* 添加收藏路径(通过文件夹选择器或拖拽 Explorer 文件夹)
* 移除收藏路径
* 重命名收藏路径(自定义显示名)
* 收藏路径排序(拖拽改变顺序)
* 临时路径自动出现/消失
* 最近路径自动记录,容量上限 30
* 用户右键最近路径可"加入收藏"或"从最近移除"

#### 5.1.2 Session 管理
* 在任一路径新建终端(默认 shell 或选择启动模板)
* 关闭终端(发 SIGTERM,5 秒后强制 kill)
* 已退出 session 保留在 UI 上(灯显灰色),用户可右键"关闭"销毁;**不可重启**
* 列出所有 session,按路径分组
* 状态指示:活跃 / 空闲 / 已退出(MVP 第一版只这三种)
* **session 与 path 的归属**:session 创建时记录的 path **永久绑定**,生命周期内不变。session 内 cwd 变化(cd 等)不会让 session 在 UI 上换 path,而是在该 session 的标签上显示 ⚠️ 提示真实 cwd(详见 ADR-008)

#### 5.1.3 启动模板
* 内置:Shell(系统默认)、Claude Code、Codex、OpenCode
* 用户自定义模板(命令、参数、环境变量、退出后行为)
* 每个收藏路径可设置"默认模板"(双击直接启动,跳过模板选择)

#### 5.1.4 终端体验
* xterm.js 渲染,支持 ANSI 颜色、Unicode 11、URL 自动识别可点击
* 终端搜索(Ctrl+F)
* 复制(选中即可)、粘贴(右键)
* 字体可配置(默认 Cascadia Mono → JetBrains Mono → Consolas → 霞鹜文楷 Mono fallback)
* 字号可配置(默认 13)

#### 5.1.5 多窗口支持
* 任意时刻可开任意多个窗口(包括 0 个)
* 所有窗口完全平等,不存在"主窗口"概念
* 关闭任意窗口不影响应用、其他窗口或任何 session
* 关闭所有窗口 → 应用进入"纯托盘模式",可随时从托盘新开窗口
* 每个窗口独立维护:当前选中路径、当前选中 session、侧栏展开状态
* 全局共享:收藏数据、最近数据、所有 session 列表、临时分类、设置
* 窗口标签:Window 1 / Window 2 / ... 自动编号(运行时分配,不持久化)
* 灰显标签 → 点击聚焦 Owner 窗口
* Owner 切换:用户在非 Owner 窗口点击同路径下的灰显 session(可选,V1 优先级低)

#### 5.1.6 守护进程与持久化
* 守护进程 = Electron Main Process,生命周期独立于任何窗口
* 第一个 EasyTerm 实例启动时拉起守护进程(单例锁保护,通过 `app.requestSingleInstanceLock()`)
* 第二次启动 EasyTerm.exe → 转发到已运行的实例,新开一个窗口
* 任何窗口关闭都不退出应用
* 用户从托盘"完全退出"→ 守护进程退出
* 数据持久化:收藏、最近、模板、设置 → JSON 文件,原子写

#### 5.1.7 系统托盘
* 常驻系统托盘,应用运行期间永不消失(除非"完全退出")
* 图标动态:有 session 在跑 / 全空闲(V1.1 加"有 session 等待用户")
* 单击托盘图标:聚焦最近活动的窗口;若无窗口,新开一个
* 右键菜单详细规格见 [6.5 系统托盘](#65-系统托盘)

#### 5.1.8 CWD 实时跟踪
* 通过 OSC 1337 序列追踪 session 内 cwd 变化
* **cwd 仅用于 UI 提示,不驱动 path 迁移**(ADR-008):session.currentCwd 与 session.pathId 对应的目录不一致时,该 session 的标签上显示 ⚠️ 图标 + tooltip 真实 cwd
* shell hook 注入透明完成,不污染用户 dotfiles(PowerShell 包装 prompt 函数;cmd.exe 嵌入 OSC 序列到 PROMPT 环境变量)
* 兜底策略:启动后首 5 秒内若未收到任何 OSC 1337,启动定期进程 API 查询(每 5 秒,Windows 用 `NtQueryInformationProcess`)。一旦收到 OSC 1337,关闭轮询

#### 5.1.9 主题系统
* 内置 5 套主题:Rose Pine(默认)、Rose Pine Dawn、Rose Pine Moon、Cutie、Business
* CSS variables 驱动,主题切换即时生效
* xterm.js 主题与 UI 主题同步切换
* 所有 `var()` 调用必须带 `#f0f` fallback,防止未定义变量污染
* stylelint 静态检查,确保所有 var() 引用的变量在所有主题里都有定义

#### 5.1.10 设置页面
完整规格见 [6.6 设置页面](#66-设置页面)。

### 5.2 V1 之后版本计划

#### V1.1

## 7. 交互规格

### 7.1 不做的交互

EasyTerm 明确**不做**以下交互:

* **应用内快捷键**:除了操作系统标准的 Ctrl+C / Ctrl+V(终端复制粘贴)和 Ctrl+F(终端搜索),不提供任何其他快捷键
* **全局快捷键**:不注册任何全局快捷键
* **命令输入框**:UI 中无任何"输入路径"或"输入命令"的文本框,所有路径必须通过文件夹选择器或拖拽
* **vim/emacs 风格快捷键**:不做(用户期待标准操作系统行为)

理由:这些交互假设用户是"愿意学习快捷键的命令行专家",而 EasyTerm 的目标用户是"愿意用鼠标的普通开发者"。坚持鼠标优先工作流,产品才能保持自洽。

### 7.2 必做的交互

#### 7.2.1 鼠标操作
* 点击侧栏路径:选中路径,在右侧显示该路径标签页 + 第一个终端(或加号)
* 点击侧栏路径展开箭头:展开/收起该路径下的终端列表
* 双击侧栏路径:在该路径用默认模板新建终端
* 点击侧栏终端条目:切换到该 session(本窗口)或聚焦 Owner 窗口
* 拖拽 Explorer 文件夹到侧栏收藏区:加入收藏
* 拖拽 Explorer 文件夹到终端区:在该路径新建终端
* 右键各处:上下文菜单

#### 7.2.2 终端内交互
* 选中文本即复制到剪贴板(类 Linux 风格,可在设置里关闭)
* 右键终端区:复制(选中时)/ 粘贴 / 清屏 / 搜索
* 滚轮:滚动 scrollback
* Ctrl+滚轮:调整字号
* Ctrl+C:终端内有选中文本时复制,无选中时发送 SIGINT(标准终端行为)
* Ctrl+V:粘贴
* Ctrl+F:打开终端搜索

### 7.3 拖拽行为

| 拖拽源 | 拖拽目标 | 行为 |
|--------|---------|------|
| Explorer 文件夹 | 侧栏收藏区 | 加入收藏 |
| Explorer 文件夹 | 终端区 / 加号区 | 在该路径新建终端(默认模板) |
| 侧栏路径条目 | 收藏分类内 | 改变收藏顺序 |
| 标签页 | 标签栏内 | 改变顺序(V1.2) |
| 标签页 | 窗口外 | 拆分到新窗口(V1.2) |

### 7.4 关闭与退出的精确语义

这是 EasyTerm 与传统应用最大的不同之一,必须精确定义:

#### 7.4.1 关闭单个窗口
* 触发方式:点窗口右上角 × / Alt+F4 / 系统菜单关闭 / 托盘菜单"关闭所有窗口"中之一
* 行为:
  1. 该窗口立即消失
  2. 它持有的所有 session 的 `owner_window_id` 变为 `null`
  3. 那些 session 在守护进程中继续运行
  4. 其他窗口完全不受影响
  5. 应用继续运行
  6. **绝不弹任何确认对话框**

#### 7.4.2 关闭所有窗口
* 等价于逐个关闭每个窗口
* 关掉最后一个后,应用进入"纯托盘模式":
  - 没有任何窗口可见
  - 托盘图标继续显示
  - 守护进程继续管理 session
  - 用户从托盘可随时新开窗口
* **不弹任何确认**

#### 7.4.3 完全退出应用
* 触发方式:仅托盘菜单"完全退出 EasyTerm"
* 行为:
  1. 若有 session 在跑且"完全退出前确认"开启 → 弹确认对话框
  2. 用户确认 → 给所有 PTY 发 SIGTERM,等待 5 秒
  3. 强制 kill 仍未退出的 PTY
  4. 持久化所有数据
  5. 守护进程退出
  6. 托盘图标消失
  7. 应用进程结束

#### 7.4.4 系统级关机/重启
* Windows 发送系统关机消息时,守护进程响应:
  1. 持久化数据
  2. 给所有 PTY 发 SIGTERM
  3. 退出
* 不阻止系统关机

## 8. 状态机定义

### 8.1 应用生命周期状态机

```
              ┌──────────────────────┐
              │  未运行 NotRunning  │
              └──────────┬───────────┘
                         │ 用户启动 EasyTerm.exe
                         │ (单例锁:首次启动)
                         ↓
              ┌──────────────────────┐
              │  启动中 Starting    │
              │  - 守护进程初始化   │
              │  - 加载持久化数据   │
              │  - 创建托盘图标     │
              └──────────┬───────────┘
                         │ 根据"启动时行为"设置
              ┌──────────┴──────────┐
              ↓                     ↓
    ┌──────────────────┐   ┌──────────────────┐
    │ 运行中(有窗口)│   │ 运行中(纯托盘)│
    │ Running w/ Win   │   │ Running Tray Only│
    └────────┬─────────┘   └────────┬─────────┘
             │                       │
             │  ←── 关闭最后一个窗口  │
             │                       │
             │  ── 从托盘新建窗口 ──> │
             │                       │
             ↓                       ↓
             └──────────┬────────────┘
                        │ 用户从托盘"完全退出"
                        ↓
              ┌──────────────────────┐
              │  退出中 Exiting     │
              │  - SIGTERM 所有 PTY │
              │  - 持久化数据       │
              │  - 托盘图标消失     │
              └──────────┬───────────┘
                         ↓
              ┌──────────────────────┐
              │  未运行 NotRunning  │
              └──────────────────────┘
```

**注意**:
* "运行中"两个子状态可以反复切换,不影响应用本身
* 用户可以通过启动 EasyTerm.exe 第二次来"新开窗口"(单例锁会转发给已运行的实例)

```
            │  ┌────────┐                     │
            └─ │ 收藏   │ ←───────────────────┘
   (移除收藏) │Bookmark│  (用户主动加入收藏)
              │   ed   │
              └────────┘
```

**注意**:
* 收藏路径即使没有终端在跑也不会进入临时(收藏优先级高)
* 收藏路径有终端在跑时,仍只显示在收藏分类(不重复显示)
* 路径首次出现(从未开过终端、从未被收藏)只能通过"加入收藏"或"在此新建终端"两种方式

### 8.3 Session 状态机

```
                      ┌─────────────┐
                      │  Created    │ (PTY 启动)
                      └──────┬──────┘
                             │ (持续接收 PTY 字节流)
                             ↓
        ┌───────────┐  有输出 ┌──────────┐  无输出  ┌──────────┐
        │ Idle      │ ←────── │ Active   │ ──────→ │ Idle     │
        │ (空闲)    │         │ (活跃)   │ N秒     │          │
        └─────┬─────┘         └──────────┘         └─────┬────┘
              │  (PTY 进程退出)                           │
              ↓                                           ↓
              └────────────────┬─────────────────────────┘
                               ↓
                      ┌─────────────────┐
                      │  Exited         │ (灯显灰色 ⚫,无时限保留)
                      │  (已退出)       │ scrollback 完整保留
                      └─────────┬───────┘
                                │ (用户右键"关闭" / 应用完全退出)
                                ↓
                          ┌─────────┐
                          │Destroyed│
                          └─────────┘
```

**Active / Idle 阈值**:
* Active → Idle:无字节输出超过 2 秒(可在设置调整)
* Idle → Active:有任何字节输出

**Exited 的特性**:
* 进入即"挂起",**不再消耗任何资源**(PTY 句柄已释放,无定时器)
* `exitCode` 字段记录进程退出码;`scrollback` 仍持有
* **不能从 Exited 回到 Active**(无重启路径)。用户想"再跑一次" → 在同一 path 新建一个 session
* 销毁来源:用户主动右键"关闭" / 应用完全退出时批量销毁

### 8.4 Window 与 Session 的所有权状态

每个 Session 有一个 `owner_window_id` 字段,可能值:
* 某个 Window 的 ID — 该 Window 当前持有此 session
* `null` — 此 session 当前无 owner

**所有权转移规则**:
* Window A 持有 Session X,用户在 Window A 切换到 Session Y → A 释放 X(变 null),A 持有 Y
* Window A 持有 Session X,Window A 关闭 → X 的 owner 变 null
* 用户在 Window B 点击 X(此时 X 的 owner 是 A)→ 守护进程聚焦 A,所有权不变
* Session X 无 owner,用户在 Window B 点击它 → B 接管,X.owner = B
* 应用进入"纯托盘模式"(无窗口)→ 所有 session 的 owner 都是 null

**关键说明**:
* 守护进程 = Electron Main Process,不是独立的可执行文件
* 每个 UI Window = 一个 BrowserWindow + 一个 Renderer 进程
* Main 与 Renderer 之间用 Electron 标准 IPC(`ipcMain` / `ipcRenderer`)
* 多窗口共享同一个 Main 进程,自然共享所有数据
* Window 数量可以是 0:此时 Main 仍在运行,只有托盘可见

### 9.2 进程职责划分

#### 9.2.1 Main Process(守护进程)
* 创建和管理所有 PTY(node-pty)
* 维护 session 状态、scrollback buffer
* 维护 path 数据(收藏/临时/最近)
* 维护 window 列表与 owner 关系
* 维护设置数据 + 广播变更
* 持久化数据(JSON I/O)
* 系统托盘
* 系统 API 调用(注册表、cwd 查询、单实例锁)
* 单一可信数据源
* **生命周期独立于任何窗口**:监听 `app.on('window-all-closed')` 但不调用 `app.quit()`

#### 9.2.2 Renderer Process(每个窗口一个)
* 渲染 React UI(侧栏、标签页、终端区、设置 UI)
* 维护**本窗口独立的视图状态**:
  - 当前选中的路径(选中状态)
  - 当前选中的 session(显示哪一个)
  - 侧栏分类的展开/折叠状态
  - 路径展开状态
  - 当前是否在设置视图
* 处理用户交互,通过 IPC 把意图发给 Main
* 接收 Main 推送的 PTY 字节流,送给 xterm.js 渲染
* 接收 Main 推送的设置变更,实时应用

**关键原则**:Renderer 不存储任何全局状态。所有"全局可见"的数据都从 Main 拉取或订阅。

### 9.3 数据流示例

#### 用户在路径 A 新建一个 Claude Code 终端

```
1. Renderer (Window 1):用户点击 "+" 按钮,选 Claude Code 模板
2. Renderer → Main:IPC 调用 createSession({ pathId: 'path-a', templateId: 'claude-code' })
3. Main:
   a. 启动 node-pty,cwd = path A 的实际目录,command = "claude"
   b. 注入 OSC 1337 hook(根据所选 shell)
   c. 创建 Session 对象,owner_window_id = window1.id
   d. 把 Session 添加到 path A 的 sessions 列表
   e. 如果 path A 不在收藏,加入临时分类
4. Main → 所有 Renderer:广播 path 树更新事件
5. Main → Window 1:推送该 session 的字节流(只推给 owner)
6. Window 1 Renderer:收到字节流,送给对应的 xterm.js 实例
```

#### 用户在 Window 2 点击 Window 1 持有的 session

```
1. Window 2 Renderer:用户点击灰显 session
2. Renderer → Main:IPC focusSessionWindow({ sessionId })
3. Main:查 session.owner_window_id == window1.id
4. Main → Window 1:命令该窗口 focus()
5. Window 1 浮到前台,选中那个 session 的标签
```

#### Session 内 cwd 变化(自 v1.2 起,**不再驱动 path 迁移**)

```
1. PTY 内的 shell 执行用户的 `cd /new/path`
2. PTY 输出 OSC 1337 序列:\x1b]1337;CurrentDir=/new/path\x07
3. Main 解析字节流时识别到 OSC 1337,提取序列后透明转发剩余字节给 owner
4. Main 更新 session.currentCwd(session.pathId 不变)
5. Main → 所有 Renderer:广播 sessionStateChanged({ id, currentCwd })
6. Renderer 比较 session.currentCwd 与 session 所属 path 的 path 字段:
   - 一致 → 标签正常显示
   - 不一致 → 标签旁显示 ⚠️ 图标,tooltip 显示真实 cwd
7. **path 状态机不被触动**(session.pathId 永久绑定,见 ADR-008)
```

#### 用户关闭 Window 2(不影响其他)

```
1. 用户点击 Window 2 的 ×
2. Electron 触发 'close' 事件
3. Main 监听到 → 找到 Window 2 持有的所有 sessions
4. 对每个 session:owner_window_id 改为 null
5. Window 2 真正销毁(Renderer 进程退出)
6. Main → 所有剩余 Renderer:广播 window-list-updated + sessions-state-updated
7. Main 检查窗口数量:
   - 还有窗口 → 继续运行
   - 没有窗口 → 进入"纯托盘模式",不退出应用
```

## 10. 技术决策

### 10.1 技术栈

| 层 | 技术 | 备选 | 理由 |
|----|------|------|------|
| 应用框架 | **Electron** | Tauri | Node 生态对终端最友好;node-pty + xterm.js 主路径;开发迭代快;Windows 系统集成成熟 |
| 守护进程语言 | **TypeScript on Node.js** | Rust | 与 UI 同语言;node-pty 原生集成;无跨语言 IPC 开销 |
| UI 前端 | **React + TypeScript** | Vue, Svelte | 生态最大;Electron 主流选择 |
| UI 状态管理 | **Zustand** 或简单 useReducer | Redux | 轻量,适合本应用复杂度 |
| 终端渲染 | **xterm.js** | (无替代) | 业界标准 |
| PTY | **node-pty** | (无替代) | Microsoft 官方,VS Code 在用 |
| IPC(Renderer-Main) | **Electron 原生 IPC** | — | 内置,无需额外配置 |
| 配置存储 | **JSON 文件** | SQLite | 数据量小,易调试 |
| 样式 | **CSS(原生 + variables)** | Tailwind, CSS-in-JS | 主题系统已用 CSS variables;不引入额外构建复杂度 |
| 打包 | **electron-builder** | electron-forge | 配置简单,跨平台模板齐全 |
| 代码检查 | ESLint + Prettier + stylelint | — | 标准配置 |

### 10.2 关键决策记录

#### ADR-001:为什么选 Electron 不选 Tauri
* node-pty 是 Microsoft 官方维护的 Node 模块,VS Code 在用
* xterm.js 是 JavaScript 库,在 Electron 的 Chromium 上验证最充分
* 多窗口 + 共享主进程的架构与 Electron 模型完全对齐
* 开发迭代速度(JS 热重载 vs Rust 编译)对单人项目重要
* 包大小、启动速度等 Tauri 优势在本场景下不重要(常驻应用,启动一次)
* 学习曲线:作者熟悉 JS / TS,不熟悉 Rust

#### ADR-002:为什么 IPC 用 Electron 原生而不是独立守护进程 + Named Pipe
* Electron Main Process 本身就是 Node.js 守护进程,无需再开一层
* Renderer 与 Main 之间用 Electron IPC 极快
* 简化部署:一个 .exe 即所有
* 缺点:守护进程不能在 UI 全部退出后独立运行 — 通过"托盘常驻"+"窗口可独立关闭"+"`window-all-closed` 不退出"解决

#### ADR-003:为什么用 OSC 1337 而非进程 cwd 查询作为主方案
* OSC 1337 是事件驱动,毫秒级响应,远胜轮询
* iTerm2、VS Code、Wave、Warp 都用此机制,生态成熟
* 进程 cwd 查询(NtQueryInformationProcess)作为兜底:每 5 秒一次,**仅在启动后首 5 秒未收到任何 OSC 1337 时才启动**(收到首条 OSC 后立即关闭轮询,大多数 session 永远不消耗轮询开销)
* **(v1.2 调整)**:OSC 1337 不再驱动 path 迁移,仅用于在标签上提示真实 cwd。详见 ADR-008

#### ADR-004:为什么 session 用完即弃,不维护跨会话上下文
* AI agent 上下文消耗 token,跨 session 累积消耗指数膨胀
* 真正需要持久化的知识应进 git / 文档 / CLAUDE.md
* agent 自己有 `claude -c` 等机制处理"continue 上次"
* 简化数据模型,降低实现复杂度

#### ADR-005:为什么一个 session 同时只有一个 owner window
* 多 owner = 多 client 字节流广播 + scrollback 同步 + 输入冲突协调,复杂度 5x
* "关 UI 不死"的核心价值不需要"多窗口同时显示同一 session"
* "多屏幕同时看"的场景由屏幕共享 / 投屏解决,不应是终端工具的责任
* 协议层留出多 owner 扩展点(owner 字段设计为 list),但 UI/服务端实现都按单 owner

#### ADR-006:为什么所有窗口完全平等,无主窗口概念
* 主窗口模型继承自 Win32 时代,与"应用 = 数据 + 多个观察窗"的现代心智不符
* 主窗口模型导致用户必须思考"我能不能关这个窗口"的决策疲劳
* 平等模型下用户的心智简化为"窗口是观察工具,关掉再开就好"
* 实现上反而更简单:无需为"主窗口"加特殊保护逻辑
* 用户可以零顾虑地新建/关闭窗口,匹配多屏多场景使用

#### ADR-007:为什么设置即改即生效,无保存按钮
* 用户改完不知道要点保存的体验糟糕
* 主题、字体等"调整型"设置必须实时反映才能边调边看
* 配置存盘代价低(JSON < 10KB),debounced 写入即可
* 仅"重置所有设置"等破坏性操作需要二次确认

#### ADR-008:为什么 path 与 cwd 解耦,且砍掉 5 分钟墓地+重启(v1.2 启动 CP-3 前确立)

**变更内容**:
1. session.pathId 在创建时确定,**生命周期内永远不变**;session 内 cd 不再让 session 在 UI 上迁移 path 分类
2. cwd 变化仅触发标签上的 ⚠️ 提示(currentCwd ≠ originalPath 时),不动 path 树
3. 砍掉 Tombstoned 的"5 分钟自动过期"和"用户重启" — 进程退出后 session 进入 Exited 灯显态,无时限保留,直到用户主动关闭

**变更理由**:

**关于 path 与 cwd 解耦**:
* 原方案让 session 跟随 cwd 在侧栏分类间"跳",用户心智上很费力 — "我刚在 ~/projects/auth 起的 claude,怎么突然出现在 ~/Downloads 了?"
* path 是"工作位置"的稳定索引,session 是该位置上的临时活动,cd 只是 session 内部的工具行为,不应该污染上层组织
* 新规则更稳:**"我在 path X 下打开的终端,就属于 path X,无论里面发生什么"**
* 实现复杂度大幅降低 — PathManager 不再需要"detach 旧 path / attach 新 path / 重新计算分类",cwd 跟踪只是 UI 装饰
* OSC 1337 仍然主推,因为我们仍需要知道真实 cwd(给 ⚠️ 提示、未来"在 Explorer 中显示"功能、可能的"在当前 cwd 新建终端"等)

**关于砍墓地**:
* 5 分钟墓地的初衷是让用户撤回误操作 / 看 claude 退出后的最终响应,但实测 5 分钟既太短(用户离开吃饭就过了)又太长(占用心智的标签灰显几分钟很烦)
* "重启"功能在 path 与 cwd 解耦后语义模糊 — 用什么 cwd 重启?用 originalCwd 用户体验和"新建一个"几乎一样
* 砍掉后:进程退出后 session 进入 Exited 状态,scrollback 完整保留,**无时限**等待用户;用户想看就看,看完右键关闭;想再跑一次就在同一 path 新建一个
* 这其实是更接近"传统终端 + 标签持久"的体验:tab 不会"自己消失"
* 数据模型更简单:删 `tombstonedAt` / `sessionTombstoneMinutes`;state 枚举是 `'active' | 'idle' | 'exited'` 三态终态可观察

```typescript
  id: string;                           // 内置模板 id 固定(如 'shell','claude-code'),自定义为 UUID
  name: string;
  icon: string;                         // emoji 或图标标识
  isBuiltin: boolean;
  command: string;                      // 实际执行的命令(如 "claude")
  args: string[];
  env: Record<string, string>;
  shellFirst: boolean;                  // 是否先启动 shell 再跑 command
  postExitAction: 'close_session' | 'keep_shell' | 'hold';
}
```

```typescript
  // 视图状态(展开/折叠)在 Renderer 侧维护,Main 不存
```

### 11.3 数据持久化策略

* 写策略:任何状态变更触发 debounced 写入(500ms 节流)
* 写方法:写入临时文件 → fsync → rename(原子操作)
* 备份:每次成功写入后,保留前一份为 `.bak`
* 损坏恢复:启动时若主文件 JSON 解析失败,尝试 `.bak`;都失败则用默认值
* 版本迁移:settings.json 中的 `version` 字段控制迁移逻辑,未识别的版本拒绝加载

## 12. 跨平台策略

### 12.1 策略:Windows-first,but cross-platform-ready

EasyTerm V1 只发布 Windows 版,作者只测试 Windows。但代码架构上为未来的跨平台扩展留好接口,让开源贡献者能在不修改核心代码的前提下添加 macOS / Linux 支持。

### 12.2 平台抽象层

```
src/main/platform/
  ├── index.ts          # 接口定义 + 平台分发
  ├── windows.ts        # Windows 实现(V1 完整实现)
  ├── macos.ts          # macOS 占位(throw "Not implemented")
  └── linux.ts          # Linux 占位(同上)
```

#### 抽象接口

```typescript
export interface PlatformAdapter {
  /**
   * 检测系统中可用的 shell
   * Windows: pwsh.exe / powershell.exe / cmd.exe / Git Bash
   * macOS: zsh / bash / fish
   * Linux: 读 /etc/shells
   */
  detectShells(): Promise<ShellInfo[]>;

  /**
   * 给定一个 shell,返回启动它时需要的额外参数和环境变量,
   * 以便注入 OSC 1337 cwd hook
   */
  buildShellLaunchParams(
    shell: ShellInfo,
    hookFilePath: string
  ): { args: string[]; env: Record<string, string> };

  /**
   * 注册 / 注销文件管理器右键集成
   * Windows: 注册表 HKCU\Software\Classes\Directory\shell\...
   * macOS: Finder Sync Extension(Swift,需独立子项目)
   * Linux: 各 DE 各做(Nautilus / Dolphin / Thunar / ...)
   */
  registerFileManagerIntegration(appExePath: string): Promise<void>;
  unregisterFileManagerIntegration(): Promise<void>;

  /**
   * 查询进程的当前工作目录(OSC 1337 hook 失败时的兜底)
   * Windows: NtQueryInformationProcess + 读 PEB
   * macOS: proc_pidinfo
   * Linux: readlink /proc/<pid>/cwd
   */
  getProcessCwd(pid: number): Promise<string | null>;

  /**
   * 设置开机启动
   */
  setAutoStart(enabled: boolean): Promise<void>;
  isAutoStartEnabled(): Promise<boolean>;
}
```

#### 实现状态

```typescript

import { WindowsAdapter } from './windows';

export const platform: PlatformAdapter = (() => {
  switch (process.platform) {
    case 'win32':
      return new WindowsAdapter();
    case 'darwin':

    case 'linux':
      throw new Error('Linux support not implemented yet. Contributions welcome!');
    default:

  }
```

### 12.3 跨平台礼仪

为了让未来贡献者方便,V1 代码必须遵守:

1. **不硬编码平台特定 API** — 通过 PlatformAdapter
2. **不硬编码路径分隔符** — 用 `path.sep` / `path.join`
3. **不硬编码行结束符** — 用 `os.EOL`
4. **不硬编码用户目录** — 用 `app.getPath('userData')`
5. **不硬编码 socket 路径** — 通过 PlatformAdapter
6. **平台特定逻辑都加注释** — 写清"其他平台思路"
7. **`window-all-closed` 在所有平台都不退出**(我们的设计要求托盘常驻,即使 Win/Linux 也不走默认行为)

### 12.4 跨平台准备的额外文件

* `electron-builder.yml`:三平台 target 模板齐全,只 Windows 启用
* `.github/workflows/release-windows.yml`:Windows CI
* `.github/workflows/release-macos.yml.template`:macOS CI 模板,贡献者改名启用
* `.github/workflows/release-linux.yml.template`:Linux CI 模板
* `CONTRIBUTING.md`:说明 PlatformAdapter 设计哲学,引导新平台贡献

### 12.5 Shell Hook 文件

为各 shell 准备的 hook 文件全部在 `src/shell-hooks/` 提交:

* `pwsh.ps1` — PowerShell(V1 实现并测试)
* `cmd.bat` — cmd.exe(V1 实现并测试)
* `bash.sh` — bash(V1 已写好但不主动启用,留给 Mac/Linux 贡献者)
* `zsh.sh` — zsh(同上)
* `fish.fish` — fish(占位)

每个 hook 文件做两件事:
1. 加载用户原本的配置(rc 文件 / profile)
2. 注入 OSC 1337 prompt hook

## 13. 非目标(明确不做的事)

### 13.1 V1 不做(为 V2+ 留)
* 多 UI 同步同一 session(只做单 owner 模型)
* WSL session 支持
* 标签页拖拽改顺序 / 拖出新窗口
* Explorer 右键集成(V1 设置 UI 显示但置灰,V1.2 启用)
* 自动更新(用户手动下载新版)
* 应用打开时恢复上次窗口布局
* 状态识别中的"等待输入"和"错误"(只做活跃/空闲/已退出)
* 系统通知
* 已退出 session 重启(若需要"再跑一次",在同一 path 新建一个 session,见 ADR-008)

### 13.2 永远不做(违反产品哲学)
* 应用内快捷键(除标准 Ctrl+C/V/F)— 违反"鼠标优先"
* 全局快捷键 — 同上
* 命令输入框输入路径 — 违反"路径只能点击"
* "主窗口"概念或任何窗口的特殊地位 — 违反"窗口平等"
* 关闭单个窗口的确认对话框 — 违反"窗口平等",窗口应是零成本开关
* Project / Workspace / Repo 等容器层级 — 违反"路径中心"
* 跨 session 上下文累积 / kanban / workstream — 违反"session 用完即弃"
* 内置 AI 助手或 chat 界面 — 让 agent 自己做 agent
* 远程 SSH session 管理 — 不是产品定位
* 内置文件编辑器 — 不是产品定位
* 设置的"保存"按钮 — 违反"即改即生效"

### 13.3 不会自己做但欢迎社区贡献
* macOS 实现
* Linux 实现
* WSL 集成
* 更多 shell hook(fish, nushell, xonsh, ...)
* 主题方案扩展(目前 5 套已足够)
* i18n 国际化(V1 中英双语已包含)

## 附录 A:术语表

| 术语 | 定义 |
|------|------|
| Path | 用户的"工作位置",一个文件系统目录 |
| Session | 一个 PTY 进程及其状态,包括 shell + 可选的子命令(如 claude code) |
| Window | EasyTerm 的一个 UI 窗口实例,临时的观察工具,可任意开关,所有窗口完全平等 |
| 应用 / 守护进程 | EasyTerm 的本体,持有所有 PTY 和数据;Electron Main Process;生命周期独立于任何窗口 |
| 纯托盘模式 | 应用在运行但 0 窗口的状态 |
| Owner Window | 当前持有某 session 字节流推送权的窗口,任一时刻最多一个 |
| 无主 Session | 当前无 owner window 的 session,在守护进程内继续运行 |
| OSC 1337 | iTerm2 定义的 ANSI 序列扩展,用于 shell 与终端的元数据通信 |
| Scrollback | 终端的历史输出缓冲区,新 owner 切换时需要重放 |
| Hook | 注入到 shell 启动流程中的脚本,负责发出 OSC 1337 序列 |
| Exited Session | Session 进程已退出但 UI 上仍可见的状态,无时限保留直到用户右键关闭(自 v1.2 起取代原"墓地期"概念,见 ADR-008) |
| originalCwd | Session 创建时的 cwd,等于该 session 所属 path 的 path 字段,生命周期内不变 |
| currentCwd | Session 内 OSC 1337 实时报告的工作目录,与 originalCwd 不一致时标签显示 ⚠️ |

## 附录 B:参考产品

理解 EasyTerm 的设计选择需要了解这些产品在做什么:

| 产品 | 平台 | 我们学到了什么 / 做得不一样 |
|------|------|--------------------------|
| Windows Terminal | Win | 微软官方,但无 session 持久化、无路径管理、close-to-tray 4 年未实现 |
| Tabby | 跨 | Electron 终端模拟器实现的优秀范本,无 session 持久化,无路径管理 |
| Wave Terminal | Mac/Linux | server-client 架构,session 持久化做得好,但项目中心 |
| Warp | Mac/Win | Block 概念创新,但仍是 session/block 中心 |
| Agents UI | Mac | 最接近 EasyTerm 的"agent 终端"产品,但 macOS only |
| tmux / zellij | Unix | session 持久化的鼻祖,但 TUI 不是 GUI |
| Nimbalyst | 跨 | Workspace + Kanban,但偏团队协作,与 EasyTerm 哲学相反 |
| Claude Code Desktop | 跨 | Anthropic 官方,但绑定 Claude,不通用 |

## 附录 C:决策历史

本文档是经过约 25 轮深度讨论后的产物,关键设计决策的演化:

1. 起点:解决"多 agent 并发跑会忘记某个"的实际痛点
2. 拓展:意识到痛点不限于 agent,普通终端也有
3. 关键转向 1:从"多 UI 同步 session"砍成"单 owner + 跨窗口聚焦",架构复杂度降至 1/3
4. 关键转向 2:从 project 中心改为 path 中心,匹配真实工作流
5. 关键转向 3:确立"session 用完即弃"哲学,不维护跨会话上下文
6. 关键发现:Windows 平台的此类组合无对手,微软自己都没解决基础需求(close to tray)
7. 跨平台策略:从"做跨平台"调整为"架构 ready,实现交给社区"
8. 主题系统:复用作者已有的 Rose Pine 5 主题方案,带 #f0f fallback 机制
9. 不做应用内快捷键:保持鼠标优先工作流,避免向"键盘原教旨主义"妥协
10. **(v1.1 修订)** 修正"主窗口"概念错误,确立"所有窗口完全平等"模型;关闭窗口不影响应用,应用退出仅来自托盘菜单
11. **(v1.1 新增)** 设置页面完整规格;即改即生效无保存按钮
12. **(v1.2 修订,ADR-008)** path 与 cwd 解耦 — session.pathId 创建后永不变;cd 仅在标签上加 ⚠️ 提示,不再驱动 path 在分类间迁移
13. **(v1.2 修订,ADR-008)** 砍掉 5 分钟墓地自动过期 + 重启功能;session 进程退出后进入 Exited 灯显态,无时限保留,直到用户主动关闭

完整讨论记录由作者保留,有需要可向作者索取。
