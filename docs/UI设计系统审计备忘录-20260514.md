## 1. 设计令牌（Design Tokens）

设计令牌指"可主题化、可复用、可静态校验"的 CSS 变量集合。当前 Marina 的 14 个语义颜色变量是优秀的起点（`--base / --surface / --overlay / --highlight-{low,med,high} / --muted / --subtle / --text / --love / --gold / --rose / --pine / --foam / --iris`），但是**非颜色维度**几乎没有 token 化。

### TKN-1 [H] · `--font-mono` 变量未定义但被引用

**位置**：`src/renderer/styles/global.css:1358`
```css
.explorer-integration-meta .text-muted {
  color: var(--muted, #f0f);
  font-family: var(--font-mono, monospace);  /* ← --font-mono 从未在 :root 定义 */
  font-size: 10px;
}
```
**现象**：设置 → 系统集成卡片里的注册表路径 / Get-AppxPackage 输出走不到 `--terminal-font-family`（用户设置的 Cascadia Mono 等），永远落到 fallback `monospace`，Windows 上是 Courier New。
**根因**：变量笔误。CSS 中**统一定义**的是 `--terminal-font-family`（global.css:30），全局其它 11 处都正确写法。这一处 1358 行落单。
**修复方向**：改成 `var(--terminal-font-family, monospace)`。1 行。AGENTS.md 已经接了 stylelint，可以考虑加 `declaration-property-value-allowed-list` 之类规则把 `var(--font-mono` 列入禁词。

### TKN-2 [M] · 阴影 / 圆角 / 透明度 / transition 时长全部硬编码

**位置**：global.css 全文 / SettingsView.tsx 行内 style
**现象**：
- 阴影：`0 6px 18px rgb(0 0 0 / 28%)`（ctx-menu）/ `0 4px 16px rgb(0 0 0 / 35%)`（search-bar / toast）/ `0 12px 36px rgb(0 0 0 / 45%)`（Modal）/ `0 4px 24px rgba(0,0,0,0.3)`（bootstrap-placeholder）共 4 套独立硬值
- 圆角：`border-radius: 2px / 3px / 4px / 6px / 8px` 5 个不同值散落全文
- transition：`80ms / 100ms / 120ms / 160ms` 4 个时长散布
- 透明度：opacity 0.45 / 0.5 / 0.55 / 0.6 / 0.65 / 0.7 用于不同语义（disabled / owned-by-other / exited / muted-icon）
- backdrop transparent / 黑色 45% 透明 / love 8% 透明（explorer-unsupported 警告框）

**根因**：缺少非颜色 token 体系，每个组件作者自行决断。
**影响**：
- 设计师调一次阴影需要全文 grep；
- 浅色主题下没法整体调浅暗投（THM-4 的根因）；
- 减动效（A11Y-2）支持时没法批量缩放；
- 视觉 rhythm 散乱（圆角 2/3/4/6/8 都用过，不同组件并排时不和谐）。

**修复方向**：在 `:root` 加一组：
```css
:root {
  --radius-sm: 3px;
  --radius-md: 4px;
  --radius-lg: 6px;
  --shadow-low: 0 2px 6px rgb(0 0 0 / 14%);
  --shadow-med: 0 6px 18px rgb(0 0 0 / 28%);
  --shadow-high: 0 12px 36px rgb(0 0 0 / 36%);
  --duration-fast: 80ms;
  --duration-med: 160ms;
  --opacity-disabled: 0.45;
  --opacity-secondary: 0.65;
}
```
浅色主题在 `[data-theme='rose-pine-dawn'] / [data-theme='cutie']` 内覆盖 `--shadow-*` 为更柔和的值（参见 THM-4）。

### TKN-3 [M] · z-index 4 个数量级散布无 scale

**位置**：
- `terminal-search-bar` → 100（global.css:1771）
- `ctx-menu` → 1000（global.css:825）
- `toast-container` → 9999（global.css:1993）
- `app-modal-backdrop` → 10000（global.css:1852）

**现象**：当前层级关系刚好对（modal > toast > ctx-menu > search-bar），但**没有显式约定**。如果未来加 onboarding tour overlay、HUD、自动更新提示等浮层，新作者会再蒙一个值。
**根因**：没有 z-index token 表。
**修复方向**：
```css
:root {
  --z-popover: 100;       /* terminal-search-bar */
  --z-menu: 1000;         /* ctx-menu */
  --z-toast: 9999;        /* toast-container */
  --z-modal: 10000;       /* app-modal-backdrop */
}
```
所有 z-index 引用都走变量。新增浮层时 reviewer 强制要求引用现有 token 或扩 token 表。

### TKN-4 [L] · macOS traffic light 三色硬编码，绕开主题系统

**位置**：`src/renderer/styles/global.css:360-380`
```css
.titlebar-traffic-btn.close { background-color: #ff5f57; }
.titlebar-traffic-btn.min   { background-color: #febc2e; }
.titlebar-traffic-btn.max   { background-color: #28c840; }
.app-root[data-window-focused='false'] .app-titlebar-macos .titlebar-traffic-btn {
  background-color: #4d4d4d;
}
```
**现象**：macOS 风格的红 / 黄 / 绿三色与所有 7 套主题完全脱钩，未聚焦时统一 `#4d4d4d` 在 dawn / cutie 浅色 titlebar 上对比度不足。
**根因**：致敬 macOS 真实色值 — 但 macOS 自己也在 Big Sur 之后让这些圆点随系统主题（高对比度模式下变高饱和度）。
**修复方向**：让 close/min/max 各加一条带 `var(--love)` / `var(--gold)` / `var(--pine)` 的"对齐主题"风格变体，由 settings 里再加一开关（"macOS 圆点：保留原色 / 跟主题"）。不紧急；当前默认主题深色下视觉 OK。

### TKN-5 [L] · 占位 fallback `#f0f` 是 stylelint 兜底色，生产代码不应"看到"它

**位置**：global.css 全文 `var(--xxx, #f0f)`
**现象**：约 300 处 `var()` 调用统一兜底到品红 `#f0f`。这是有意设计 — 一旦主题变量缺失，UI 立即呈现刺眼粉色，开发期立即可见（软件定义书 5.1.9）。
**结论**：**不是 bug**，是优良实践。记录在此提醒未来 PR review 时不要把这些 fallback 改成"看起来更合理"的值（那会让缺失的变量隐身）。

## 2. 主题与对比度（Theme & Contrast）

7 套主题对应 7 个 `[data-theme="..."]` 块，每套覆盖 14 个语义变量。问题集中在**浅色主题的弱对比文字**与**主题色在中性 chrome 元件中的应用**。

### THM-1 [H] · Cutie / Rose Pine Dawn 主题下二级文字对比度不达标

**位置**：global.css:55-71（dawn）, 93-109（cutie）
**现象**：
- Cutie: `--muted: #b8809e` 在 `--surface: #ffeef5` 上 — 估算对比度约 **2.1:1**（WCAG AA 标准正文 4.5:1，大字 3:1）
- Cutie: `--subtle: #8a4d6f` 在 `--surface: #ffeef5` 上 — 估算约 **5.3:1**（达标）
- Dawn: `--muted: #9893a5` 在 `--surface: #fffaf3` 上 — 估算约 **3.0:1**（不达标）
- Dawn: `--subtle: #797593` 在 `--surface: #fffaf3` 上 — 估算约 **4.7:1**（勉强达标）

**实际可观察的位置**（`--muted` 被用作主要颜色的地方）：
- `.sidebar-category-count`（路径计数 12 / 3）
- `.sidebar-empty`（"空"占位文字）
- `.path-expand-arrow`（▶ 展开图标）
- `.tab-close`（× 关闭按钮）
- `.welcome-state`（欢迎页主文字 — 主要标题！）
- `.toast-icon` / `.toast-close`
- `.settings-info-text`（about 页 commit / version 信息）
- `.bootstrap-placeholder .subtitle`（启动失败副标题）

**根因**：浅色主题的语义变量是参考 Rose Pine 上游配色直接搬来的，未做 WCAG 对比度回归。
**修复方向**：
1. 把 dawn 的 `--muted` 从 `#9893a5` 降到约 `#6b6680`（提到 4.5:1）；
2. 把 cutie 的 `--muted` 从 `#b8809e` 降到约 `#7a4566`（提到 4.5:1）。
3. 用 [WebAIM contrast checker](https://webaim.org/resources/contrastchecker/) 或 stylelint 插件 `@adobe/leonardo` 跑一遍 7 套主题 × 14 变量 × 3 上下文（base / surface / overlay）的矩阵。

**注**：深色主题（rose-pine / moon / business / ubuntu / windows-terminal）的对比度 spot-check 都在 4.5:1 以上，问题集中在两个浅色主题。

### THM-2 [M] · `<select>` 展开后是 OS 原生窗口，浅色主题下黑底白字

**位置**：`src/renderer/styles/global.css:1486-1490`（注释已承认）
```css
/* select 的 option 在 Chromium 下展开后是 OS 原生窗口,无法用 CSS 主题完全覆盖
 * (Windows 一直是黑底白字,深浅色主题下都还可接受)。我们至少把 option 的
 * 背景与文字色显式给出来,避免极端浅色主题下不可读。 */
select.settings-input option {
  background-color: var(--surface, #f0f);
  color: var(--text, #f0f);
}
```
**现象**：在 cutie / dawn 浅色主题下点 `<select>`，弹出来的是 Windows 原生 dark/light 主题下的菜单（Edge 默认 dark mode 是黑底白字），与 app body 风格完全脱节，跳眼。
**根因**：Chromium `<select>` 弹出来的 `<option>` 列表走的是 OS native menu，CSS 只能"提示"颜色，OS 决定具体绘制。
**修复方向**：
- **短期**：在 `:root` 加 `color-scheme: dark light`，并在每个 `[data-theme]` 内单独写 `color-scheme: dark` 或 `light`。Chromium 11x 已实装 `color-scheme`：dark 时 native 菜单走 OS dark 主题（深底浅字），light 时走 light。能让原生菜单与应用主题色相至少同步，避免"浅色主题但弹深色菜单"。
- **长期**：自绘 `<select>` 替换。`@radix-ui/react-select` 或自写一个 200 行的 popover-list。但当前阶段（Marina 几乎所有 `<select>` 都在设置页，且选项少），自绘成本可能 > 收益。

### THM-3 [M] · `.tab-close:hover` 用 `--love` 背景 + `--base` 文字，浅色主题反差不足

**位置**：global.css:969-972
```css
.tab-close:hover {
  background-color: var(--love, #f0f);
  color: var(--base, #f0f);
}
```
**现象**：
- 深色主题 rose-pine：`--love: #eb6f92`（红）+ `--base: #191724`（近黑文字）→ 高对比、效果好。
- 浅色主题 dawn：`--love: #b4637a`（玫红）+ `--base: #faf4ed`（米白文字）→ 浅红底上浅米白 ×，几乎不可读。
- 浅色主题 cutie：`--love: #ff5d9e`（亮粉）+ `--base: #fff5f9`（粉白文字）→ 同样浅+浅。

**根因**：用 `--base` 当对比文字是深色主题假设；浅色主题下 `--base` 是白系，与 hover 的 `--love` 红系并不构成强对比。
**修复方向**：把 hover 状态的 `color` 改成"对比专用语义变量" `--on-love`（深色主题 = base 黑、浅色主题 = text 深紫红）。或者最简方案：浅色主题下 `--base` 不当 on-love 用，统一用 `#fff`：
```css
.tab-close:hover { color: #fff; }   /* 14 处类似的反色文字都改成 #fff */
```
但这绕开了主题系统。更系统的做法是引入 `--on-accent / --on-love / --on-pine / --on-gold` 4 个"在该色之上的文字色"变量，每个主题独立指定。

### THM-4 [M] · 浅色主题下阴影"墨水块"感

**位置**：
- `.ctx-menu` global.css:843-845 — `0 6px 18px rgb(0 0 0 / 28%)`
- `.app-modal-panel` global.css:1877-1879 — `0 12px 36px rgb(0 0 0 / 45%)`
- `.toast` global.css:2010 — `0 4px 16px rgb(0 0 0 / 35%)`
- `.terminal-search-bar` global.css:1770 — `0 4px 16px rgb(0 0 0 / 35%)`
- `.bootstrap-placeholder` global.css:2097 — `0 4px 24px rgba(0,0,0,0.3)`

**现象**：以上五处全部用纯黑半透明阴影。在深色主题（base / surface 都是暗色）下，黑色阴影正好"隐身"在背景里，只透出 6/4 像素轮廓，效果优雅。但在浅色主题（dawn / cutie，base / surface 是浅米色 / 浅粉色）下，黑色阴影在浅底上是清晰可见的灰团，对 Modal 这种带 45% 透明度的尤其明显，视觉上"砸"在 UI 上。
**根因**：阴影颜色硬编码为黑色，不参与主题切换。
**修复方向**：
1. 给 `.ctx-menu` 已有的做法（暗投 + `0 0 0 1px var(--overlay)` 主题色 halo）扩散到 Modal / Toast / search-bar。
2. 浅色主题下把 shadow opacity 从 28-45% 降到 14-22%。如果走 TKN-2 的阴影 token，只改 `--shadow-*` 变量即可。
3. **更进一步**：阴影颜色用 `color-mix` 或 `oklch` 算 surface 色的略暗版本作为阴影色，而不是纯黑。例：`box-shadow: 0 6px 18px color-mix(in srgb, var(--text) 22%, transparent);`。

### THM-5 [L] · 缺少 `prefers-color-scheme` 联动

**位置**：CSS 全文 `@media` 块**零**个。
**现象**：用户切 Windows 11 系统主题（深色 ↔ 浅色），Marina 不会跟随。
**根因**：主题完全由 settings 持久化的 `appearance.theme` 决定。
**修复方向**：
- 选择 A（保守）：在 `App.tsx` 里读 `window.matchMedia('(prefers-color-scheme: dark)').matches`，仅在用户**没显式选过主题**（settings 字段为 null / `'auto'`）时用 OS 偏好；用户一旦在设置页选了主题就锁定。

### A11Y-1 [H] · 几乎所有按钮没有 `:focus-visible` 样式 — 键盘浏览不可见焦点

**位置**：global.css 中 `:focus-visible` 选择器仅出现 2 次：
- `.ctx-menu-item:focus-visible:not(:disabled)`（line 884）
- `.app-modal-button:focus-visible`（line 1963）

**现象**：用键盘 Tab 在 Marina 里依次浏览：
- 标题栏三按钮 → 看不到焦点
- Sidebar "+" / 设置入口 → 看不到焦点
- Tab 栏每个 tab → 看不到焦点
- 主页 EmptyPathState 的 Shell / 模板按钮 → 看不到焦点
- 设置页 7 个分类导航 → 看不到焦点
- 设置页所有 settings-button / settings-input → 看不到焦点
- 主题卡片 → 看不到焦点

实际：Chromium 默认的 `outline: -webkit-focus-ring-color auto 1px` 在按钮 hover/active 时**被 button:focus 内置样式覆盖**，加上 .titlebar-btn / .settings-button 都重置了 `border: none`，焦点轮廓几乎不可见（深色主题下尤其黑）。

**根因**：全局未铺设 `:focus-visible` token。
**影响**：
- 键盘用户（包括只能用键盘的 power user）找不到当前焦点
- 屏幕阅读器用户依赖焦点视觉提示帮助沟通
- WCAG 2.1 SC 2.4.7（Focus Visible）不达标

**修复方向**：在 global.css 末尾加一个统一兜底块：
```css
/* —— 全局键盘焦点可见性 —— */
:focus-visible {
  outline: 2px solid var(--iris, #f0f);
  outline-offset: 2px;
  border-radius: inherit;
}
button:focus-visible,
.ctx-menu-item:focus-visible,
.settings-nav-item:focus-visible,
.tab:focus-visible,
.template-button:focus-visible,
.settings-theme-card:focus-visible {
  outline: 2px solid var(--iris, #f0f);
  outline-offset: 2px;
}
input:focus-visible,
select:focus-visible,
textarea:focus-visible {
  /* 已有 .settings-input:focus 样式，可保留 */
}
```
按主题色 `--iris` 作焦点环，所有 7 套主题自动 OK。Cutie 的 `--iris` 是 `#c77dff`（饱和紫）在浅粉底上对比足够；windows-terminal 主题的 `--iris` 是 `#3b78ff`（蓝）在黑底上也 OK。

### A11Y-2 [H] · 完全未支持 `prefers-reduced-motion`

**位置**：global.css 全文
**现象**：用户开启系统"减少动效"（Windows 设置 → 辅助功能 → 视觉效果 → 动画效果 关闭；或 macOS 辅助功能 → 显示器 → 减少动态效果），Marina 仍然播放：
- `.app-modal-backdrop` 100ms fade-in
- `.app-modal-panel` 120ms scale-in（包含 `transform: scale(0.96 → 1)`）
- `.toast` 160ms slide-in（`transform: translateX(40px → 0)`）
- `.path-expand-arrow.expanded` 100ms rotate transition
- 各处 hover transition `background-color 80-120ms ease`
- `.titlebar-traffic-btn:hover` filter brightness transition

**根因**：CSS 全文**零** `@media (prefers-reduced-motion)` 块。
**影响**：前庭功能障碍 / 偏头痛 / 注意力敏感 用户在 Marina 中会被频繁的微动效干扰。WCAG 2.1 SC 2.3.3（Animation from Interactions）不达标。
**修复方向**：在 global.css 末尾加：
```css
@media (prefers-reduced-motion: reduce) {

    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
  .app-modal-backdrop {
    backdrop-filter: none;  /* 减动效用户多数也敏感 GPU 合成的色彩涂抹 */
  }
}
```
3 行配置覆盖全文，立刻达标。

### A11Y-3 [M] · `.path-expand-arrow` 不可键盘聚焦，aria-label 错位

## 4. 浮层一致性（Overlay）

Marina 有 4 类浮层：ContextMenu、Toast、Modal、Terminal Search Bar。它们各自实现，没有共同的浮层基础（如统一 backdrop、portal、关闭策略）。

### OVR-1 [H] · Modal 没有真正的 Tab 焦点 trap

**位置**：`src/renderer/components/Modal.tsx:152-188`
**现象**：Modal mount 时聚焦 primary button（line 184-187），但 keydown handler 只处理 Escape 和 Enter。用户按 Tab 时焦点可以**跳出** panel 进入背后的 sidebar / tab / 终端。
**根因**：实现的是"打开时记录、关闭时归还"，但**打开期间不限制焦点**。注释里写了"焦点 trap"（line 11）但代码没实装。
**修复方向**：mount 后查询 panel 内所有可聚焦元素列表 `panel.querySelectorAll('button, input, ..., [tabindex]:not([tabindex="-1"])')`，keydown Tab/Shift+Tab 时手动循环到首/尾。约 30 行。或者用 `focus-trap-react`（4KB 加密 npm 依赖，但需走 AGENTS.md 1.2 边界 2 询问）。

## 9. 字体与排版（Typography）

### TYP-1 [M] · 默认 UI 字体首选 'LXGW WenKai'（手写中文字体），英文等宽性差

**位置**：`src/renderer/styles/global.css:28-32`
```css

  --ui-font-family:
    'LXGW WenKai', 'Segoe UI', 'PingFang SC', 'Microsoft YaHei', system-ui, sans-serif;
}
```
**现象**：
- LXGW WenKai 是基于霞鹜文楷的开源中文楷体，中文渲染优秀。
- 但**英文不等宽**，且 ASCII 字符的字体设计偏手写风（细笔触、变化笔画宽度）。
- Sidebar 显示文件夹路径（如 `C:\Users\liyue\Documents\project-name`）时英文部分笔画粗细变化，与 .session-name（中文为主）混排时不协调。
- 标题栏 `Marina — Window 1` 中英混排，"Window" 字间距与 "Marina" 不同。

**根因**：LXGW WenKai 设计目标是"日常阅读优秀中文楷体"，不是"产品 UI 中英混排"。
**修复方向**：
- 短期：默认 UI 字体改为 `'Segoe UI Variable', 'Segoe UI', 'PingFang SC', 'Microsoft YaHei', 'LXGW WenKai', system-ui, sans-serif` — 把 Segoe UI Variable（Win11 默认）放最前，LXGW 作为深度中文 fallback。

- 用户可在设置里自己改，已经有 FontPicker 组件 — 但默认值影响首次安装体验。

### TYP-2 [M] · 默认终端字体 fallback 链最后是 'LXGW WenKai Mono'（手写等宽中文）

**现象**：

**根因**：Cascadia 不含中文，LXGW WenKai Mono 是合理的中文 fallback，但风格差异大。
