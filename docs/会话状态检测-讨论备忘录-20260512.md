# 会话状态检测 —— 讨论备忘录

> 日期:2026-05-12
> 状态:**仅讨论,未落地**。本文档归档预研结论,供后续 milestone 评估时直接接入。

---

## 0. TL;DR

- 当前 session 状态(active/idle/exited)**完全由 PTY 输出流静默时长驱动**,会把"运行中无输出"的进程(Vite、等输入的 Claude)误判为 idle。
- 真正难点不是"进程在不在跑",而是"**前台进程是否在等用户输入**"。单一信号源永远分不清。
- 探索过的方案里:**主动 ping、监听 DOM 键鼠事件、stdin 写入观察**都不成立(原因见 §4)。
- 可行的分层路径(性价比从高到低):
  1. **OSC 133 shell integration** —— 解决 prompt/running 边界,Vite 不再误报 idle。
  2. **静默 + 敲键响应启发式** —— 单 session 可识别"等输入",但多 session 监控失效。
  3. **LLM 屏幕复核(opt-in)** —— 解决多 session 盲区;序列化文本送 Haiku 即可,不需要 VLM 截图。
- 本轮**不实施**任何方案,先记账。下一个相关 milestone 再评估。

## 1. 当前实现快照

**信号源**:仅 PTY passthrough 字节(OSC 序列被剥离后剩余的字节)。

**状态机**(`src/main/session-manager.ts:819-839`):
- 来字节 → `active`
- 静默超过 `advanced.activeIdleThresholdSeconds`(默认 2s)→ `idle`
- PTY 退出 → `exited`

**抑制窗口**:
- 启动 grace 1.5s(避开 shell banner 闪)
- Resize quiet 500ms(避开 ConPTY 重绘)

**UI 投影**:
- `Sidebar.tsx:46-50` —— 绿 / 黄 / 灰三色点
- `MainPane.tsx:549-550` —— tab CSS 类
- `TerminalView.tsx:730-732` —— 状态栏点

**核心问题**:语义是"**最近 N 秒有没有字节流过**",不是"**进程在不在跑**",也不是"**是不是在等你**"。三种语义被压成一个状态字。

## 2. 四类目标场景

| # | 场景 | 进程在跑 | 输出活跃 | 等用户 | 当前行为 | 期望行为 |
|---|------|---------|---------|--------|---------|---------|
| 1 | Vite dev server(稳定后) | ✓ | ✗ | ✗ | 误报 idle | 显示 running |
| 2 | Claude Code 流式输出 | ✓ | ✓ | 兼可 | active ✓ | running |
| 3 | Claude Code 停下等你 | ✓ | ✗ | ✓ | 误报 idle | **必须识别 waiting** |
| 4 | 脚本问 y/n(`apt`、`rm -i`) | ✓ | ✗ | ✓ | 误报 idle | 识别 waiting |

**优先级共识**:场景 3 是核心诉求,场景 1 可接受识别失败。场景 2 已经 work。场景 4 是顺路加分项。

## 3. 建议状态模型(4 态)

| 状态 | 含义 | UI |
|------|------|-----|
| `exited` | PTY 死 | 灰 |
| `prompt` | shell 在 prompt,可输入新命令 | 绿稳态 |
| `running` | 命令在跑(输出活跃 或 进程在算)| 黄脉冲 |
| `waiting` | 命令在跑且**确认**卡在等输入 | 橙闪 / 红 |

不引入"running-quiet"中间态 —— 用户认知负担太大。**宁可漏报 waiting,不要误报 waiting**:把 Vite 错标 waiting 比保守标 running 更让用户烦躁。

## 4. 探索过的方案及裁决

### 4.1 ❌ DOM 层监听键鼠(xterm.js 上的 keydown/mousedown)
反映的是"**用户**活跃"不是"**终端**活跃"。在 vim 里来回切 tab 就 idle,反例直接破。

### 4.2 ❌ 监听 stdin 写入
同上,只反映"用户在打字",无法回答"前台进程是否还能响应"。

### 4.3 ❌ 主动 ping PTY 看回包
xterm 控制序列(DSR `\x1b[5n`、CPR `\x1b[6n`、DA `\x1b[c`)是**应用→终端模拟器**方向 —— xterm.js 自己回的,**不是前台进程响应**。反向(terminal → 前台进程)没有标准探测协议。

### 4.4 ❌ 主程序自己往 PTY 偷写"无害字节"试探
**没有无害字节**。BEL 响铃、空格进 claude 输入框、Esc 切 vim 模式、Ctrl+C 杀进程。所有控制字符在某个常见应用里都有副作用。**探测一次污染一次。**

### 4.5 ⚠️ 被动观察"敲键 → 响应"组合(用户提议)
**逻辑成立**:A=应用响应键盘 ∧ B=最近无输出 ⇒ 等你。

"响应"的唯一可观测定义 = **(用户写入 t0)→(t0+K ms 内 PTY 有 onData)**:
- cooked mode → line discipline 回 echo 字节 ✓
- raw mode TUI → 应用自己重绘 ✓
- 不读 stdin(Vite)→ 无字节 ✓(正确不命中)

**两个固有限制**:
1. **首次盲区**:用户敲第一下之前,系统不知道应用在等。
2. **多 session 失效**:你在 A 工作,B 上没敲键事件 → B 永远停在 running。**而场景 3 的真实价值恰恰在多 session。**

**触发条件精确化**(若将来落地):
```
进入 waiting:
  state == running
  AND 输出静默 ≥ 3s
  AND 静默期内用户向 PTY 写入过字节
  AND 写入后 ≤ 300ms 出现 PTY 输出

退出 waiting:
  收到 OSC 133;D → prompt
  OR 输出再次涌动 ≥ T 秒 → running
```
**关键**:只在静默期内归因敲键。streaming 期间敲键 → 200ms 内必有字节但归因不可靠。

### 4.6 ✓ OSC 133 Shell Integration
业内标准答案,解决 prompt/running 边界:

| 序列 | 含义 |
|------|------|
| `ESC ] 133 ; A BEL` | prompt 开始 |
| `ESC ] 133 ; B BEL` | prompt 结束 —— 准备好接受命令 |
| `ESC ] 133 ; C BEL` | 命令开始执行 |
| `ESC ] 133 ; D ; <exitcode> BEL` | 命令结束 |

iTerm2 / WezTerm / Kitty / VS Code / Windows Terminal 都识别。

- 收 `;B` → `prompt`
- 收 `;C` → `running`
- 收 `;D` → 回 `prompt` + 拿 exitCode
- 命令运行中无论输出活不活跃,**保持 running**(场景 1 修复)

**收益**:Vite 不再误报 idle;prompt 真空闲可被精确识别;为命令分块/历史/cwd 精度都打基础。

**成本**:
- `osc1337-parser.ts` 扩展识别 OSC 133(顺路,基础设施已在)
- shell integration 脚本注入用户 rc(bash/zsh/fish/PowerShell 各一份)—— 这才是真工作量
- Windows 上 git-bash / pwsh / cmd 各有路径,要逐个处理

**仍不解决**:场景 3(Claude TUI 等输入)。OSC 133 只能告诉你"shell prompt"或"命令在跑",**TUI 内部"在算"vs"在等"它分不出**。

### 4.7 ⚠️ termios / 进程态(L2 强信号)
- `tcgetattr(pty_master)` 看 `ICANON`,`/proc/<fg_pid>/wchan` 看是否阻塞在 `n_tty_read`
- 命中 → 场景 4(`apt y/n`)**强信号 waiting**
- raw mode TUI(claude/vim)`ICANON` off,**不命中**(避免和场景 2 混)
- Linux 好做,macOS `proc_pidinfo` 可做,**Windows ConPTY 黑盒**

**裁决**:Linux/macOS 顺手加,Windows 跳过。如果项目主战场是 Windows(看 README/软件定义书是 Windows-first),这层的边际价值有限。

### 4.8 ✓✓ LLM 屏幕复核(opt-in 终极方案)
解决场景 3 的多 session 盲区。**关键决策**:

**用文本序列化,不用 VLM 截图**。
- xterm.js `terminal.buffer.active` 给结构化 cell + 光标位置
- 拼"最后 20 行 + 光标位置"→ 纯文本 ~1k-2k tokens
- 截图反而:tokens 多、必须视觉模型、像素噪声分散注意力、隐私视觉冲击大

**准确度**(直觉):Haiku 4.5 在底部 10 行 + 光标位置上判"是否等输入" ≥ 90%。
- `> _` + claude 输入框边框 + `? for shortcuts` → waiting ✓
- `[Y/n] _` → waiting ✓
- `password:` → waiting ✓
- `VITE ready ... press h` 光标顶部 → running(提示语而非真等)✓
- vim `~` + 底栏 → 标 `interactive-foreground` 与 waiting 区分
- 边界:vim/less 这种"一直等键但用户主动操作"—— 给 LLM 明确分类标签即可

**触发**:只在前几层(L0/L1/L2)都判定"可能 waiting"但不能定夺时调,**不是 always-on**。
- 限频:同 session 30s 内最多一次
- 预估真实调用:每个活跃 session 每小时 < 10 次

**成本**:
- ~2k input + ~50 output ≈ $0.002/次(Haiku 4.5)
- 重度用户 1000 次/天 ≈ $2/天
- 加 prompt cache(系统提示 + few-shot 静态)更省
- 延迟 300-800ms,用户无感

**隐私(必须前置)**:
1. **默认关**,显式 opt-in
2. UI 提示"本会话已开启 AI 状态复核"
3. 序列化前轻量遮罩:`sk-`、`ghp_`、`AKIA` 前缀,32+ 位 `[A-Za-z0-9_-]` 串擦掉
4. 提供本地模型路径(Ollama + 3B 级 SLM 完全够用)
5. 单次复核请求即焚,不持久化

### 4.9 ⭐ 应用协作 OSC(远期理想)
让长 running TUI 主动发自定义 OSC 报告自身状态:
- Claude Code 进等待 → `ESC ] <code> ; waiting BEL`,收回输入 → `ESC ] <code> ; busy BEL`
- iTerm2 / WezTerm 生态有相关扩展但未成行业标准
- 如果 Claude Code 是可影响的产品,这是**多 session 场景 3 的最终解**。其他 TUI 走 LLM 复核。

## 5. 推荐分层架构(将来落地参考)

```
L0 流式静默检测(现有,always-on,零成本)
   ↓ 判定"可能 waiting"
L1 OSC 133(若 shell integration 已注入,直接定夺 prompt/running 边界)
   ↓ 命令未结束、未拿到 ;D
L2 termios / 进程态(Linux/macOS 强信号直接定夺 waiting)
   ↓ 仍不确定(场景 3:TUI 内部状态)
L3 LLM 屏幕复核(opt-in,决议)
   ↓
正式升级 waiting + 触发桌面通知 / 侧边栏闪
```

落地优先级:
1. **L1 OSC 133** —— 性价比最高,无副作用,跨平台。Vite 误报直接消失。
2. **L0 调阈值** —— 保留,作为兜底。
3. **L3 LLM 复核** —— opt-in,产品差异化点(其他终端没有),放"高级设置/AI 增强"里。
4. **L2 termios** —— Linux/macOS 可加,Windows 跳过。L3 出来后边际价值低,可砍。

## 8. 不做清单(明确否决)

- ❌ 主动往 PTY 写探测字节(任何字节都有副作用)
- ❌ 用 DOM 键鼠事件作为终端活跃信号(语义错位)
- ❌ 用 stdin 写入作为应用响应信号(同上)
- ❌ VLM 截图复核(序列化文本足够,截图徒增成本)
- ❌ "running-quiet"中间态(UI 认知负担)
- ❌ 默认开启 LLM 复核(隐私必须 opt-in)
