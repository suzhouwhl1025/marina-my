## 0. 摘要与优先级建议

### 一句话结论

**核心交付物（exe / setup）能用，但发布到外部前有 3 个 H 级硬伤必须处理**：①Marina 主程序 0 代码签名 → SmartScreen 蓝屏；②NSIS 安装器不写 `HKCU\Software\Marina\InstallLocation` → Win11 新右键菜单点击后找不到 Marina.exe；③`alpha.3` 的安装包是 `38e8abc` 之前打的，**不含** Win11 新菜单 MSIX 资源，今天分发的包等于功能阉割版。

### 体感最强的 7 个问题（按用户感知度排序）

| 现象 / 风险 | 直接根因 | 备忘录条目 |
|---|---|---|
| 第一次双击安装器 SmartScreen 直接拦 + "未知发布者" | 没启用 electron-builder 代码签名 | CRT-1 |
| Win11 新菜单点"在 Marina 终端中打开"无反应 / 报错 | DLL 读 `HKCU\Software\Marina\InstallLocation` 但没人写过 | MSX-1 |
| 当前 `release/0.1.0-alpha.3/` 装好后设置页面"Win11 新菜单 → 启用"必失败 | 旧 build 未带 MSIX/cer/install.ps1 extraResources | REL-1 |
| 证书 2029 年到期一刹那所有已分发 MSIX 同时失效 | MSIX 签名无时间戳 (RFC 3161 countersign) | CRT-2 |
| `.pfx` 私钥文件在工作区裸文件 + 密码 `marina-dev` 硬编码在 git 内脚本 | 没接 CI secret 管理 / 没分离 dev/release 凭证 | CRT-3 |
| 经典右键菜单条目左边没图标 | 写注册表时不带 `Icon` 值 | ICN-3 |
| 安装包体积 80MB、解包 291MB | 内含 55 个 locale .pak（~33MB）+ 4 份历史 release | REL-3 + REL-4 |

### 推荐先修清单（按"投入/收益"排序）

1. **MSX-1** · NSIS hook customInstall 写 `HKCU\Software\Marina\InstallLocation`（卸载时清掉）。约 8 行 NSIS，**修复 Win11 新菜单完全无功能**。
2. **REL-1** · 用当前 HEAD 重新 `npm run build`，产出 `0.1.0-alpha.4`（或就地覆盖 alpha.3，并清掉旧目录），让 extraResources 真正进包。
3. **MSX-2** · MSX-1 落地后顺手把 `.\scripts\install-context-menu.ps1` 跑通的 happy-path 做一次 cold-install 录像 / checklist，写进 `pre-release-测试单-v0.1.0-alpha.X.md`。
4. **CRT-1**（决策项）· 在内测 alpha 阶段先记入"已知限制"，beta 前申请 OV/Code-Signing 证书或转 Azure Trusted Signing。**没有便宜路；不修就接受 SmartScreen 警告**。
5. **CRT-2** · `signtool sign` 加 `/tr http://timestamp.digicert.com /td SHA256`。30 秒改 1 行，**避免 2029 年集体过期**。
6. **ICN-3** · `WindowsAdapter.registerFileManagerIntegration` 多写一对 `reg add ... /v Icon /d "<exe>,0" /f`。3 行，**经典菜单条目显示图标**。
7. **ICN-1** · `generate-icon.cjs` 产出多尺寸 ICO（16/24/32/48/64/128/256），不再用单一 PNG-in-ICO。约 30 行 Buffer 拼接，**Explorer 缩略图 / taskbar 小尺寸不糊**。
8. **REL-3** · 在 `electron-builder.yml` 增加 `electronLanguages: [en-US, zh-CN]`，把安装包体积从 80MB 砍到 ~50MB。
9. **REL-4** · 把 `release/0.1.0-alpha.0..2/` 共 ~1.3GB 旧产物从工作区删掉（已 gitignore，纯本地磁盘）。
10. **BLD-3** · `app-update.yml` 指向不存在的 `marina` 仓库；要么接 electron-updater 要么把它从 build 移除。
11. **NSI-2** · `installer.nsh` `customUnInstall` 里 ExecWait 没设 `/SILENT` flag；MSIX 卸载时仍可能弹 PS 窗一闪而过。
12. **CRT-4** · `dev-cert.ps1` 把 `.pfx` 密码 `marina-dev` 硬写脚本，且 `.cer` 会进发布包；密码本身不机密（cert 公钥包是公开的）但工作流应区分 dev / release 证书。

中期项：
- 接 electron-updater 做内测分发的自更新（README 已声明，但缺实现）
- 让 MSIX 内的 logo 走真正 44×44 / 150×150 像素（设计稿用 Aseprite 出小尺寸版本）
- `electron-builder.yml` 加 `compression: maximum`（牺牲打包时间换更小 setup）
- `release/${version}/` 改为 `release/`（让 latest.yml 路径稳定）

## 1. 图标资源（ICN-*）

### ICN-1 [M] `icon.ico` 只内嵌单张 256×256 PNG-in-ICO，缺多尺寸位图

- **位置**：`build/icon.ico` (1762 B)、`scripts/generate-icon.cjs:272-288 encodeIco`
- **现状**：generate-icon.cjs 把 256×256 PNG 包一层 ICONDIR 头就得到 .ico。`file` 报 `1 icon, 256x256 with PNG image data`。
- **影响**：Windows Vista+ 接受 PNG 嵌入 ICO 没问题，但 Explorer 列表小图标、taskbar 24×24 / 32×32、Alt+Tab 缩略图都是从 ICO 资源里**按尺寸挑最接近的位图**——只有 256×256 时，Windows 会动态缩到 16×16，圆角矩形 + 紫色 ">" 笔画在 16×16 锯齿严重。`build/README.md` 方案 B（ImageMagick `-define icon:auto-resize=256,128,64,48,32,24,16`）会产出多尺寸，但当前管道没有走。
- **修复方向**：把 `generate-icon.cjs` 改成同时产 16/24/32/48/64/128/256 七尺寸（每个尺寸都是程序化重画，不要单纯 PNG resize 让笔画再糊一次），ICONDIR count=7。或者放弃纯 JS 路线，引入 ImageMagick 一次性出货后 commit ICO。

### ICN-2 [L] `build/README.md` 与 `electron-builder.yml` 状态描述与实际反向

- **位置**：`build/README.md:32-35`
- **现状**：`build/README.md` 状态段写"`icon.ico` ❌（手动生成后放入）"、"`electron-builder.yml` 的 `win.icon` 行在 ico/png 都不存在时是注释的"——但实际 `build/icon.ico` 已存在，`electron-builder.yml:65 icon: build/icon.ico` 也是激活的。文档已过时。
- **影响**：低，仅误导新贡献者。
- **修复方向**：把 README 状态段改成"✅ 已生成"，或者直接删该段，统一指向 `scripts/generate-icon.cjs`。

### ICN-3 [M] 经典右键菜单条目无图标

- **位置**：`src/main/platform/windows.ts:234-263 registerFileManagerIntegration`
- **现状**：代码注释里写"Icon 字段暂不写 — build/icon.ico 还没生成，exe 内嵌图标走 Electron 默认"，但 build/icon.ico 已生成。Win11 新菜单（MSIX）会通过 `GetIcon()` 显示 `menu-icon.ico`，**经典菜单的 HKCU 注册表项不带 Icon 值** → Explorer 右键时菜单条左侧空白。
- **影响**：用户感知到"Marina 的右键菜单看起来不专业"。Win10 用户只能看到经典菜单，影响放大。
- **修复方向**：对每个 hive 多写一条 `reg add <hive> /v Icon /d "<exe>,0" /f`（`,0` 是 exe 资源里的第一个 ICON resource，electron-builder 已经把 build/icon.ico 嵌进 Marina.exe 了）。`unregisterFileManagerIntegration` 不需要单独删——`DeleteRegKey` 父 key 时 Icon 子值连带。

### ICN-4 [L] MSIX 内 logos 三件套全部是 256×256 同一份 PNG

- **位置**：`native/context-menu/package/assets/{Square150x150,Square44x44,StoreLogo}.png` SHA1 全为 `CAEC1E…` = `build/icon.png` 的副本
- **现状**：AppxManifest 期望 `Square150x150Logo`（开始菜单磁贴）和 `Square44x44Logo`（任务栏 / 通知 / Alt+Tab）；实际三张都是 256×256。Windows 在用 44×44 时下采样 5.8x。设计稿在 16×16/24×24 关键尺寸的 ">" 已是单像素笔画，再缩会糊。
- **影响**：用户在「设置 → 已安装应用」里搜索 Marina 看到模糊磁贴。但 AppxManifest `AppListEntry="none"`，开始菜单 / Apps 列表里**不显示**，所以只会偶发出现在 Settings 搜索结果里。**实际可见性低**。
- **修复方向**：optional。若处理：在 `scripts/generate-icon.cjs` 里增加 `OUT_44`/`OUT_150` 两种尺寸输出（同设计语言、手调像素），打包前替换。

### ICN-5 [L] `src/main/tray.ts` doc comment 提到 `variant='build-icon'` 实现已撤掉

- **位置**：`src/main/tray.ts:51-53`
- **现状**：注释说"variant='build-icon' 时优先尝试从 build/icon.ico 读取真实设计图标（打包后用）。读取失败统一 fallback 到 default 程序生成版"，但 `generateTrayIcon` 的实参签名是 `(variant: 'default' | 'active')`，没有 `'build-icon'`。
- **影响**：低，仅注释鬼影。
- **修复方向**：删该段注释，或者把 build-icon 真的接进来（性价比不高，托盘 16×16 程序化绘制目前足够清晰）。

### ICN-6 [L] BrowserWindow 创建时不显式 `icon:` 字段

- **位置**：`src/main/window-manager.ts:122-152`
- **现状**：Electron 在 Windows 上对**未打包** Marina（dev 模式）会用 Electron 默认 atom 图标作为窗口左上角小图标；**已打包**模式从 exe PE 资源读 ICON，会变成 Marina 紫色。dev 模式下 alt+tab 缩略图是 Electron 默认。
- **影响**：仅影响开发体验，生产用户无感。
- **修复方向**：optional。若处理：dev 模式下显式 `icon: nativeImage.createFromPath(join(__dirname, '../../build/icon.png'))`，用 `app.isPackaged` 分支。

## 2. electron-builder.yml 打包配置（BLD-*）

### BLD-1 [H] 没有任何 `win.signingHashAlgorithms` / `certificateFile` / `signtoolOptions` —— 代码签名完全没启用

- **位置**：`electron-builder.yml:56-66 win:` 段
- **现状**：`Get-AuthenticodeSignature` 实测：
  - `Marina-Setup-0.1.0-alpha.3-x64.exe` → Status=NotSigned
  - `Marina-Portable-0.1.0-alpha.3-x64.exe` → Status=NotSigned
  - `win-unpacked/Marina.exe` → Status=NotSigned
- **影响**：用户双击 setup.exe → Windows SmartScreen **蓝屏** "Windows protected your PC"，必须点 "More info → Run anyway"。心智门槛极高，alpha 测试者中至少 20% 会到此放弃。详见 CRT-1。
- **修复方向**：见 CRT-1。

### BLD-2 [M] `extraResources` 引入了 alpha.3 不存在的资源；当前 HEAD build 才会有

- **位置**：`electron-builder.yml:36-49`
- **现状**：当前 HEAD 的 yml 列出 4 项 context-menu 资源，但 `native/context-menu/build/MarinaContextMenu.msix`、`native/context-menu/dev-cert.cer`、`scripts/install-context-menu.ps1`、`scripts/uninstall-context-menu.ps1` 这 4 项都是 `38e8abc` 提交才引入的，而 alpha.3 setup 是 `afcd7e1` 时间点打的。任何新 build 必须保证 4 个源文件存在，否则 electron-builder 直接 fail。**`*.msix` 和 `*.cer` 是 `.gitignore`，没在 CI 上不存在，CI build 会爆**。
- **影响**：本地构建：开发者必须先跑 `native/context-menu/scripts/dev-cert.ps1` + `build.ps1` 才能 `npm run build`。CI 构建：不存在 CI workflow，但若未来接 GitHub Actions，会立刻挂。
- **修复方向**：
  - 短期：在 `npm run build` 前置一个 `prebuild` 脚本检查这 4 个文件，缺失时**输出友好错误**（"先跑 native/context-menu/scripts/build.ps1"）而不是让 electron-builder 抛 cryptic 错误。
  - 中期：CI 流程中 secret 管理 .pfx，build.ps1 接受 -PfxPath 参数。

### BLD-3 [M] `app-update.yml` 指向 github repo "marina"，但仓库是 `terminal`、且无 electron-updater 实现

- **位置**：`release/0.1.0-alpha.3/win-unpacked/resources/app-update.yml`、`package.json:8 homepage`
- **现状**：app-update.yml 写 `owner: Liyue-Cheng / repo: marina`。`package.json` homepage 也是 `https://github.com/Liyue-Cheng/marina`。但实际 git remote 仓库是 `terminal`（gitStatus 起手时 main branch 推 main）。同时 `package.json` 没引 `electron-updater`，主进程也没 `autoUpdater.checkForUpdates()` 代码。
- **影响**：app-update.yml 在用户机器上等于死文件；electron-builder 仍然会生成它 + `latest.yml`，但无人消费。误导仓库迁移者 / 贡献者。
- **修复方向**：
  - 决策 A：决定不做自动更新（alpha 阶段合理）→ 在 `electron-builder.yml` 加 `publish: null`，停止生成 app-update.yml / latest.yml / blockmap。
  - 决策 B：要做自动更新 → 安装 electron-updater + 修正 repo 名 + 主进程接入。

### BLD-4 [L] `requestedExecutionLevel: asInvoker` 正确，但 `nsis.perMachine: false` 与 `createStartMenuShortcut` 在某些 OneDrive 同步路径会失败

- **位置**：`electron-builder.yml:66, 71-76`
- **现状**：`asInvoker + perMachine: false` = 装到 `%LOCALAPPDATA%\Programs\Marina\`。Win11 默认很多用户的「桌面」「文档」「开始菜单」在 OneDrive 同步路径下。`createDesktopShortcut: true` 在 OneDrive 离线时偶发失败。
- **影响**：用户偶发反馈"装完没桌面快捷方式"，但程序本体能跑。
- **修复方向**：低优先。监控用户反馈即可，问题真的出现再改 NSIS hook 显式重试。

### BLD-5 [L] `files:` 默认 include 列表未显式排除 `.test.ts` / `vitest.config.ts`

- **位置**：`electron-builder.yml:21-27`
- **现状**：电子打包基础是 `out/**/*`（vite build 产物），test 源文件不会进 out/，不进 asar。但若有人误把 vitest config 写进 out 目录，没有 fail-safe。
- **影响**：当前实际无问题，纯加固。
- **修复方向**：低，可选。

### BLD-6 [L] `asarUnpack: node_modules/node-pty/**` 对路径表达式正确，已验证产物有效

- **位置**：`electron-builder.yml:30-31`
- **现状**：`release/0.1.0-alpha.3/win-unpacked/resources/app.asar.unpacked/node_modules/node-pty/build/Release/conpty/` 下有 `OpenConsole.exe`、`conpty.dll`——这是 node-pty 在 Windows 启动 ConPTY 的必要二进制，必须在 asar 外才能被 LoadLibraryW。已正确解出。

## 3. 证书 / 代码签名（CRT-*）

### CRT-1 [H] Marina.exe / Setup / Portable 三件套全部未代码签名

- **位置**：`electron-builder.yml win:` 段缺 `certificateFile/certificatePassword/signtoolOptions`
- **现状**：见 BLD-1 实测。生产层面 Marina 主程序 = 完全未知发布者。
- **影响**：
  - SmartScreen 二段 "More info → Run anyway"，必劝退 20-40% 用户
  - Defender Smart App Control（Win11 24H2+ 默认开）可能**直接拦截执行**
  - 企业策略 AppLocker 拦截
  - 用户右键 Marina.exe → 属性 → 数字签名 tab 不存在，看不到发布者
- **修复方向**：
  - **预算路线**：Sectigo / SSL.com 个人代码签名证书约 $200/yr OV 版本，需要邮寄身份证明，但 SmartScreen 信誉积累期 1-3 个月（"超过 X 个下载量后清除警告"）
  - **快速路线**：Azure Trusted Signing（前身 Azure Code Signing）订阅制 $9.99/mo，签发后立即清除 SmartScreen，但要求验证企业身份（个人开发者可申请 Individual 等级）
  - **接入实现**：electron-builder.yml `win:` 加：
    ```yaml
    certificateFile: ${env.MARINA_PROD_PFX}     # CI 注入路径
    certificatePassword: ${env.MARINA_PROD_PFX_PASSWORD}
    signtoolOptions:
      additionalCertificateFile: ...           # 中间证书
      sign: ./scripts/sign.js                  # 或自定义脚本接 timestamp 服务
    publisherName: "Liyue Cheng"
    ```
  - **不修接受现状**：在 README / 安装页提示用户 "首次启动需在 SmartScreen 弹框点 More info → Run anyway"，并把这段写进 docs/known-issues.md。**alpha 阶段可接受**，正式 release 前必须解决。

### CRT-2 [M] MSIX 签名无 RFC 3161 时间戳 → 2029-05-12 一刻所有已分发 MSIX 集体失效

- **位置**：`native/context-menu/scripts/build.ps1:53` 的 `signtool sign /f $pfx /p marina-dev /fd SHA256 $msix`
- **现状**：`Get-AuthenticodeSignature` 报 `TimeStamperCertificate: <空>`。证书有效期 2026-05-12 → 2029-05-12（NotAfter）。**未加 `/tr` 参数，未挂时间戳服务**。
- **影响**：当证书 NotAfter 一过，已分发 MSIX 的签名状态变 `NotValid`，新装会被 Add-AppxPackage 拒绝；**已装的用户重启后 Explorer 加载 DLL 会失败**（COM CLSID 解析 unchanged，但若用户清理过 trusted store 重装就废）。
- **修复方向**：`build.ps1:53` 改：
  ```powershell
  & $signTool sign /f $pfx /p marina-dev /fd SHA256 /tr http://timestamp.digicert.com /td SHA256 $msix
  ```
  时间戳服务签名后**即使证书过期，签名永远有效**。30 秒改动。

### CRT-3 [M] `dev-cert.pfx` 与硬编码密码 `marina-dev` 同步在 git 工作树/脚本中

- **位置**：`native/context-menu/dev-cert.pfx` (本地存在，`.gitignore` 排除)、`native/context-menu/scripts/dev-cert.ps1:40` + `build.ps1:53` 写明 `"marina-dev"`
- **现状**：`.pfx` 已 gitignore。但密码字符串明文在 git 内的两个脚本中。任何 fork 都能拿到。**.pfx 本身机密** → 持有 .pfx + 密码 = 可冒充 Marina Dev 签 MSIX。
- **影响**：
  - 自签证书的信任范围只到 `Cert:\CurrentUser\TrustedPeople`（用户在自己机器上手动 import），所以泄漏的 .pfx **只能影响"已经信任过 Marina Dev"的那些用户**。攻击面有限。
  - 但若未来同一个证书签到 Marina 主程序、或被加入信任范围更广的 store（如 Root），影响放大。
- **修复方向**：
  - 短期可接受。明确"dev-cert 是开发用一次性凭证，不外发"。
  - 中期：分离 dev / release 证书，release 证书私钥**只存 CI secret**（HashiCorp Vault / GitHub Secrets / Azure Key Vault），本地永远不接触。release MSIX 由 CI 签发。

### CRT-4 [L] `Cert:\LocalMachine\TrustedPeople` vs `Cert:\CurrentUser\TrustedPeople` 不一致

- **位置**：`native/context-menu/scripts/dev-cert.ps1:50` 导 LocalMachine，`scripts/install-context-menu.ps1:43` 导 CurrentUser
- **现状**：开发期 `dev-cert.ps1` 弹 UAC 导入 LocalMachine\TrustedPeople（全机器范围）；生产期用户从 Marina 设置页点"启用 Win11 新菜单"走 `scripts/install-context-menu.ps1`，导入 CurrentUser\TrustedPeople（当前用户范围）。
- **影响**：行为不一致——开发者机器上多机用户都能用 Marina MSIX；普通用户只有当前账号能用。multi-user 家庭机器换账号会"消失"。alpha 阶段不重要。
- **修复方向**：保持 install-context-menu.ps1 用 CurrentUser（无需 UAC，体验好），把 dev-cert.ps1 的 LocalMachine 注释清楚是"开发期方便，跨账号测"。文档统一。

### CRT-5 [L] `dev-cert.cer` 进 extraResources 后**会以发布包形式分发**，安全审计上需明确

- **位置**：`electron-builder.yml:44-45` 把 `native/context-menu/dev-cert.cer` 复制到 `resources/context-menu/marina-cert.cer`
- **现状**：`.cer` 是公钥证书，完全公开无机密。安装包带它没问题。但**用户看到"未知发布者的安装包带一个 .cer"**心理上会担心。
- **影响**：低，仅观感。安全角度无问题。
- **修复方向**：optional。在 README/about 页面声明"marina-cert.cer 是自签公钥，仅用于验证 Marina 右键菜单 MSIX 包，不授予任何系统权限"。

## 4. NSIS 安装/卸载脚本（NSI-*）

### NSI-1 [H] `installer.nsh` 缺 `customInstall` —— **不写 `HKCU\Software\Marina\InstallLocation`**，Win11 新菜单 DLL 找不到 Marina.exe

- **位置**：`build/installer.nsh` 只有 `customUnInstall`，`native/context-menu/src/command.rs:171-178 locate_marina_exe`
- **现状**：DLL 在 Invoke 时：
  ```
  1. 读环境变量 MARINA_EXE（开发期）
  2. 读 HKCU\Software\Marina\InstallLocation + \Marina.exe（生产期）
  ```
  但 `installer.nsh` / `electron-builder.yml` 全文都没人写 `HKCU\Software\Marina\InstallLocation`。`native/context-menu/README.md:37` 写"生产期 NSIS 写"——**承诺没兑现**。
- **影响**：用户安装 Marina → 在设置页启用 Win11 新菜单（MSIX 注册成功）→ 在 Explorer 右键选「在 Marina 终端中打开」→ **DLL 找不到 exe，菜单点了无反应（panic 兜底，Invoke 返回 E_FAIL，无 toast）**。**主功能直接坏**。
- **修复方向**：`build/installer.nsh` 加：
  ```nsis
  !macro customInstall
    WriteRegStr HKCU "Software\Marina" "InstallLocation" "$INSTDIR"
  !macroend
  ```
  同时 `customUnInstall` 多删一条：
  ```nsis
  DeleteRegValue HKCU "Software\Marina" "InstallLocation"
  DeleteRegKey /ifempty HKCU "Software\Marina"
  ```
  另：portable 形态没安装流程，无法写注册表 → portable 用户启用 Win11 新菜单是 unsupported（`explorer-integration.ts` 已经在 `checkModernSupport` 里返回 false）。

### NSI-2 [L] `customUnInstall` 内 `ExecWait powershell ...` 无窗口隐藏标记

- **位置**：`build/installer.nsh:27`
- **现状**：`ExecWait 'powershell.exe -NoProfile -NonInteractive ...'` —— NSIS 的 `ExecWait` 默认会让子进程的控制台窗口短暂可见（PS 5.1 NoLogo + NonInteractive 已经压低，但仍有一个 conhost 窗口闪一下）。
- **影响**：用户卸载 Marina 时看到黑色窗口一闪而过，怀疑"安装包恶意行为"。
- **修复方向**：要么用 `nsExec::Exec`（NSIS 内置无窗口 exec），要么 PowerShell 改成 `-WindowStyle Hidden`：
  ```nsis
  ExecWait 'powershell.exe -NoProfile -NonInteractive -WindowStyle Hidden -ExecutionPolicy Bypass -Command "..."'
  ```

### NSI-3 [L] `customUnInstall` MSIX 卸载失败不阻塞，但**不向用户反馈**

- **位置**：`build/installer.nsh:27`
- **现状**：注释明确"失败也只是残留 reg 项已经清完了"。但若 MSIX 卸载失败（罕见，如 Explorer 进程持有 DLL），用户重装 Marina 时新菜单可能出现重复条目或"已安装的包版本不匹配"。
- **影响**：极低概率边缘 case。
- **修复方向**：optional。可在卸载日志（NSIS DetailPrint）里记一行警告。

### NSI-4 [L] `customUnInstall` 不删 `Cert:\CurrentUser\TrustedPeople\Marina Dev` 证书

- **位置**：`build/installer.nsh` 整体缺
- **现状**：注释明确"证书保留不动，下次安装免 UAC"。这是个设计选择不是 bug。
- **影响**：用户彻底卸载 Marina 后仍有一张 Marina Dev 自签证书躺在 trusted store 里。理论上是冷数据，无主动威胁。
- **修复方向**：optional。可在卸载 UI 加一个"完全清除（含证书）"复选框。

### NSI-5 [L] `shortcutName: Marina` 与 `productName: Marina` 重复

- **位置**：`electron-builder.yml:14, 74`
- **现状**：electron-builder 在 `shortcutName` 未指定时默认用 `productName`，所以多余配置但无害。
- **影响**：无。
- **修复方向**：optional 删 line 74。

## 5. Win11 新菜单 MSIX 包（MSX-*）

### MSX-1 [H] 与 NSI-1 同根问题：DLL 找不到 Marina.exe

详见 NSI-1。这条用 MSX-1 编号方便从右键菜单视角索引。

### MSX-2 [M] MSIX 在 portable 形态 / dev 形态下 extraResources 不存在，错误信息不明确

- **位置**：`src/main/explorer-integration.ts:275-289 setModernIntegration`
- **现状**：portable 包不含 context-menu/ 子目录（extraResources 是 NSIS 安装才落地）。`getMsixScriptPath` 返回 null → 报 "安装资源 install.ps1 缺失；请确认 Marina 是从 NSIS 安装包安装（便携版不含）。" —— 措辞还行。但 `getMsixAssetPath` 在 `enabled && (!msixPath || !certPath)` 时报 "安装包 / 证书资源缺失" —— **没区分是哪个缺失**，调试时不够明确。
- **影响**：debug 难度，正常用户不会看到。
- **修复方向**：把第二条 message 改成具体说哪个文件缺。

### MSX-3 [L] AppxManifest 内 Identity Version 与 marina-app version 解耦

- **位置**：`native/context-menu/package/AppxManifest.xml:16 Version="0.1.0.0"`、`native/context-menu/Cargo.toml:3 version = "0.1.0"`、`package.json:3 version = "0.1.0-alpha.3"`
- **现状**：AppxManifest 是 4-segment Windows 版本号（不支持 -alpha.x SemVer）。手动改 `package.json` 时不改 AppxManifest，Marina 主程序升级 0.2.0 后 MSIX 还停在 0.1.0.0。
- **影响**：Add-AppxPackage 会因为 "版本号未提升" 静默跳过更新（用户 MSIX 仍是老版本）。但 Marina 主程序版本和 MSIX DLL 的契约稳定（CLSID 不变 + invoke 走 cmdline），版本不一致**功能上不会坏**。
- **修复方向**：在 `native/context-menu/scripts/build.ps1` 头部读 `../../package.json` 的 version，把 `0.1.0-alpha.3` 转成 `0.1.0.3`（去掉 `-alpha.`），替换 AppxManifest 内的 Version 后再 MakeAppx。

### MSX-4 [L] `MaxVersionTested="10.0.26200.0"` 已接近当前用户实际 build

- **位置**：`native/context-menu/package/AppxManifest.xml:27`
- **现状**：MaxVersionTested 仅是声明性字段（不限制实际安装），但 26200 是 Win11 24H2。
- **影响**：无功能影响。
- **修复方向**：optional 升到 26300+。

### MSX-5 [L] DLL `GetIcon` 用绝对路径，跨用户 InstallLocation 不同时仍 OK

- **位置**：`native/context-menu/src/command.rs:128-148 locate_menu_icon`
- **现状**：用 `GetModuleFileNameW(self_HMODULE)` 取当前 DLL 路径再拼 `assets\menu-icon.ico`。MSIX sparse package 模式下 ExternalLocation 指向 `<Marina>\resources\context-menu\`，路径稳定。
- **影响**：无。**纯审计确认**。

### MSX-6 [L] DLL `Invoke` 启动 Marina.exe 不传任何环境过滤

- **位置**：`native/context-menu/src/command.rs:217-244 launch_marina`
- **现状**：CreateProcessW 不设 lpEnvironment（继承调用进程环境）。调用进程是 `marina-context-menu-host.exe` (sparse package surrogate)，环境继承自 Explorer.exe，会带 Explorer 的全部环境变量。
- **影响**：极低。Marina 应该在 argv-parse 后自洽，不依赖未声明环境。但若 Explorer 环境含 `MARINA_EXE` 类污染变量，可能影响 build-type 探测。
- **修复方向**：低优先。

### REL-1 [H] `release/0.1.0-alpha.3/` 不含 `extraResources` 加入的 Win11 新菜单资源

### REL-2 [L] `latest.yml / *.blockmap` 生成但无 publish target

- **位置**：`release/0.1.0-alpha.3/latest.yml`、`Marina-Setup-...-x64.exe.blockmap`
- **现状**：electron-builder 默认生成 publish artifacts（即使 `publish: null` 没设）。`*.blockmap` ~85KB 是差分更新用的，blockmap + electron-updater 才有意义。
- **影响**：磁盘占用增加 + 给用户的下载页若也挂这些文件会让人困惑"哪个是要下载的"。
- **修复方向**：见 BLD-3 决策路径。

### REL-3 [M] 安装包含 55 个 locale .pak（~33MB），可瘦身到 5 个

- **位置**：`release/0.1.0-alpha.3/win-unpacked/locales/` 55 files
- **现状**：每个 .pak ~600KB，55 个共 ~33MB。Marina 界面只支持中文 / 英文 + 内嵌字符串（没用 Chromium 的 i18n 框架），其他 53 个 locale 全是浪费。
- **影响**：安装包 80MB → 一半是 locale + chromium. 中等优化空间。
- **修复方向**：`electron-builder.yml` 加：
  ```yaml
  electronLanguages:
    - en-US
    - zh-CN
  ```
  实测可砍 ~30MB。

### REL-4 [M] `release/0.1.0-alpha.{0,1,2,3}/` 共 ~1.7GB 旧产物堆在工作区

- **位置**：du 实测：alpha.0 = 428MB, alpha.1 = 450MB, alpha.2 = 451MB, alpha.3 = 451MB
- **现状**：`release/` 已 gitignore 不入 commit，但本地磁盘四份。alpha.0 还是 "EasyTerm-Setup-..." 命名（改名前）。
- **影响**：磁盘占用 1.7GB，新 build 时 electron-builder 会问"是否覆盖"。开发节奏变慢。
- **修复方向**：删 `release/0.1.0-alpha.{0,1,2}/` 三个旧目录（alpha.3 保留作对照）。
- **执行风险**：低，已 gitignore，可重建。

### REL-5 [L] Setup / Portable 命名规范不区分 channel

- **位置**：`electron-builder.yml:62, 82`
- **现状**：`Marina-Setup-${version}-${arch}.${ext}` 在 v1.0 时是 `Marina-Setup-1.0.0-x64.exe`，下载页 alpha/beta/stable 同名不可同地分发。

### REL-6 [L] win-unpacked Marina.exe 是 180MB（含完整 Electron）
